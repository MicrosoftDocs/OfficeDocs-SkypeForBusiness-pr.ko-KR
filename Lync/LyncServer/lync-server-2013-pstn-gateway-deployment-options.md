## <a name="pstn-gateways"></a>PSTN 게이트웨이

PSTN (공개 통신 네트워크) 게이트웨이는 직접 또는 SIP trunks 연결을 통해 기업 음성 인프라와 PSTN 간의 신호 및 미디어를 변환 하는 타사 하드웨어 구성 요소입니다. 어느 토폴로지에서 든 게이트웨이는 PSTN을 종료 합니다. 게이트웨이는 해당 서브넷에서 격리 되 고 중재 서버를 통해 엔터프라이즈 네트워크에 연결 됩니다.

일반적으로 여러 사이트가 있는 엔터프라이즈는 각 사이트에 하나 이상의 게이트웨이를 배포 합니다. 지점 사이트는 게이트웨이를 통해 또는 게이트웨이 및 서버를 단일 상자에 결합 하는 Survivable Branch 기기를 통해 PSTN에 연결할 수 있습니다. 지점 사이트에서 게이트웨이를 사용 하는 경우 WAN 링크가 탄력적으로 인 한 사이트에는 등록자와 중재 서버가 둘 다 필요 합니다. 프런트 엔드 서버에서 collocated 한 하나 이상의 중재 서버는 각 사이트에서 하나 이상의 게이트웨이에 대 한 통화를 라우팅할 수 있습니다. 사이트에 필요한 등록자, 중재 서버 및 게이트웨이는 Survivable Branch 기기로 배포 하는 것이 좋습니다.

PSTN 게이트웨이의 수, 크기 및 위치를 결정 하는 것은 엔터프라이즈 음성 인프라를 계획할 때 수행 해야 하는 가장 중요 하 고 부담이 큰 의사 결정 일 수도 있습니다.

고려해 야 할 주요 질문은 다음과 같습니다. 이러한 질문에 대 한 답변이 모두 상호 의존적인 지는 점에 유의 하세요.

  - 얼마나 많은 PSTN 게이트웨이가 필요 한가요? 답은 사용자 수, 예상 동시 통화 수 (트래픽 로드), 사이트 수 (각 사이트 마다 하나씩)에 따라 달라 집니다.

  - 게이트웨이는 어떤 크기를 가질 수 있나요? 응답은 사이트의 사용자 수와 트래픽 로드에 따라 달라 집니다.

  - 게이트웨이가 어디에 있나요? 해답은 부분적으로 조직의 지리적 배포에 따라 달라 집니다.

또한 게이트웨이 토폴로지 옵션도 고려해 야 합니다 (자세한 내용은이 항목의 뒷부분에 있는 게이트웨이 토폴로지 참조).

## <a name="mn-trunk-support"></a>M:N 트렁크 지원

중재 서버는 여러 게이트웨이, 인터넷 전화 통신 서비스 공급자가 제공 하는 SBCs (세션 경계 컨트롤러) 또는 두 가지 조합으로 전화를 라우팅할 수 있습니다. 또한 풀의 여러 중재 서버는 여러 게이트웨이와 상호 작용할 수 있습니다. 중재 서버와 게이트웨이 사이에 정의 된 논리 경로를 *트렁크*라고 합니다. 내부 사용자가 PSTN 통화를 하는 경우 프런트 엔드 풀의 아웃 바운드 라우팅 논리는 특정 통화를 라우팅하는 데 사용할 수 있는 모든 가능한 조합으로 경로를 지정할 트렁크를 선택 합니다. DNS 부하 분산을 사용 하면 풀의 특정 조정 서버에 대 한 문제로 인해 호출이 게이트웨이에 연결 되지 않으면 풀의 대체 중재 서버로 통화가 다시 시도 됩니다.

여러 게이트웨이를 계획 하는 방법에 대 한 자세한 내용은 [Lync Server 2013에서 M:n 트렁크](lync-server-2013-m-n-trunk.md)을 참조 하세요.

다른 아웃 바운드 라우팅 개선 사항에 대 한 자세한 내용은 [Lync Server 2013의 음성 경로](lync-server-2013-voice-routes.md)를 참조 하세요.

## <a name="gateway-topologies"></a>게이트웨이 토폴로지

게이트웨이 배포의 기본 질문을 고려 하는 경우 다음 단계를 따릅니다.

1.  엔터프라이즈 음성을 사용 하 여 PSTN 연결을 제공 하려는 사이트의 개수를 계산 합니다.

2.  각 사이트에서 소통량을 예측 합니다 (사용자 수, 사용자 당 평균 통화 수).

3.  각 사이트에서 하나 이상의 게이트웨이를 배포 하 여 예상 되는 트래픽을 처리 합니다.

다음과 같은 분산 게이트웨이 토폴로지가 다음 그림에 나와 있습니다.

**분산 게이트웨이 토폴로지**

![분산 게이트웨이 토폴로지 다이어그램](images/Gg398899.f0f65a0b-a462-491a-878b-4d4bf0a96f6d(OCS.15).jpg "분산 게이트웨이 토폴로지 다이어그램")

이 토폴로지에서는 각 사이트의 작업자와 사이트 간 통화가 모두 인트라넷을 통해 라우팅됩니다. PSTN으로 거는 통화는 대상 번호의 위치에 가장 가까운 게이트웨이로 엔터프라이즈 IP 네트워크를 통해 라우팅됩니다. 하지만 조직에서 하나 이상의 대륙에 걸쳐 있는 수십 개나 수십만 개의 사이트를 지 원하는 경우 많은 금융 기관과 기타 대기업 에서도 이러한 기능을 활용할 수 있습니다. 이러한 경우 각 사이트에 별도의 게이트웨이를 배포 하는 것은 실용적이 지 않습니다.

이 문제를 해결 하기 위해 많은 대규모 회사는 다음 그림과 같이 하나 또는 여러 개의 대규모 전화 통신 센터 사이트를 배포 하는 것이 좋습니다.

**전화 통신 중앙 사이트 토폴로지**

![데이터 센터 게이트웨이 토폴로지](images/Gg398899.927f4808-bf74-405a-be20-2cd9cd87af6d(OCS.15).jpg "데이터 센터 게이트웨이 토폴로지")

이 토폴로지에서는 예상 사용자 부하를 수용할 수 있는 여러 대규모 게이트웨이가 각 중앙 사이트에 배포 됩니다. 기업의 사용자에 대 한 모든 통화는 회사의 전화 서비스 공급자에 게 중앙 사이트로 전달 됩니다. 중앙 사이트의 라우팅 논리는 전화를 인트라넷을 통해 보낼지 아니면 PSTN으로 라우팅해야 할지를 결정 합니다.

## <a name="gateway-location"></a>게이트웨이 위치

게이트웨이 위치에는 사용자가 선택 하는 게이트웨이 유형 및 구성 방법을 결정할 수도 있습니다. 다양 한 PSTN 프로토콜이 있으며, 전세계 표준에 해당 하지 않는 것도 있습니다. 모든 게이트웨이가 단일 국가/지역에 있는 경우에는 문제가 되지 않지만 여러 국가/지역에서 게이트웨이를 찾은 경우 해당 국가/지역의 PSTN 표준에 따라 각각을 구성 해야 합니다. 더욱이 캐나다와 같이 작업을 인증 한 게이트웨이는 인도, 브라질 또는 유럽 연합에서 인증 되지 않을 수 있습니다.

## <a name="gateway-size-and-number"></a>게이트웨이 크기 및 번호

대부분의 조직에서 배포를 가장 많이 하는 PSTN 게이트웨이는 2에서 960 포트로의 크기의 범위를 고려 합니다. (대규모 게이트웨이는 있지만 주로 전화 서비스 공급자가 사용 합니다.) 조직에 필요한 포트 수를 예상할 때 다음 지침을 사용 합니다.

  - 사용량이 적은 전화 통신을 사용 하는 조직 (시간당 사용자 당 하나의 PSTN 통화)은 모든 15 명의 사용자에 대해 하나의 포트를 할당 해야 합니다. 예를 들어 20 명의 사용자가 있는 경우에는 두 개의 포트가 있는 게이트웨이가 필요 합니다.

  - 일반 전화 통신 사용량이 있는 조직 (시간당 사용자 당 2 개의 PSTN 통화)은 10 명의 사용자 마다 하나의 포트를 할당 해야 합니다. 예를 들어 100 사용자가 있는 경우 하나 이상의 게이트웨이에 할당 된 총 10 개의 포트가 필요 합니다.

  - 과도 한 전화 통신을 사용 하는 조직 (시간 당 사용자 당 3 명 또는 그 이상의 PSTN 통화)은 5 명의 사용자 마다 포트를 하나 할당 해야 합니다. 예를 들어 47000 사용자가 있는 경우 10 개 이상의 대형 게이트웨이에서 할당 된 총 9400 포트가 필요 합니다.

  - 사용자 수가 증가 하거나 조직의 트래픽 양이 증가할수록 추가 포트를 얻을 수 있습니다.

지원 해야 하는 지정 된 수의 사용자에 대 한 배포 방법에는 더 적거나, 더 크거나, 게이트웨이 중 하나를 사용 하는 것이 좋습니다. 일반적으로 하나의 게이트웨이에 장애가 발생 한 경우에는 조직에 대해 최소 두 개의 게이트웨이를 사용 하 여 유지 관리 하는 것이 좋습니다.

배포 하는 각 PSTN 게이트웨이에는 해당 중재 서버가 하나 이상 있어야 합니다.
