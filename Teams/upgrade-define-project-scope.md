# <a name="define-your-project-scope"></a>프로젝트 범위 정의

![프로젝트 정의 스테이지를 강조 표시 하 여 여행 다이어그램 업그레이드](media/upgrade-banner-project-definition.png "프로젝트 정의 단계에 대 한 강조를 사용 하 여 업그레이드 여행 단계")

이 문서는 업그레이드 여행에 대 한 프로젝트 정의 단계의 일부 이며, 확인 한 관련자 로부터 후원 coalition 및 프로젝트 팀을 만든 후 완료 한 활동은 프로젝트의 성공에 대 한 핵심입니다. 계속 하기 전에 다음 활동을 완료 했는지 확인 합니다.

- [참여 한 프로젝트 이해 관계자](upgrade-enlist-stakeholders.md)

프로젝트 비전, 범위, 목표, 관리를 정의 하는 데 시간이 오래 걸리면 모든 프로젝트 이해 관계자가 동일한 최종 결과를 기준으로 정렬 되 고 작업 하 게 됩니다. 이는 특히 기술 준비 팀과 사용자 준비 팀이 각자의 각 부분을 함께 하는 데 독립적으로 작동 하는 경우에 중요 합니다. 이 섹션을 완료 한 후에는 프로젝트 전체에서 해당 작업을 참조 하 여 원하는 종료 상태를 달성 하는 것이 진행 중인지 확인 합니다. 아래에서 확인 하는 목표를 사용 하 여 결과를 기준으로 측정 하 고 필요에 따라 줄입니다.

## <a name="project-vision"></a>프로젝트 비전

여러분의 비전은 "" 대규모 사진 "" 또는 "이 프로젝트를 진행 하는 이유는 무엇 인가요?" 라는 질문에 대 한 답을 제공 하는 궁극적인 종료 상태입니다. 이상적인 비전은 다음 예제에 표시 된 대로 조직의 비즈니스 드라이버 및 사용자 값을 다룹니다.

- **조직 비즈니스 드라이버**: Microsoft 팀에서 표준화 하 여 디지털 작업 공간 변환에 맞추고, 중복 솔루션을 제거 하 고, USD5를 저장할 수 있도록 합니다.
- **사용자 값-추가**: Microsoft 팀 (1)은 프로젝트 노트, Office 문서, 팀 구성원, 대화, 모임에 대 한 단일 위치를 제공 하 여 시간을 절약 합니다. (2) 중앙 연락처 목록과 대화에 빠르게 액세스할 수 있는 지속적인 채팅 추적을 사용 하 여 통신을 간소화 하 고 (3) 한 곳에서 파일을 저장 하 고 액세스 하 여 손실 된 전자 메일 첨부 파일을 찾으려고 할 때의 불편을 줍니다.

다음의 토론 요점을 고려 하 여 비전을 개선 하세요.

- 현재 비즈니스 프로세스에 대 한 설명

- 기존 비즈니스 프로세스의 문제

- 기술이 이러한 문제를 해결 하는 데 도움이 되는 방법

- 이러한 문제를 극복 하는 경우 예상 되 고 측정 가능한 비즈니스 결과

> [!TIP]
> 사용 사례 및 가상 사용자를 식별 하 여 프로젝트 비전을 세부적으로 조정 합니다.

## <a name="project-scope"></a>프로젝트 범위

비전은 다양 한 단계를 통해 시간에 따라 실현 될 수 있습니다. 프로젝트 범위는 지금 프로젝트의 초점을 정의 하 고, 프로젝트 팀 _이_ 현재 작업을 중점적으로 진행 하 여 장기간의 비전을 실현할 수 있도록 하는 역할을 합니다. 예를 들어 범위에서 파일럿을 실행 하거나, 음성 또는 모임 등의 특정 작업을 배포 하거나, 시간에 따라 업그레이드를 계획할 때 비즈니스용 Skype와 함께 팀을 사용 하도록 설정할 수 있습니다. 프로젝트 범위의 일부로 다음을 평가 해야 합니다.

- [다양 한 공존 모드](https://aka.ms/SkypeToTeams-Coexist)와 조직에 가장 적합 합니다.
- 팀으로 이동 하기 전에 비즈니스용 Skype와 팀이 공존할 수 있는 가장 좋은 방법입니다.
- 조직의 기술 및 사용자 준비 상태를 확인 하기 위해 [파일럿](https://aka.ms/SkypeToTeams-Pilot) 을 수행 해야 하는지 여부

## <a name="project-goals"></a>프로젝트 목표

목표는 원하는 결과를 정의 하 고 프로젝트의 성공 여부를 측정할 수 있도록 합니다. 목표는 _목표 및 키 결과_ (okrs)로 정의 될 수 있으며, 프로젝트 성공의 측정값은 _키 성공 표시기_ (ksis)로 정의할 수 있습니다. 사용자가 소유권을 인식 하 고 정의 된 프로젝트 작업에 이러한 측정값을 정렬 하는 데 도움이 되도록, OKRs와 KSIs를 정의 하는 것은 프로젝트 관련자 로부터 완전히 참여 하는 것이 중요 합니다. 목표에는 기술 및 사용자 중심 성공의 혼합이 포함 되어야 합니다.

- **Okrs** 는 프로젝트 시작 부분에서 설정한 목표와 정의 된 흐름 (예: 월 또는 분기)에 대해 측정 하는 주요 결과를 포함 합니다. 주요 결과를 검토 하 여 프로젝트 인도품이 일정에 맞는지 확인 하거나 프로젝트를 다시 추적 하기 위해 문제를 식별 하 고 줄일 수 있습니다. 일반적으로 OKRs는 "획득" 또는 "달성 하지 않음"으로 분류 됩니다.
- **Ksis** 는 주요 결과의 품질과 성공 여부를 측정 하 고 좋은 결과 및/또는 잘못 된 결과를 자세히 설명 하 여 okrs의 이진 특성을 보완 합니다. KSIs를 정의할 때는 "특정, 측정 가능, 할당 가능, 실제, 시간 관련" (스마트) 조건을 사용 하는 것이 좋습니다.
  - 구체적: 개선을 위한 특정 영역 대상 지정
  - 측정 가능한 대상: 수량화 또는 최소한 표시기 추천, 진행 중
  - 할당 가능: 작업을 수행할 사용자 지정
  - 실제: 사용할 수 있는 리소스에 대해 사실적으로 달성 해야 하는 결과를 제공 합니다.
  - 시간 관련: 결과를 얻을 수 있는 시기를 지정 합니다.

## <a name="risks-and-mitigation"></a>위험 및 완화

 프로젝트를 사용 하면 예기치 않은 이벤트 나 다른 요소가 발생할 수 있으며 프로젝트가 추적 되지 않습니다. 잠재적 위험을 사전 평가 하 고 발생할 수 있는 문제를 극복 하는 완화 계획을 정의 하는 것이 중요 하므로 프로젝트는 목표를 계속 진행할 수 있습니다. _위험 등록기_ 는 프로젝트 위험을 추적 하는 훌륭한 도구 이며, 잠재적인 영향과 잠재적 영향, 완화 계획을 캡처할 수 있습니다. 다음 표에서는 샘플 위험 등록기를 보여 줍니다.

## <a name="timeline"></a>시간대

업그레이드 여행의 범위를 지정할 때 주요 중요 시점에 대 한 시간 표시 (예: 모든 사용자에 대해 비즈니스용 Skype와 함께 팀 사용 가능)를 설정 하 여 완료 날짜 외에도이를 수행 하세요. 정의 된 시간 표시 막대를 사용 하면 프로젝트 팀이 일관성 있는 종료 상태를 유지 하 고 적절 한 작업 진행 일정을 알려 주므로 프로젝트를 추적할 수 있습니다. 가속 되지 않는 시간 표시 막대 (작업이 간과 될 수 있는 위치) 또는 너무 멀리 (통해 계속 앞서 손실 될 수 있는 위치)를 고려 하세요. 다음에 대 한 적절 한 시간 표시 막대 계정

- 제품에 대 한 **규정 준수 및 사용자 시나리오 요구 사항**: [제품 로드맵](https://aka.ms/O365Roadmap) 을 참조 하 여 팀이 조직에 적합 한지 평가 합니다.
- **그룹 업그레이드**: 팀을 사용할지 아니면 업그레이드 그룹을 통해 사용자를 업그레이드할 것인지 결정 합니다 .이는 전체 업그레이드의 시간 표시 막대에 영향을 줄 수 있습니다.
- **변경 중지, 회계 연도 끝, 배포 수명 주기**: 업그레이드 시간 표시에 영향을 줄 가능성이 있는 모든 내부 프로세스에 대 한 논의 및 계정입니다.
- **동시에 발생 하는 기타 변경**사항: 변경 내용 묶음 또는 간격을 조정 하 여 긍정적 사용자 환경을 활용 하 고 생산성에 미치는 영향을 최소화 하세요.
- **Resourcing**: 함께 만든 프로젝트 팀이 필요한 모든 작업을 완료 하는 데 충분 한 대역폭을 보유 하 고 있는지 확인 하기 위해 프로젝트 관련자와 자원 배정을 확인 합니다.
