#  <a name="use-cqd-to-manage-call-and-meeting-quality-in-microsoft-teams"></a>Microsoft 팀에서 통화 및 모임 품질을 관리 하기 위해 CQD 사용 

이 문서는 팀 관리자 또는 지원 및 헬프데스크 엔지니어에 게 Microsoft 팀의 통화 품질 대시보드 (CQD)를 사용 하 여 조직의 통화 및 모임 품질을 모니터링 하 고 유지 관리 하는 프로세스를 개발 하는 데 도움을 줍니다. 오디오 환경을 개선 하기 위해 향상 된 네트워크 기능 향상으로 비디오 및 공유의 향상 된 기능으로 변환 되므로 오디오 품질 시나리오가 강조 표시 됩니다.

이 지침의 핵심은 두 개의 [큐 레이 팅 CQD 서식 파일](https://aka.ms/QERtemplates) 입니다 .이 문서에 나와 있는 지침을 진행 하기 전에 다운로드 하는 것이 좋습니다.

이 문서에서는 이미 [CQD를 설정한](turning-on-and-using-call-quality-dashboard.md)것으로 가정 합니다.

이 문서에서 설명 하는 영역을 지속적으로 평가 하 고 수정 사용자에 게 부정적인 영향을 줄 수 있습니다. 대부분의 사용자 문제는 다음과 같은 범주로 분류할 수 있습니다.

-   방화벽 또는 프록시 구성이 완료 되지 않음
-   불량 Wi-fi 서비스
-   대역폭 부족

-   일관성 없거나 오래 된 클라이언트 버전 및 드라이버
-   최적화 되지 않음 또는 내장 오디오 장치
-   문제가 있는 서브넷 또는 네트워크 장치

팀 또는 비즈니스용 Skype Online을 배포 하기 전에 적절 한 계획과 디자인을 통해 고품질 환경을 유지 하는 데 필요한 작업량을 줄일 수 있습니다.

이 문서에서는 채택 및 영향을 최대화 하기 위해 오디오에 특별 한 초점을 두어 각 영역을 보고 하 고 조사 하는 기본 도구로 CQD (통화 품질 대시보드)를 온라인으로 사용 하는 방법에 대해 중점적으로 설명 합니다. 오디오 환경을 개선 하기 위해 네트워크에 대 한 개선 사항은 비디오 및 데스크톱 공유의 향상 된 기능으로 직접 번역 될 수 있습니다.

평가의 속도를 높이려면 모든 네트워크를 관리 하는 방법과 다른 하나는 관리 되는 (내부) 네트워크용으로 필터링 된 [두 개의 큐 레이 팅 CQD 템플릿을](https://aka.ms/qertemplates) 제공 합니다. 모든 네트워크 템플릿 보고서는 빌드 및 네트워크 정보를 표시 하도록 구성 되지만, 빌드 정보를 수집 하 고 업로드 하는 동안에도 계속 사용할 수 있습니다. CQD에 빌드 정보를 업로드 하면 외부 서브넷에서 내부를 구분 하는 동안 사용자 지정 빌드, 네트워크 및 위치 정보를 추가 하 여 해당 서비스에서 보고 기능을 향상 시킬 수 있습니다. 자세한 내용은 [문서 매핑](CQD-building-mapping.md)읽기를 참조 하세요.

## <a name="what-is-quality"></a>품질 이란?

이 컨텍스트에서 quality는 서비스 메트릭과 사용자 환경의 조합입니다.


### <a name="service-metrics"></a>서비스 메트릭

서비스 메트릭은 특정 클라이언트 기반 메트릭으로 구성 됩니다. 각 통화 중에 클라이언트는 호출에 대 한 원격 분석을 수집 하 고, 나중에 CQD 또는 [사용자 단위 통화 분석](set-up-call-analytics.md)에서 액세스할 수 있는 각 호출이 끝날 때 보고서를 제출 합니다. 이러한 메트릭에는 다음이 포함 되어 있습니다 (제한 되지는 않음).

-   불량 스트림 (들어오고 나가는)
-   설치 실패 비율
-   실패 율의 낙하


#### <a name="poor-stream-rate"></a>불량 스트림 속도

PSR (불량 스트림 속도)은 품질이 좋지 않은 조직의 전반적인 스트림 백분율을 나타냅니다. 이 메트릭은이 값을 줄이고 사용자 환경을 개선 하는 데 가장 큰 영향을 줄 수 있는 영역을 강조 표시 하는 데 사용 되며,이 경우 PSR을 볼 때 [관리 네트워크가](#managed-versus-unmanaged-networks) 기본 포커스가 됩니다. 외부 사용자도 중요 하지만 조사가 조직의 기준에 따라 달라 집니다. 외부 사용자에 게 모범 사례를 제공 하 고 전체 조직과 관계 없이 외부 통화를 조사 하는 것이 좋습니다.

CQD의 실제 측정은 작업 부하에 따라 다르지만,이 문서의 목적을 위해 주로 _오디오 불량 비율_ 에 초점을 두고 있습니다. PSR은 다음 표에 설명 된 다섯 가지 네트워크 메트릭 평균값으로 구성 됩니다. 스트림이 불량으로 분류 되는 경우 한 메트릭으로 정의 된 임계값을 초과 해야 합니다. CQD는 "낮은 기한 ..."을 제공 합니다. 측정값을 더 잘 이해 하기 위해 스트림을 불량으로 분류 한 상태입니다. 자세한 내용을 보려면 [CQD에서 스트림 분류](stream-classification-in-call-quality-dashboard.md)를 읽어보십시오.

> [!Note]
> CQD는 "낮은 기한 ..."을 제공 합니다. 측정값을 더 잘 이해 하기 위해 스트림을 불량으로 분류 한 상태입니다.

##### <a name="why-do-we-prefer-to-use-streams-instead-of-calls"></a>호출 대신 스트림을 사용 하는 것이 좋을 이유는 무엇 인가요?

스트림은 통화의 특정 레그 중 불량 또는 수신이 발생 한 것을 알 수 있습니다. 통화 분석에서 잘못 된 통화를 보고 있는 경우 해당 호출자의 스트림 (아웃 바운드) 또는 피호출자의 스트림 (인바운드)에 의해 불량 통화 인지 여부를 확인 하세요. 통화 음질에 영향을 주는 스트림을 결정 하는 것은 컨퍼런스에 더욱 중요 합니다. 통화 데이터만 표시 하는 경우에는 해당 사용자가 참여 하는 회의 수를 확인할 수 있지만, 어떤 사용자가 어떤 사람을 진행 하 고 있는지는 대부분 화면을 공유 합니다.

데이터를 호출 하면 사용 메트릭이 제공 되지만, 통화 품질이 좋지 않은 근본 원인으로 이어질 필요는 없습니다. 스트림 방향을 살펴보면 관리 되는 네트워크에 있지 않은 통화, 비 직원 (예: 공급 업체 또는 다른 네트워크의 다른 사용자)의 통화와 같은 요소를 식별할 수 있습니다. 이러한 경우, 다른 사용자의 네트워크 연결이 낮을 경우 전체 통화에 좋지 않은 플래그가 표시 됩니다. 외부 요인에 대 한 작업을 수행할 수 없기 때문에이 데이터가 도움이 되지 않습니다.

또한 스트림 방향은 문제가 있는 장치나 클라이언트를 식별 하는 데 도움이 됩니다.

 - 예를 들어 디바이스에 대해 제한 된 예산이 있고 고급 오디오 사용자 에게만 장치를 제공 하려는 경우 VoIP (오디오 사용 보고서) 및 아웃 바운드 스트림 및 회의 필터링을 사용 합니다. 기본 제공 마이크에 말하는 대용량 오디오 사용자를 찾습니다 .이는 poorer 통화 품질과 관련이 있으며 이러한 사람들을 위한 오디오 장치를 제공 하는 것이 좋습니다. 더 명확 하 게 하기 위해 패킷 사용률을 필터링 하 여 특히 고용량 오디오 사용자를 대상으로 할 수 있습니다. 

  - 또 다른 예로는 화면 공유가 있습니다. 고객이 이전 팀 클라이언트를 사용 하는 경우 화면 공유 성능이 영향을 받을 수 있습니다. 많은 화면 공유 사용자에 대 한 클라이언트 업그레이드 우선 순위를 매기기로이 문제를 해결할 수 있습니다.

 - 특정 스트림의 방향으로 인해 통화 품질이 저하 되는 경우 QoS 또는 대역폭 관련 문제가 있는지 확인할 수 있습니다. QoS를 완전히 구현 하지 않았거나, 인바운드 스트림이 아닌 클라이언트 에서만 패킷을 표시 하는 경우 poorer 통화 품질이 표시 될 수 있습니다. 스트림 방향을 확인 하 여 패킷 손실, 대기 시간 또는 지터를 지정 된 방향으로 더 세밀 하 게 볼 수 있습니다. 

   - 예를 들어 유선 연결 (지터)을 사용 하는 동안 로보틱 오디오의 사용자 complains 가정해 보겠습니다. 스트림과 방향을 살펴보면 특정 서브넷 집합에 대해서만 문제가 인바운드 스트림에서 발생 하는 것을 확인할 수 있습니다. 네트워킹 팀에이 정보를 부여한 후에는 미디어 트래픽을 우회 하지 않은 잘못 된 WAN 가속기를 통해이를 추적할 수 있습니다. 네트워크 팀이 WAN 가속기를 재구성 한 후에는 지터가 사라지고 통화 품질이 향상 됩니다. 


#### <a name="setup-failure-rate"></a>설치 실패 비율

CQD의 _총 통화 설정 실패 백분율_ 측정치 라고도 하는 설치 실패 비율은 통화 시작 시 끝점 간에 미디어 경로를 설정할 수 없는 스트림의 수입니다.

이는 설정할 수 없는 모든 미디어 스트림을 나타냅니다. 사용자 환경에 대 한이 문제의 영향에 대 한 심각도를 감안 하면이 값을 0에 가까운 수로 줄이는 것이 목표입니다. 이 메트릭의 값이 높으면 완벽 하 게 배포 되는 것 보다 더 많은 수의 방화벽 규칙이 적용 되지만, 정기적으로 감시 해야 하는 것은 여전히 중요 합니다.

이 메트릭은 CDR (성공적인 호출 정보 기록)를 제출한 총 스트림 수로 나누기를 설정 하지 못한 총 스트림 수를 가져와 계산 합니다.

-   **설치 실패 비율** = 총 통화 설정 실패 스트림 수/총 CDR 사용 가능한 스트림 수

#### <a name="drop-failure-rate"></a>실패 율의 낙하

CQD의 _총 통화 손실 실패 백분율_ 측정치 라고도 하는 낙하 오류 발생률은 미디어 경로가 정상적으로 종료 되지 않은 성공적으로 설정 된 스트림의 백분율입니다.

이는 예기치 않게 종료 된 모든 미디어 스트림을 나타냅니다. 이는 설정에 실패 한 스트림의 영향이 심각 하지는 않지만, 여전히 사용자 환경에 부정적인 영향을 줍니다. 갑작스런 빈번한 미디어는 사용자 환경에 심각한 영향을 미칠 수 있을 뿐 아니라 사용자가 다시 연결 하 여 생산성을 상실 (불편을 주지는 않음) 할 수도 있습니다.

메트릭은 성공적으로 설정 된 총 스트림 개수로 나눈 총 끌어 놓은 스트림의 수를 계산 합니다.

-   **낙하 실패 비율** = 총 통화 손실 됨 스트림 수/총 통화 설정 성공 스트림 수

### <a name="user-experience"></a>사용자 환경

여기에 수집 된 메트릭은 네트워크 또는 서비스에 문제가 있다는 것을 의미 하는 것이 아니라 사용자가 문제를 인식 하 고 있다는 것을 나타내므로 사용자 경험을 분석 하는 것은 과학 보다 더 많은 것입니다. CQD에는 전체 사용자 환경을 평가 하는 데 도움이 되는 기본 제공 설문 조사 메커니즘 (RMC)이 포함 되어 있습니다. RMC는 사용자의 관점에서 다음과 같은 질문에 대 한 통찰력을 제공 합니다.

-   솔루션을 사용 하는 방법을 알고 계십니까?
-   솔루션을 사용 하는 것이 쉽고 직관적 이며 일일 통신 요구를 지원 하나요?
-   해결 방법이 작업을 수행 하는 데 도움이 되나요?
-   솔루션의 전반적인 인식
-   어디에 있든 관계 없이 언제 든 지 솔루션을 사용할 수 있나요?
-   전화를 설정 하 고 유지 관리할 수 있나요?

#### <a name="rate-my-call"></a>통화 요금 평가 

내 통화 속도 (RMC)는 팀과 비즈니스용 Skype에 기본으로 제공 됩니다. 10 개 통화 마다 또는 10%가 표시 되 면 자동으로 팝업 됩니다. 이 간단한 설문 조사를 통해 사용자에 게 통화를 평가 하 고 통화 품질이 좋지 않은 이유에 대 한 약간의 컨텍스트를 제공 하는 것을 요청 합니다. 하나 또는 두 개의 등급이 낮은 것으로 간주 되 고, 3에서 4로 양호 하며 5는 매우 우수 합니다. 이는 lagging 지표 중 일부 지만, 서비스 메트릭에서 놓치지 못하는 uncovering 문제에 대 한 유용한 메트릭입니다.

> [!Note]
> 인간 요인은 사용자가 통화 품질이 좋을 때 설문 조사를 무시 하 고 통화 품질이 좋지 않을 때이를 입력 합니다. 결과적으로, 서비스 메트릭이 좋을 때에도 RMC 보고서가 낮은 쪽으로 기울어집니다.

CQD를 사용 하 여 RMC 사용자 응답을 보고 하 고, 예제 보고서가 CQD 템플릿에 포함 될 수 있습니다. 그러나이 문서에서는 자세히 설명 하지 않습니다. 

#### <a name="client-and-device-readiness"></a>클라이언트 및 장치 준비

사용자에 게 일관 되 고 긍정적인 사용자 환경을 제공 하는 데 도움이 되는 견고한 클라이언트 및 디바이스 전략이 필요 합니다. 각 준비 전략은 몇 가지 주요 원칙을 통해 구동 됩니다.

##### <a name="client-readiness"></a>클라이언트 준비

팀 클라이언트를 최신 상태로 유지 하면 사용자에 게 항상 최상의 환경을 제공할 수 있습니다. Microsoft는이 기능을 해제 한 경우를 제외 하 고는 [팀 클라이언트에 대 한 빈번한 업데이트](teams-client-update.md) (업데이트는 백그라운드에 자동으로 설치 됩니다. 권장 되지 않음)를 해제 합니다. 네트워크, 비디오, USB, 오디오 드라이버는 종종 간과 되 고 있으며 통화와 모임 품질에 영향을 줄 수 있으므로 패치 하는 것도 중요 합니다. 네트워크, Wi-fi, 비디오, USB, 오디오 드라이버를 현재 패치 관리 프로세스에 추가 하는 것이 좋습니다.


##### <a name="device-readiness"></a>디바이스 준비

한 가지 방법으로 장치 준비 전략 보다 사용자 환경에 영향을 줄 수 있습니다. 예를 들어 랩톱 스피커와 마이크를 사용 하는 사용자에 게는 통화와 모임에서 많은 백그라운드 소음이 발생 합니다. 팀은 거의 모든 장치에서 작동 하도록 설계 되었지만 장치 관련 문제가 있는 경우 [팀의 전화](phones-for-teams.md)를 확인 하세요.


### <a name="categories-of-quality"></a>품질 범주

Operationalize는 좋은 통화 및 모임 품질을 제공 하는 고품질 관리 관행 좋은 품질 관리 계획은 다음 범주를 다룹니다.

-   **네트워크:** PSR (불량 스트림 비율) 메트릭, TCP 사용, 유선 및 무선 서브넷, 그리고 HTTP 프록시 및 VPN 사용 식별에 중점을 둘 때의 오디오 음질

-   **끝점:** 오디오 장치 및 최신 클라이언트

-   **서비스 관리:** 이 범주는 다음 두 섹션으로 이루어집니다.

    -   첫 번째는 팀과 비즈니스용 Skype Online 서비스를 관리 하 고 유지 관리 하는 데 있어 Microsoft의 책임입니다.

    -   두 번째는 서비스에 인프라를 추가 하 여 조직에서 서비스에 대 한 신뢰할 수 있는 액세스를 보장 하기 위해 관리 되는 작업 (예: 새 Office 365 IP 주소에 대 한 빌드 정보 업데이트 및 방화벽 유지 관리)

![조직의 품질 범주 그래프](media/qerguide-image-categories.png "조직의 품질 범주 (서비스 관리, 끝점, 네트워크)입니다.")

품질을 유지 하기 위해 권장 하는 다음 작업 목록을 검토 합니다. 정기적으로 이러한 작업을 수행 해야 합니다 (예: 매주).

#### <a name="service-management-tasks"></a>서비스 관리 작업

이러한 작업은 포화 인터넷 링크 없이 서비스에 도달 하는 데 충분 한 대역폭이 있는지 확인 하 고, 모든 관리 네트워크 영역에서 QoS (서비스 품질)가 현재 위치에 있으며, [방화벽에 대 한 Office 365 IP 범위](https://aka.ms/o365ips)를 유지 하도록 하는 것입니다.

#### <a name="network-tasks"></a>네트워크 작업

네트워크 작업에는 안정성 및 품질 이라는 두 가지 범주가 있습니다. 안정성은 전화를 걸고 연결을 유지 하는 사용자의 기능을 측정 하는 데 중점을 둡니다. 품질은 팀에 게 전송 되는 집계 된 원격 분석 및 종료 후 사용자의 클라이언트에의 한 비즈니스용 Skype Online에 중점을 둔 것입니다. 

안정성이 사용자 환경에 미치는 중대 한 영향을 감안 하 여 품질을 시작 하기 전에 안정성 기준을 평가 하 고 조사 하는 것이 좋습니다. 

#### <a name="endpoints-tasks"></a>끝점 작업

이 범주의 기본 작업은 일반 [팀 클라이언트 업데이트](teams-client-update.md)에 대 한 장애를 제거 합니다. 기본적으로 팀은 자동으로 정기적으로 업데이트 됩니다 (권장 하지 않는 설정을 해제 하지 않는 한). 

또한 장치와 관련 된 문제를 식별할 때마다 장치를 모니터링 하 고 업데이트를 제공 해야 합니다.

## <a name="use-cqd-to-manage-call-quality"></a>CQD를 사용 하 여 통화 품질 관리

[CQD를 설정한](turning-on-and-using-call-quality-dashboard.md)후에는이를 사용 하 여 조직의 통화 및 모임 품질을 관리할 준비가 된 것입니다.

팀 성능에 대 한 대부분의 문제는 다음과 같은 범주로 분류 됩니다.

-   방화벽 또는 프록시 구성이 완료 되지 않음
-   불량 Wi-fi 서비스
-   대역폭 부족
-   VPN
-   일관성 없거나 오래 된 클라이언트 버전 및 드라이버
-   최적화 되지 않음 또는 내장 오디오 장치
-   문제가 있는 서브넷 또는 네트워크 장치

팀에서 이러한 영역을 평가 하 고 결함을 수정 하기 전에 시간을 사용 하는 경우 모든 사용자에 게 고품질 팀 환경을 유지 관리 하는 데 필요한 작업량을 줄일 수 있습니다. 팀 출시 준비를 위해 네트워크를 평가 하 고 팀에 대 한 [관리자](use-advisor-teams-roll-out.md) 를 읽고 [팀에 맞게 네트워크를 준비할](prepare-network.md)수 있습니다.

### <a name="expectations-using-cqd"></a>CQD 사용 예상

CQD (통화 품질 대시보드)를 사용 하 여 팀과 비즈니스용 Skype 서비스를 사용 하 여 발생 하는 통화 품질에 대 한 통찰력을 얻을 수 있습니다. CQD는 팀과 비즈니스용 Skype 관리자 및 네트워크 엔지니어가 네트워크를 최적화 하 고 품질, 안정성 및 사용자 환경에 대 한 가까운 시각을 유지할 수 있도록 설계 되었습니다. CQD는 전체 조직에 대 한 집계 원격 분석에서 전체적인 패턴이 명확 하 게 표시 될 수 있는지를 살펴봅니다. 이를 통해 정보를 확인 하 고 관리를 계획 합니다. CQD는 전반적인 품질, 안정성 및 사용자 환경에 대 한 통찰력을 제공 하는 메트릭의 보고서를 제공 합니다.

CQD는 경향과 서브넷을 분석 하는 데 유용 하지만 항상 지정 된 시나리오에 대 한 특정 원인을 제공 하지는 않습니다. CQD를 사용 하는 경우이를 이해 하 고 올바른 기대치를 설정 하는 것이 중요 합니다.

-   CQD는 모든 시나리오에 대 한 근본 원인을 제공 하지 않습니다.
-   CQD에는 전화 시스템 또는 오디오 회의 스트림이 포함 되지 않습니다.
-   CQD는 추세에 따라 추가 조사를 위해 영역을 호출 합니다.

### <a name="managed-versus-unmanaged-networks"></a>관리 및 관리 되지 않는 네트워크

기본적으로 CQD의 모든 끝점은 외부로 분류 됩니다. 빌드 파일이 도입 되는 즉시 관리 되는 끝점 데이터를 볼 수 있습니다. 앞에서 설명한 대로 CQD의 네트워크는 다음과 같이 정의 됩니다.

-   내부 또는 내부 라고도 하는 _관리 되는 네트워크_는 조직에 의해 영향을 받을 수 있습니다. 여기에는 내부 LAN, 원격 WAN, VPN이 포함 됩니다.
-   일반적으로 외부 또는 외부 라고도 하는 _관리 되지 않는 네트워크_는 조직에서 영향을 받지 않거나 제어할 수 없습니다. 관리 되지 않는 네트워크의 예로는 호텔 또는 공항 네트워크를 들 수 있습니다.

### <a name="dimensions-measures-and-filters"></a>차원, 측정값 및 필터

올바른 형식의 CQD 쿼리는 다음 세 가지 매개 변수를 모두 포함 합니다.

-   **차원:** 데이터를 피벗 하는 방법

-   **측정값:** 보고 하려는 항목

-   **필터:** 쿼리를 통해 반환 되는 데이터 집합을 줄이는 방법

이를 확인 하는 또 다른 방법은 _차원이_ 그룹화 함수이 고, _측정값_ 은 관심이 있는 데이터 이며, _필터가_ 결과를 쿼리에 관련 된 것으로 좁히는 방법입니다.

제대로 구성 된 쿼리의 예는 **건물 6 [Filter]에 대해 서브넷 [Dimension]에 따라 불량 스트림 [Measure]을 표시**합니다. 자세한 내용은 [CQD에서 사용할 수 있는 차원 및 측정값](https://aka.ms/cqd-dm)을 참조 하세요.

### <a name="first-vs-second"></a>첫 번째와 두 번째 비교 

CQD의 여러 차원과 측정값은 first 또는 second로 분류 됩니다. CQD는 호출자/호출 수신자 필드를 사용 하지 않으며, 호출자와 호출 수신자 사이에는 _다른_ 단계가 있으므로 _먼저_ 이름이 변경 되었습니다. 다음 논리는 관련 된 끝점에 첫 번째 레이블을 지정 합니다.

-   서버가 스트림이나 통화에 관여 하는 경우 **첫 번째** 는 항상 서버 끝점 (회의 서버, 중재 서버 등)입니다.

-   **두 서버** 끝점 사이에 스트림이 없으면 항상 클라이언트 끝점이 됩니다.

-   두 끝점의 유형이 같은 경우 첫 번째 항목은 사용자 에이전트 범주의 내부 순서 지정을 기준으로 합니다. 이렇게 하면 순서가 일치 합니다.

둘 다 같은 경우 첫 번째 또는 두 번째 끝점을 확인 하는 방법에 대 한 자세한 내용은 [CQD에서 사용할 수 있는 차원 및 측정값](https://aka.ms/cqd-dm)을 참조 하세요.

### <a name="stream-vs-call"></a>스트림 vs 통화

CQD에서 보고 싶은 치수나 측정값을 적절히 선택 하기 위해 호출과 스트림 간의 차이를 이해 해야 합니다. CQD의 기본 포커스는 스트림에 있지만, 통화 기반 측정도 사용할 수 있습니다.

-   **스트림:** _스트림이_ 두 끝점 사이에 있습니다. 각 방향에 대해 하나의 스트림만 있고 통신에는 두 개의 스트림이 필요 합니다. 스트림은 건물, 네트워크 또는 서브넷을 조사 하는 데 유용 합니다. 경우에 따라 호출 및 스트림을 측정 이름 (예: 호출 설정 스트림 또는 호출 끊김 스트림)에 사용 합니다. 이는 여전히 스트림으로 분류 되어 있습니다.

-   **통화:** _통화_ 는 모든 참가자의 모든 스트림을 그룹화 한 것입니다. 통화는 최소 두 개의 스트림으로 이루어집니다. 단일 호출에는 두 개 이상의 종점이 있으며, 각 끝점에는 최소한 하나의 스트림이 있습니다.

차원 또는 측정값이 호출이 나 스트림을 참조 하는지 여부에 대 한 자세한 내용은 [CQD에서 사용할 수 있는 차원 및 측정값](https://aka.ms/cqd-dm) 을 참조 하세요.

### <a name="good-poor-and-unclassified-calls"></a>좋음, 나쁨, 미분류 통화

통화는 정상, 나쁨 또는 분류 되지 않은 것으로 분류 됩니다. 각 항목에 대해 자세히 설명 하겠습니다.

-   **좋음 또는 나쁨:** 정상 또는 불량 통화는 서비스에 의해 전체 체감 품질 보고서가 생성 되 고 수신 된 전체 서비스 메트릭 집합을 포함 하는 호출로 구성 됩니다. [이 문서의 앞부분에](#poor-stream-rate)는 스트림의 정상 또는 불량 여부를 결정 하는 방법이 나와 있습니다.

-   **분류** 되지 않음: 분류 되지 않은 스트림은 전체 서비스 메트릭스 집합을 포함 하지 않습니다. 이것은 평균을 계산할 수 없고 체감 품질 보고서가 생성 되지 않은 짧은 통화 (일반적으로 60 초 미만) 일 수 있습니다. 미분류로 거는 것이 가장 일반적인 이유는 패킷 사용률이 거의 없다는 것입니다. 예를 들어 음소거로 모임에 참가 하 고 말할 수 없는 참가자가 있습니다. 참가자가 미디어를 받고 있지만 전송 하지는 않습니다. 미디어를 전송 하지 않으면 CQD에서 끝점의 아웃 바운드 미디어 스트림을 분류 하는 데 사용할 수 있는 메트릭이 없습니다.

자세한 내용을 보려면 [CQD에서 스트림 분류](stream-classification-in-call-quality-dashboard.md)를 읽어보십시오.

### <a name="common-subnets"></a>공통 서브넷

공용 서브넷은 호텔, 가정용 네트워크, 핫스폿, 유사한 영역에 사용 되는 특정 개인 서브넷입니다. 이러한 서브넷은 광범위 하 게 사용 되기 때문에 심사 하기가 어렵습니다. 조직에서 이러한 공통 서브넷 중 하나를 사용 하는 경우 해당 네트워크를 다른 서브넷으로 이동 하는 것이 좋습니다. 이를 통해 CQD에서 더욱 쉽게 보고할 수 있습니다. 참고로, 모든 네트워크 템플릿의 보고서는 이러한 서브넷을 제외 하 여 낮은 품질의 원본으로 제거 하도록 구성 되었습니다. 공통 서브넷은 다음과 같이 정의 됩니다. 조직의 영향은 조직에 따라 달라 집니다.

-   10.0.0.0/24
-   192.168.0.0/24
-   192.168.1.0/24
-   192.168.2.0/24
-   172.20.10.0/24
-   192.168.43.0/24

공용 서브넷을 사용 하는 관리 되는 네트워크를 조사할 때는 두 번째 재귀 로컬 IP 차원을 그룹 서브넷에 사용 해야 합니다. 이 차원에는 끝점의 공용 IP 주소가 포함 됩니다.


## <a name="reliability-investigations"></a>안정성 조사

품질을 개선 하는 첫 번째 단계는 조직에서 안정성의 상태를 평가 하는 것입니다. 안정성은 긍정적 사용자 환경에 필수적 이므로 안정성을 측정 하는 두 가지 구성 요소를 사용 하 여 시작 합니다.

1.  **설치 실패:** 통화를 설정할 수 없습니다.

2.  **삭제 오류:** 통화가 설정 되었으며 예기치 않게 종료 되었습니다.

이 섹션에서는 두 영역을 조사 하는 방법을 다루겠습니다.

> [!NOTE]
> 서식 파일에 포함 된 모든 보고서에는이 문서에서 다루지 않습니다. 그러나 아래에 설명 된 조사 방법에도 적용 됩니다. 자세한 내용은 개별 보고서 설명을 참조 하세요.


### <a name="setup-failures"></a>설치 실패

이러한 오류는 사용자 환경에 부정적인 영향을 수정 때문에 먼저이 영역에 대 한 설정 실패의 우선 순위를 지정 합니다.

조직의 전체 설치 실패 비율을 평가한 다음 건물 또는 네트워크 별로 가장 높은 백분율을 기준으로 조사 영역에 우선 순위를 지정 하 여 조사를 시작 합니다. 

#### <a name="setup-failure-trend-analysis"></a>설치 실패 추세 분석

이 보고서에는 스트림의 총 양, 스트림 설정 실패 및 스트림 설정 실패 비율이 표시 됩니다. 열 중 하나를 가리켜 개별 값을 표시 합니다. 

##### <a name="analysis"></a>분석

이 보고서를 사용 하 여 다음 질문에 답하고 다음 작업 과정을 확인할 수 있습니다.

-   현재 달의 총 통화 설정 실패 백분율은 무엇 인가요?

-   정의 된 대상 메트릭 이하의 총 통화 설정 실패 백분율입니다.

-   실패 추세가 지난 달 보다 악화 되거나 더 나은 가요?

-   실패 추세가 증가, 꾸준히, 감소 하는가?

이 질문에 대 한 답변에 관계 없이 도우미 하위 보고서를 사용 하 여 수정이 필요할 수 있는 개별 건물이 나 서브넷을 확인 하 여 더 많은 시간을 조사 하세요. 전체 오류율이 대상 메트릭 보다 아래에 있을 수 있지만 하나 이상의 건물 또는 네트워크에 대 한 실패 비율이 대상 메트릭 위에 있을 수 있으며 조사가 필요 합니다.

#### <a name="setup-failure-investigations"></a>설치 실패 조사 

이 요약 보고서는 수정이 필요할 수 있는 모든 건물이 나 네트워크를 검색 하 고 격리 하는 데 사용 됩니다.

> [!NOTE]
> 이번 달에 대 한 월 연도 보고서 필터를 조정 해야 합니다. **편집**을 선택 하 고 **월 연도** 보고서 필터를 조정 하 여 새 기본 달을 저장 합니다.

##### <a name="remediation"></a>업데이트 

가장 큰 실패가 발생 하는 건물이 나 서브넷에 대 한 첫 번째 재구성 노력을 집중 하세요. 이렇게 하면 사용자 환경에 미치는 영향을 극대화 하 고 조직의 통화 설정 실패 비율을 빠르게 줄일 수 있습니다. 다음 표에는 CQD에서 보고 한 설치 실패의 두 가지 이유가 나와 있습니다.

| 통화 설정 실패 이유       | 일반적인 원인                    |
|----------------------------------|----------------------------------|
| 누락 된 FW 딥 패킷 검사 예외 규칙 | 경로 상의 네트워크 장비에서 딥 패킷 검사 규칙으로 인해 미디어 경로가 설정 되지 않았음을 나타냅니다. 방화벽 규칙이 올바르게 구성 되지 않았기 때문일 수 있습니다. 이 시나리오에서는 TCP 핸드셰이크가 성공 했지만 SSL 핸드셰이크가 실패 했습니다.      |
| FW IP 차단 예외 규칙이 없음      | 경로를 따라 네트워크 장비에서 Microsoft 365 또는 Office 365 네트워크에 미디어 경로를 설정 하지 못하도록 지정 합니다. 이는 팀과 비즈니스용 Skype 트래픽에 사용 되는 IP 주소 및 포트에 대 한 액세스를 허용 하도록 프록시 또는 방화벽 규칙이 올바르게 구성 되지 않았기 때문일 수 있습니다. |

수정을 시작 하면 특정 건물이 나 서브넷에 집중할 수 있습니다. 앞의 표에 나와 있는 것 처럼 이러한 문제는 방화벽 또는 프록시 구성 때문에 발생 합니다. 다음 표의 옵션을 검토 하 여 업데이트 관리 작업을 수행 합니다.

|      업데이트      |지침  |
|-----------------------|----------|
| 방화벽 구성 | 네트워크 팀과 협력 하 여 [Office 365 IP 주소 목록](https://aka.ms/o365ips)에 대해 방화벽 구성을 확인 합니다.<br><br>[미디어 서브넷](https://support.office.com/article/Office-365-URLs-and-IP-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2#bkmk_teams) 및 포트가 방화벽 규칙에 포함 되어 있는지 확인 합니다. <br><br>[필요한 포트가](prepare-network.md) 방화벽에서 열려 있는지 확인 합니다. TCP는 오디오, 비디오 및 비디오 기반 화면 공유에 대 한 장애 복구 프로토콜로 간주 되 고 해당 사용은 통화 품질에 영향을 주므로 UDP는 우선 순위를 부여 해야 합니다. 레거시 RDP 응용 프로그램 공유는 TCP만 사용 합니다.|

### <a name="drop-failures"></a>놓기 실패

설치 오류 코드와 달리 CQD에는 드롭 실패가 발생 하는 이유를 설명 하는 손실 오류 코드가 없으므로 특정 근본 원인을 격리 하기가 어렵습니다. 삭제 오류를 더 잘 분류 하려면 유추 된 방법을 사용 합니다. 미디어에 관심 있는 영역을 수정, 클라이언트와 드라이버를 패치 하 고, 팀과 비즈니스용 Skype에 대해 인증 된 장치를 사용 하는 경우, 낙하에 대 한 장애가 발생 하는 것을 기대할 수 있습니다.

#### <a name="drop-failure-trend-analysis"></a>실패 추세 분석 놓기

이 보고서는 총 오디오 스트림 양, 총 저장 실패 횟수, 낙하 실패 율을 표시 합니다. 열 중 하나를 가리켜 해당 값을 표시 합니다. 


##### <a name="analysis"></a>분석

이러한 유형의 보고서를 사용 하 여 다음 질문에 대답할 수 있습니다.

-   현재 낙하에 대 한 실패 율은 무엇 인가요?
-   지정 된 대상 메트릭 이하의 낙하에 대 한 요금
-   실패 추세가 지난 달 보다 악화 되거나 더 나은 가요?
-   실패 추세가 증가, 꾸준히, 감소 하는가?

위의 질문에 대 한 답변에 관계 없이, 하위 보고서를 사용 하 여 수정이 필요한 건물이 나 네트워크를 검색 하는 시간을 조사 하세요. 전체 낙하 장애가 대상 메트릭 보다 아래에 있을 수 있지만 하나 이상의 건물 또는 네트워크에 대 한 오류율이 대상 메트릭 보다 높거나 조사가 필요할 수 있습니다.

#### <a name="drop-failure-investigations"></a>실패 조사 삭제

여기에 표시 된 오류는 예기치 않게 호출을 손실 하 여 부정적인 사용자 경험을 유발한 것입니다. 이 보고서는 추세 보고서와 달리 추가 조사가 필요한 특정 서브넷에 대 한 추가적인 정보를 제공 합니다.


##### <a name="remediation"></a>업데이트

포함 된 테이블 보고서를 사용 하면 낙하 비율이 정의한 대상 메트릭의 위쪽에 있는 네트워크의 문제 영역을 격리할 수 있습니다. 가장 높은 총 스트림 수가 있는 건물이 나 서브넷에 대 한 첫 번째 교정 노력을 집중 하 여 가장 큰 영향을 줍니다.

통화 낙하의 일반적인 원인:

-   프로비저닝된 네트워크 또는 인터넷 송신
-   제약이 있는 네트워크에 구성 된 QoS 없음
-   이전 클라이언트 버전
-   사용자 동작

문제 영역을 발견 한 후 [사용자 단위 통화 분석](use-call-analytics-to-troubleshoot-poor-call-quality.md) 을 사용 하 여 특정 문제에 대 한 해당 건물의 사용자를 자세히 검토할 수 있습니다. 통화 분석에는 추가 EUII 데이터가 포함 되어 있으며,이에 대 한 잠재적인 오류 발생 이유를 더 자세히 격리 하는 데 유용할 수 있습니다.

다음 단계에 관계 없이 특정 건물이 나 서브넷에서 문제가 발견 되었음을 헬프 데스크에 알리는 것이 좋습니다. 이렇게 하면 헬프 데스크는 수신 전화에 신속 하 게 응답 하 고 사용자에 게 더욱 효율적으로 응답할 수 있습니다. 그러면 플래그가 지정 된 사용자는 자세히 조사를 위해 엔지니어링 팀에 게 다시 보고할 수 있습니다.

다음 표에는 삭제 오류를 관리 하 고 수정 하는 몇 가지 일반적인 방법이 나와 있습니다.

| 업데이트                              | 지침                      |
|------------------------------------------|-------------------------------|
| **네트워크/인터넷**                         | **정체**: 네트워크 팀과 협력 하 여 특정 건물/서브넷의 대역폭을 모니터링 하 여 초과 사용에 문제가 있음을 확인 합니다. 네트워크 혼잡이 있는지 확인 하는 경우 해당 빌드 또는 QoS 적용에 대 한 대역폭을 늘리는 것이 좋습니다. 포함 된 [품질 불량 스트림 요약 보고서](#quality-investigations) 를 사용 하 여 문제 서브넷에서 지터, 대기 시간, 패킷 손실 등의 문제를 검토 하는 것이 일반적으로 손실 된 스트림입니다 앞에 있습니다.<br><br>**QoS**: 대역폭 증가가 실용적이 지 않거나 비용이 너무 많은 경우 qos를 구현 하는 것이 좋습니다. 이 도구는 혼잡 된 트래픽을 관리 하는 데 매우 효과적 이며, 관리 되는 네트워크의 미디어 패킷이 비 미디어 트래픽 보다 우선 순위를 가질 수 있도록 보장 합니다. 또는 대역폭의 원인에 대 한 명확한 증거가 없다면 다음 해결 방법을 고려 하세요.<ul><li>[Microsoft 팀 QoS 지침](qos-in-teams.md)</li></ul><br>**네트워크 준비 평가 수행**: 네트워크 평가는 예상 대역폭 사용량, 대역폭과 네트워크 변경에 대처 하는 방법, 팀과 비즈니스용 Skype에 대해 권장 되는 네트워킹 방법에 대 한 세부 정보를 제공 합니다. 앞의 표를 원본으로 사용 하면 평가에 대 한 훌륭한 후보자 인 건물이 나 서브넷 목록이 표시 됩니다.<ul><li>[Teams에 대한 조직의 네트워크 준비](prepare-network.md)</li></ul> |
| **클라이언트 (비즈니스용 Skype Online에만 해당)** | 일부 구형 비즈니스용 Skype 클라이언트는 미디어 신뢰성과 관련 하 여 알려지지 않은 알려진 문제를가지고 있습니다. 영향을 받는 여러 사용자의 통화 분석 보고서를 검토 하거나 총 통화 손실 실패% 측정값을 사용 하 여 특정 건물 또는 서브넷으로 필터링 된 CQD에서 사용자 지정 클라이언트 버전 테이블 보고서를 만듭니다. 이 정보는 특정 건물에서 통화를 사용 하는 경우와 특정 버전의 클라이언트에 관계가 있는지를 이해 하는 데 도움이 됩니다.     |

## <a name="quality-investigations"></a>품질 조사

조직 전체의 오디오 품질 상태를 평가 하는 다음 단계는 불량 스트림 속도 (PSR), TCP 및 프록시 사용을 조사 하는 것입니다. CQD 데이터는 특정 근본 원인을 제공 하지 않으며 대신 문제 해결을 위해 적절 한 팀과 공동 작업을 시작 하는 데 문제가 있을 수 있습니다. 

> [!NOTE]
> 서식 파일에 포함 된 모든 보고서에 대해서는이 문서에서 설명 합니다. 그러나 아래 설명 된 조사 방법은 해당 보고서에도 적용 됩니다. 자세한 내용은 개별 보고서 설명을 참조 하세요. 

### <a name="quality"></a>화질은

PSR 비율은 조직이 지정 된 포커스 영역에 대해 정의 된 미터법 대상을 충족 하는지 여부를 나타내는 데 사용 됩니다. 상위 수준 백분율이 정의 된 대상 내에 있는 경우에도 개별 서브넷 또는 건물이 정의 된 대상에 맞지 않을 수 있으므로 추가 조사가 필요 합니다. 예를 들어 4 월의 전체 오디오 PSR 백분율이 2% 인 경우 샘플 대상을 충족 하는 경우, 개별 건물과 서브넷에는 해당 2%의 전체적인 분포에 따라 문제가 지속 될 수 있습니다. 

불량 스트림의 백분율을 평가 하려면 품질 보고서를 사용 합니다. 전체, 회의, 2-파티, PSTN 통화, VPN, 회의실에 대 한 메트릭을 검토 하기 위해 다양 한 품질 보고서가 제공 됩니다. 이 프로세스를 지원 하기 위해 월, 주, 일 보고서가 제공 됩니다. 주간 및 일일 보고서의 효율성을 높이고 노이즈를 줄이려면 관리 네트워크 서식 파일로 제한 됩니다. 

#### <a name="quality-trend-analysis"></a>품질 추세 분석

추세 보고서에는 시간에 따른 품질 정보가 표시 되며 각 관심 영역 내의 품질 경향을 식별 하 고 이해 하는 데 도움이 됩니다. 위에서 언급 한 것 처럼 품질을 조사 하는 데 사용할 수 있는 보고서 트리가 서식 파일에 포함 되어 있습니다. 회의, 2-파티, PSTN 통화, VPN, 회의실. 품질 분석을 위해 investigative 프로세스는 동일 합니다. 그러나, 컨퍼런스 품질이 향상 되어 다른 모든 영역에도 영향을 미칠 수 있으므로 먼저 회의를 시작 하는 것이 좋습니다. 

> [!Note]
> 2-파티, PSTN 통화, 회의실을 조사 하는 것은 회의를 조사 하는 것과 비슷합니다. 포커스는 낮은 품질의 건물이 나 서브넷을 분리 하 고 품질이 좋지 않은 이유를 식별 하는 것입니다.

> [!Important]
> VPN 기반 보고서는 두 번째 VPN 차원을 사용 하 여 필터링 됩니다. 이 차원을 사용 하려면 VPN 네트워크 어댑터가 원격 액세스 어댑터로 올바르게 등록 되어 있어야 합니다. VPN 공급 업체는이 플래그를 안정적으로 사용 하지 않으며, 조직에 배포 된 VPN 공급 업체에 따라 거리는 달라 집니다. 필요한 경우 건물 또는 네트워크 이름을 사용 하 여 [VPN](CQD-upload-tenant-building-data.md#vpn) 보고서를 수정 합니다.

##### <a name="investigation"></a>검사

이러한 보고서를 사용 하 여 다음 질문에 대답할 수 있습니다.

-   현재 달의 총 PSR은 무엇 인가요?
-   지정 된 대상 메트릭 아래에 PSR이 있습니까?
-   지난 달에 비해 PSR의 성능이 떨어질 수 있나요?
-   PSR 추세가 증가, 일정 대로 또는 감소 되나요?

위의 질문에 대 한 답변에 관계 없이 조사 해야 하는 건물이 나 서브넷을 찾기 위해 하위 보고서를 사용 하 여 조사 하는 시간을 고려해 야 합니다. 전체 PSR이 대상 메트릭 보다 아래에 있을 수 있지만 하나 이상의 건물 또는 네트워크에 대 한 PSR이 메트릭 위에 있으며 수정이 필요 합니다.

#### <a name="quality-investigations"></a>품질 조사

품질 요약 보고서에서는 스트림에 기여 하는 부분에 대 한 자세한 정보를 제공 하 고 관리 되는 네트워크의 문제 영역을 구분 하는 데 도움을 줍니다.

사용 되는 차원은 보고서 간에 약간씩 다를 수 있지만, 각 보고서에는 총 스트림, 총 불량 스트림, PSR, 그리고 그에 따른 낮은 품질에 대 한 측정값이 포함 됩니다. 회의, 2 면, PSTN 통화, VPN, 회의실 등과 관련 하 여 각 영역에 대 한 보고서를 만들었습니다. 관리 되는 네트워크 템플릿에는 빌드 파일을 통해 업로드 된 위치 정보를 활용 하는 추가 보고서가 포함 됩니다.


> [!Note]
> 공통 서브넷은 널리 사용 되기 때문에 심사 하기가 어렵습니다. 클라이언트의 공용 IP를 표시 하는 별도의 보고서 (두 번째 재귀 로컬 IP)는 일반 네트워크를 사용 하는 수정 사무소를 지원 하기 위해 모든 네트워크 서식 파일에 추가 되었습니다.


![불량 오디오 스트림 요약을 보여 주는 스크린샷](media/qerguide-image-poorqualitysummary.png)

##### <a name="remediation"></a>업데이트

가장 큰 스트림을 보유 하는 건물이 나 서브넷에 대 한 수정 노력에 중점을 두기 때문에이로 인해 영향이 최대화 되 고 사용자 환경을 빠르게 개선 하는 데 도움이 됩니다. 지터, 패킷 손실, RTT (왕복 시간) 측정값을 사용 하 여 낮은 품질로 기여 하는 사항을 파악 합니다 (둘 이상의 문제가 있을 수 있음).

-   **지터**: 미디어 패킷이 다른 속도로 도달 하 여 스피커가 사운드를 재생 하도록 합니다.
-   **패킷 손실**: 미디어 패킷이 손실 되어 누락 된 단어나 음절의 효과를 생성 합니다.
-   **RTT**: 미디어 패킷이 목적지에 도달 하는 데 시간이 오래 걸려 walkie-talkie 효과를 생성 합니다.

품질 문제를 조사 하는 데 도움이 되도록 [사용자 단위 통화 분석](use-call-analytics-to-troubleshoot-poor-call-quality.md)을 사용 합니다. 통화 분석을 사용 하 여 특정 컨퍼런스 또는 사용자의 통화 보고서를 볼 수 있습니다. 이 보고서는 EUII/PII 데이터를 포함 하며 오류 원인을 찾을 때 유용 합니다. 영향을 받는 빌드를 확인 한 후 해당 건물에서 사용자를 추적 하는 것은 간단 합니다. 

이 네트워크에 품질 문제가 발생 하는 것을 헬프 데스크에 알려 주고 걸려오는 전화를 신속 하 게 심사 하 고 응답할 수 있도록 하는 것을 잊지 마세요.

### <a name="tcp"></a>NET.TCP 

TCP (전송 제어 프로토콜)는 실시간 미디어에 대해 원하는 기본 전송이 아닌 장애 복구 전송으로 간주 됩니다. 이는 장애 복구 (failback) 전송이 TCP의 상태 저장 특성으로 인해 발생 하는 이유입니다. 예를 들어, 숨어 있는 네트워크에 대 한 호출이 지연 되는 경우, 더 이상 유용 하지 않는 몇 초 전에는 패킷이 수신기에 도달 하기 위해 대역폭을 경합 하기 때문에 잘못 된 상황이 발생할 수 있습니다. 이렇게 하면 오디오 healer 오디오를 걸고 늘이고, 결과적으로 일반적으로 지터 형태를 사용 하는 소리가 들 리 게 됩니다.

이 섹션의 보고서는 양호한 스트림과 불량 스트림을 구분 하지 않습니다. UDP를 사용 하는 것이 좋습니다 .이 보고서는 오디오, 비디오, 비디오 기반 화면 공유 (VBSS)에 대해 TCP를 사용할 수 있는지 확인 합니다. 해당 영향이 가장 큰 부분에 집중할 수 있도록 UDP 품질과 TCP 품질을 비교 하는 데 도움이 되는 불량 스트림 요금이 제공 됩니다. TCP 사용은 기본적으로 불완전 한 방화벽 규칙으로 인해 발생 합니다. 팀과 비즈니스용 Skype Online에 대 한 방화벽 규칙에 대 한 자세한 내용은 [Microsoft 365 및 Office 365 url 및 IP 주소 범위](https://aka.ms/o365ips)를 참조 하세요.

> [!Note]
> 오디오, 비디오 및 VBSS 모두 기본 전송으로 UDP를 선호 합니다. 레거시 RDP 응용 프로그램 공유 작업 부하는 TCP만 사용 합니다.

#### <a name="tcp-usage"></a>TCP 사용

TCP 보고서는 지난 7 개월 동안의 전체적인 TCP 사용량을 나타냅니다. 이 섹션의 모든 추가 보고서는 TCP가 가장 일반적으로 사용 되는 특정 건물과 서브넷의 범위를 좁히는 데 중점을 둔 것입니다. 회의 및 2-파티 스트림 모두에 대해 별도의 보고서를 사용할 수 있습니다.

![TCP를 사용 하는 오디오 스트림의 백분율을 보여 주는 차트](media/qerguide-image-audiostreamswithtcp.png)

##### <a name="investigation"></a>검사

이 보고서를 사용 하 여 다음 질문에 대답할 수 있습니다.

-   현재 달에 대 한 TCP 스트림의 총 량은 얼마 입니까?
-   지난 달에는 더 이상 하거나 좋은 가요?
-   TCP 사용량 추이는 증가, 일정에 또는 감소 되나요?
-   TCP PSR이 내 전체 PSR과 동일 한가요?

TCP 사용량 추세가 보통 월 사용량 보다 높거나 높은 경우, 검토를 필요로 하는 모든 건물이 나 네트워크를 조회 하 여 하위 보고서를 사용 하 여 조사 하는 데 시간을 찍습니다. 원칙적으로는 관리 되는 네트워크에서 몇 가지 TCP 기반 오디오 세션을 최대한 사용할 수 있습니다.

##### <a name="analysis"></a>분석

TCP 사용이 가능한 한 낮게 표시 되기는 하지만, 그렇지 않은 경우에는 정상적인 배포로 인해 TCP 사용량이 약간 다를 수 있습니다. TCP는 불량 통화에 기여 하지 않으므로 TCP 사용량이 품질이 좋지 않은 참가자 인지 여부를 식별 하는 데 도움이 되도록 스트림 요금이 제공 됩니다. 

#### <a name="tcp-investigations"></a>TCP 조사

제공 된 CQD 템플릿에서, 관리 네트워크 또는 모든 네트워크 템플릿을 사용 하 여 보고서를 작성 하 고 서브넷으로 이동 하 여 TCP 스트림을 탐색 합니다. TCP 사용을 조사 하는 데는 프로세스가 동일 하므로 여기서는 회의에 대 한 논의에 집중 하겠습니다.


##### <a name="remediation"></a>업데이트

이 보고서는 TCP 사용량의 볼륨에 기여 하는 특정 건물과 서브넷을 식별 합니다. 누락 된 방화벽 규칙을 격리 하는 데 도움이 되는 통화에 사용 된 Microsoft 릴레이 IP를 식별 하는 추가 보고서도 포함 됩니다. 가장 높은 TCP 스트림 볼륨을 보유 하 고 있는 건물에 대 한 수정 노력을 집중 하 여 영향을 최대화 하세요.

가장 일반적으로 사용 되는 방화벽 또는 프록시에는 발생 하는 예외 규칙이 없습니다. 이번에는 다음 섹션의 프록시에 대해 설명 하므로 이제 방화벽에 대 한 노력에 집중 하 게 됩니다. 제공 된 건물이 나 subnet을 사용 하 여 업데이트 해야 하는 방화벽을 결정할 수 있습니다.

| 업데이트        | 지침     |
|--------------------|--------------------------------------|
| 방화벽 구성 | [Microsoft 365 또는 Office 365 IP 포트 및 주소가](https://aka.ms/o365ips) 방화벽에서 제외 되는지 확인 합니다. 미디어 관련 TCP 문제는 다음에 대 한 초기 노력을 집중 하세요.<ul><li>클라이언트 미디어 서브넷 13.107.64.0/18 및 52.112.0.0/14가 방화벽 규칙에 있는지 확인 합니다.</li><li>UDP 포트 3478 – 3481는 필수 미디어 포트 이며, 그렇지 않은 경우 클라이언트는 다시 TCP 포트 443로 장애 조치를 수행 합니다.</li></ul> |
| 나타나는지             | [Microsoft 네트워크 평가 도구](https://www.microsoft.com/download/details.aspx?id=53885) 를 사용 하 여 영향을 받는 건물 또는 서브넷의 특정 Microsoft 365 또는 OFFICE 365 IP 주소 및 포트에 대 한 연결 문제가 있는지 확인 합니다.    |

### <a name="http-proxy"></a>HTTP 프록시

HTTP 프록시는 여러 가지 이유로 미디어 세션을 설정 하기 위한 기본 경로가 아닙니다. 대부분의 경우 서비스에 대 한 연결이 완료 되는 것을 방지 하 고 중단을 발생 시킬 수 있는 심층 패킷 검사 기능이 포함 되어 있습니다. 또한 거의 모든 프록시는 UDP를 허용 하는 대신 TCP를 강제 하 여 최적의 오디오 음질을 권장 합니다.

항상 팀과 비즈니스용 Skype 서비스에 직접 연결 하도록 클라이언트를 구성 하는 것이 좋습니다. 이는 미디어 기반 트래픽에 특히 중요 합니다.


> [!IMPORTANT]
> 프록시 사용을 분석할 때 외부 오디오 스트림 내에서 구별할 수 있도록 [올바른 빌드 파일](CQD-upload-tenant-building-data.md) 을 업로드 하는 것이 좋습니다. 


#### <a name="http-proxy-usage"></a>HTTP 프록시 사용

이 서식 파일 섹션의 HTTP 프록시 스트림 보고서는 TCP 보고서와 매우 유사 합니다. 전화가 불량 하거나 양호한 지만 호출이 HTTP를 통해 연결 되었는지 여부는 확인 하지 않습니다.

![HTTP를 사용 하는 오디오 스트림 보고서 스크린샷](media/qerguide-image-audiostreamswithhttp.png)

##### <a name="analysis"></a>분석

가능 하면 몇 가지 HTTP 미디어 스트림을 표시 하려고 합니다. 스트림이 프록시를 통과 하는 경우에는 네트워킹 팀에 문의 하 여 클라이언트가 팀 또는 비즈니스용 Skype Online 미디어 서브넷으로 직접 라우팅 되도록 적절 한 제외가 적절 한지 확인 합니다.

조직에 인터넷 프록시가 하나만 있는 경우 적절 한 [Microsoft 365 또는 Office 365 url 및 IP 주소 범위 제외](https://aka.ms/o365ips)를 확인 합니다. 조직에 여러 인터넷 프록시가 구성 되어 있는 경우 HTTP 하위 보고서를 사용 하 여 영향을 받는 빌드 또는 서브넷을 격리 합니다.

프록시를 우회 하지 못하는 조직의 경우 비즈니스용 Skype 클라이언트에서 [직접 연결을 시도 하는 대신 프록시 서버를 사용 하 여 로그인 해야](https://support.microsoft.com/help/3207112/skype-for-business-should-use-proxy-server-to-sign-in-instead-of-tryin)하는 경우에는 프록시 뒤에 위치 하 여 로그인 하도록 구성 되어 있는지 확인 합니다. 


#### <a name="http-proxy-investigations"></a>HTTP 프록시 조사

이 보고서는 HTTP 사용에 기여 하는 특정 건물과 서브넷을 식별 합니다.


##### <a name="remediation"></a>업데이트

항상 비즈니스용 Skype와 팀 (특히 미디어 트래픽)에 대 한 프록시를 우회 하는 [것이 좋습니다](proxy-servers-for-skype-for-business-online.md) . 프록시는 트래픽이 이미 암호화 되어 있기 때문에 비즈니스용 Skype를 더 안전 하 게 만들 수 없습니다. 성능 관련 문제는 대기 시간 및 패킷 손실을 통해 환경에 도입 될 수 있습니다. 이러한 문제는 실시간 스트림이 필수적인 오디오, 비디오 및 화면 공유에 대 한 부정적 경험을 초래 합니다.

HTTP 사용의 가장 일반적인 원인에는 프록시에서 발생 하는 예외 규칙이 없습니다. 제공 된 건물이 나 subnet을 사용 하 여 미디어 바이패스에 대해 구성 해야 할 프록시를 빠르게 결정할 수 있습니다.

필요한 [Microsoft 365 또는 Office 365 fqdn](https://aka.ms/o365ips) 이 프록시에서 허용 목록 확인 합니다.

### <a name="client-versions"></a>클라이언트 버전

이러한 보고서는 사용 중인 비즈니스용 Skype 클라이언트 버전을 식별 하 고 환경에서 상대 볼륨을 확인 하는 데 집중 됩니다.

> [!IMPORTANT]
> 현재 팀 클라이언트는 Azure 콘텐츠 배달 네트워크를 통해 자동으로 배포 되 고 업데이트 되며 서비스에 의해 최신 상태로 유지 됩니다. 결과적으로, 자동 업데이트를 해제 하지 않는 한 팀 클라이언트 버전을 모니터링할 필요가 없습니다 (권장 하지 않음).

페더레이션 참가자 데이터를 제외 하지 않는 한 이러한 보고서에는 페더레이션된 끝점의 클라이언트 원격 분석이 포함 됩니다. 페더레이션 끝점을 제외 하려면 조직의 [테 넌 트 id](CQD-data-and-reports.md#how-to-find-your-tenant-id)에 설정 된 두 번째 테 넌 트 id에 대 한 쿼리 필터를 추가 해야 합니다. 또는 [URL 필터](CQD-data-and-reports.md#url-filters) 를 사용 하 여 페더레이션된 참가자 원격 분석을 제외할 수 있습니다.



#### <a name="remediation"></a>업데이트

고품질 사용자 환경에서 중요 한 부분은 관리 되는 클라이언트가 최신 버전의 비즈니스용 Skype를 실행 하 고 있는지 확인 하는 것과 지원 되는 오디오, 비디오, 네트워크 및 USB 드라이버가 최신 상태 인지 확인 하는 것입니다. 이는 여러 가지 이점을 제공 합니다. 

-   일부 버전 및 여러 버전을 관리 하는 것이 더 쉽습니다.
-   이는 경험 수준의 일관성을 제공 합니다.
-   이를 통해 통화 품질과 편리성 때문에 문제를 쉽게 해결할 수 있습니다.
-   Microsoft는 제품 전반에 걸친 일반적인 개선 및 최적화 기능을 지속적으로 수행 합니다. 사용자가 이러한 업데이트를 받을 수 있도록 하 여 이미 해결 된 문제가 발생할 위험을 줄일 수 있습니다.

배포를 6 개월 미만의 클라이언트 버전으로 제한 하면 전반적인 사용자 환경이 개선 되 고 지원 해야 하는 버전 수가 줄어들기 때문에 관리 효율성이 향상 됩니다.

Office 간편 실행만 사용 하는 경우에는 6 개월 내에 자동으로 표시 됩니다. 추가 작업은 필요 하지 않습니다.

간편 실행 및 installer 패키지 (MSI)를 혼합 한 경우 보고서를 사용 하 여 MSI 클라이언트가 정기적으로 업데이트 되 고 있는지 확인할 수 있습니다. 클라이언트가 지연 되는 경우 Office 업데이트를 관리 하는 팀과 협력 하 여 클라이언트 패치를 정기적으로 승인 하 고 배포 하 고 있는지 확인 합니다.

또한 네트워크, 비디오, USB 및 오디오 드라이버에도 패치가 적용 되는지 확인 하는 것이 중요 합니다. 이러한 드라이버를 간과 하 고 패치 관리 전략에 포함 하지 않는 것이 더 쉬울 수 있습니다.

-   [엔터프라이즈 용 Microsoft 365 앱에 대 한 업데이트 기록](https://docs.microsoft.com/officeupdates/update-history-office365-proplus-by-date)
-   [비즈니스용 Skype 다운로드 및 업데이트](/SkypeForBusiness/software-updates)

### <a name="devices"></a>디바이스

마이크 장치 보고서를 사용 하려면 SPECIALIST (평균 의견 성과)의 개념을 이해 해야 합니다. SPECIALIST는 인식 되는 음질을 측정 하는 골드 표준 단위입니다. 0에서 5 까지의 정수 등급으로 표시 됩니다.

음성 품질의 모든 측정값은 사용자가 음성 품질을 인식 하는 방법을 기반으로 합니다. 이는 사람의 인식에 영향을 받지 않으므로 본질적으로 주관적입니다. 주관적인 테스트에는 여러 가지 다른 방법론을 사용할 수 있습니다. 대부분의 음성 음질 측정은 ACR (절대 분류 등급) 배율을 기반으로 합니다.

ACR 주관적인 테스트에서 통계적으로 중요 한 사용자 수는 1 (잘못 됨)에서 5 (매우 좋음) 까지의 규모에 대 한 경험을 평가 합니다. 점수 평균은 SPECIALIST입니다. 결과 SPECIALIST 그룹에 노출 된 환경과 평가할 경험의 범위에 따라 달라 집니다.

실시간 통신 시스템의 음성 품질에 대 한 주관적인 테스트를 수행 하는 것은 실용적이 지 않기 때문에 Microsoft 팀과 비즈니스용 Skype는 고급 알고리즘을 사용 하 여 주관적인 테스트의 결과를 객관적으로 예측 하 여 SPECIALIST 값을 생성 합니다.

사용할 수 있는 SPECIALIST 및 연결 된 메트릭은 오디오 장치를 통해 사용자에 게 제공 되는 환경의 품질에 대 한 보기를 제공 합니다. 

사용자에 게 팀과 비즈니스용 Skype에 대 한 인증 장치를 제공 하면 장치 자체에 따라 부정적인 문제가 발생할 가능성을 줄일 수 있습니다 (예: 내장형 랩탑 스피커 및 마이크 사용). 자세한 내용은 [인증 프로그램](/SkypeForBusiness/certification/overview) 및 [파트너 솔루션 카탈로그](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)에 대 한 문서를 참조 하세요.

장치 보고서는 볼륨 및 SPECIALIST 점수 (오디오만)로 디바이스 사용량을 평가 하는 데 사용 되며, 클라이언트 & 장치에서 함께 제공 된 템플릿에서 찾을 수 있습니다. 

> [!IMPORTANT]
> 페더레이션 참가자 데이터를 제외 하지 않는 한 이러한 보고서에는 페더레이션된 끝점의 클라이언트 원격 분석이 포함 됩니다. 페더레이션 끝점을 제외 하려면 조직의 [테 넌 트 id](CQD-data-and-reports.md#how-to-find-your-tenant-id)에 설정 된 **두 번째 테 넌 트 id** 에 대 한 쿼리 필터를 추가 해야 합니다. 또는 [URL 필터](CQD-data-and-reports.md#url-filters) 를 사용 하 여 페더레이션된 참가자 원격 분석을 제외할 수 있습니다.


> [!Note]
> 이 보고서를 보면서 같은 장치가 여러 번 보고 되었음을 알 수 있습니다. 이는 디바이스가 CQD에 보고 되는 방식 때문입니다. 하드웨어 및 OS 로캘의 차이로 인해 디바이스 데이터가 보고 되는 방식이 다릅니다.

##### <a name="remediation"></a>업데이트

일반적으로 인증 되지 않은 장치를 검색 하 고 단계별로 진행 하 여 인증 된 장치로 교체해 야 합니다. 장치 보고서를 검토할 때 고려해 야 할 사항은 다음과 같습니다.

-   장치를 팀과 비즈니스용 Skype에 대해 인증 했습니까? 
-   사용자별 [통화 분석](use-call-analytics-to-troubleshoot-poor-call-quality.md)을 사용 하 여 특정 디바이스의 사용자를 식별할 수 있습니다. 최신 장치 드라이버가 있고 장치가 USB 허브나 도킹 스테이션을 통해 연결 되지 않았는지 확인 합니다. 
-   사용 중인 다양 한 드라이버의 여러 버전이 얼마나 되나요? 정기적으로 패치가 적용 되나요? 오디오, 비디오 및 Wi-fi 드라이버는 정기적으로 패치 되는 것을 방지 하 여 품질 문제의 원인으로 제거 하 고 사용자 환경을 예측 가능 하 고 일관 되 게 만들 수 있습니다.

##### <a name="audio"></a>오디오

다음 작업은 [정식 오디오 장치의](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs)전체적인 사용을 확인 하는 것입니다. 적어도 80%의 모든 오디오 스트림은 정식 오디오 장치를 사용 하는 것이 좋습니다. 이 작업을 수행 하는 것이 가장 좋은 방법은, 마이크 장치 보고서를 Excel로 내보내 정식 또는 승인 디바이스의 사용을 계산 하는 것입니다. 조직에서는 일반적으로 승인 된 모든 디바이스 목록을 유지 하므로 데이터를 필터링 하 고 정렬 하는 것은 간단 합니다.

##### <a name="video"></a>비디오

비디오 드라이버는 업데이트 된 상태로 유지 하는 것이 중요 합니다. 비디오 카드가 정기적으로 패치 되 고 있는지 확인 하면 비디오 드라이버를 비디오 스트림 품질이 좋지 않은 원본으로 제외할 수 있습니다. [인증 된 비디오 장치](https://partnersolutions.skypeforbusiness.com/solutionscatalog/personal-peripherals-pcs) 를 사용 하면 원활한 고품질 사용자 환경을 유지할 수 있습니다. 비디오 회의 중 CPU 사용량을 줄이기 위해 H-264 네이티브 인코딩을 지 원하는 비디오 장치를 사용 하는 것이 좋습니다.

##### <a name="wi-fi"></a>Wi-fi

Wi-fi 드라이버는 일반 흐름에도 패치를 적용 해야 하며 패치 관리 전략에 포함 시켜야 합니다. 최신 Wi-fi 드라이버를 유지 관리 하 여 다양 한 품질 문제를 해결할 수 있습니다. Wi-fi 인프라를 최적화 하는 방법에 대 한 자세한 내용은 [wi-fi 계획에 대 한이 문서](/skypeforbusiness/certification/networking-wifi)를 참조 하세요.
