### <a name="understanding-the-types-of-ip-network-services--basic-ip-and-mpls"></a><span data-ttu-id="71efa-736">IP Network Services의 유형 이해 - 기본 IP 및 MPLS</span><span class="sxs-lookup"><span data-stu-id="71efa-736">Understanding the types of IP Network Services- Basic IP and MPLS</span></span>

<span data-ttu-id="71efa-737">원래 IP 패킷 전달은 "최선의 노력"의 원칙에 따라 작동됩니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-737">The original IP packet forwarding operated on the principle of "best effort."</span></span> <span data-ttu-id="71efa-738">즉, 해당 IP 패킷을 전달하는 라우터가 해당 대상에 전달하기 위해 최선을 다하지만 목적지에 도착하는 경우와 관련하여 보장할 수 없습니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-738">That meant that the routers forwarding those IP packets would do their best to deliver them to their destinations, but there was absolutely no guarantee with regard to when or if they would arrive at their destinations.</span></span> <span data-ttu-id="71efa-739">가정 인터넷 연결을 비롯한 기본 인터넷 서비스가 오늘 작동하는 방식입니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-739">That is how basic Internet services, including your home Internet connection, work today.</span></span> <span data-ttu-id="71efa-740">특정 애플리케이션에 대한 안정성이 필요한 경우 프로토콜 스택에서 더 높은 수준에서 제공될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-740">The idea was that if reliability was required for a particular application, it would be provided at a higher level in the protocol stack.</span></span> <span data-ttu-id="71efa-741">신뢰할 수 있는 배달 메커니즘은 TCP(전송 제어 프로토콜)입니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-741">The reliable delivery mechanism is the Transmission Control Protocol (TCP).</span></span> <span data-ttu-id="71efa-742">실시간 음성 및 비디오에 사용되는 UDP(사용자 데이터그램 프로토콜)는 신뢰할 수 없는(예: "최상의 노력") 배달 메커니즘입니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-742">The User Datagram Protocol (UDP), which is used for real-time voice and video, is the unreliable (i.e. "best effort") delivery mechanism.</span></span> 
  
<span data-ttu-id="71efa-743">MPLS(다중 프로토콜 레이블 전환)는 네트워크 서비스 공급자가 지연, 지터 및 패킷 손실에 대한 성능 보장이 있는 IP 서비스를 제공하는 수단으로 개발됩니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-743">Multi-Protocol Label Switching (MPLS) was developed as a means for network service providers to offer an IP service with performance guarantees for delay, jitter and packet loss.</span></span> <span data-ttu-id="71efa-744">이러한 성능 보장을 제공하기 위해 MPLS는 기존 IP에서 예측할 수 없는 일부를 활용합니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-744">To deliver on those performance guarantees, MPLS takes some of the unpredictability out of traditional IP.</span></span> <span data-ttu-id="71efa-745">먼저 각 패킷이 라우터와 라우터를 대상에 연결하는 방법을 찾는 대신(그 결과 각 패킷이 원본에서 대상으로 다른 경로를 사용하게 될 수 있습니다). MPLS는 모든 패킷을 "가상 회로" 연결에서 LSP(레이블 전환 경로)라는 고정 경로로 라우팅합니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-745">First, rather than having each packet find its way router-to-router to its destination (the result of which could be that each packet takes a different route from the source to the destination), MPLS routes all of the packets on a "virtual circuit" connection with a fixed route called a Label Switched Path (LSP).</span></span> <span data-ttu-id="71efa-746">해당 경로의 링크 중 하나에 실패하면 해당 링크를 사용하는 모든 LSP가 빠르게 다시 연결됩니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-746">If one of the links in that path fails, all of the LSPs using that link are quickly rerouted.</span></span>
  
<span data-ttu-id="71efa-747">패킷이 MPLS 네트워크로 전송될 때 네트워크 서비스 공급자의 에지 라우터는 적절한 LSP를 전달하는 데 사용되는 레이블이 포함된 패킷에 추가 헤더를 추가합니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-747">When a packet is sent into the MPLS network the network service provider's edge router appends an additional header to the packet that includes a label that is used to forward it over the appropriate LSP.</span></span> <span data-ttu-id="71efa-748">레이블은 MPLS 네트워크의 다른 끝에 있는 에지 라우터에 의해 제거됩니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-748">The label is stripped off by the edge router at the other end of the MPLS network.</span></span>
  
<span data-ttu-id="71efa-749">전달 프로세스를 단순화하는 것 외에도 MPLS가 제공하는 다른 이점은 네트워크 관리 시스템이 네트워크의 모든 링크에서 수행되는 연결을 알 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-749">Beside simplifying the forwarding process, the other advantage MPLS provides is that the network management system will know what connections are being carried on every link in the network.</span></span> <span data-ttu-id="71efa-750">트래픽이 네트워크를 통해 라우팅되는 방법을 제어하여 운영자는 각 경로가 제공할 QoS를 보장할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-750">By controlling the way traffic is routed through the network, the operator can guarantee the QoS each path will provide.</span></span> <span data-ttu-id="71efa-751">따라서 기존 또는 기본 IP의 최상의 노력 성능과 달리 MPLS 운영자는 예측 가능한 성능을 사용하여 IP 서비스를 제공할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-751">So unlike the best effort performance of traditional or basic IP, MPLS operators can provide an IP service with predictable performance.</span></span> <span data-ttu-id="71efa-752">또한 LSP를 사용하면 기존 인터넷 서비스보다 MPLS가 본질적으로 더 안전합니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-752">That LSP also makes MPLS inherently more secure than traditional Internet services.</span></span> <span data-ttu-id="71efa-753">따라서 기본 IP 서비스를 통해 네트워크가 양질의 음성을 제공하고 FEC와 같은 기술을 사용하여 배당률을 향상시키는 데 사용할 수 있도록 충분히 잘 수행될 수 있지만 MPLS를 사용하여 확신할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="71efa-753">So with basic IP service we can hope that the network will perform well enough to provide good quality voice and use techniques like FEC and more resilient voice coding to improve the odds, but by using MPLS, we can be sure of it.</span></span>
