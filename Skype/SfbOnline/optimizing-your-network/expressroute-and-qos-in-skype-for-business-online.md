# <a name="expressroute-and-qos-in-skype-for-business-online"></a>비즈니스용 Skype Online의 ExpressRoute 및 QoS

Microsoft 365 용 Azure Express 또는 Office 365 및 비즈니스용 Skype Online을 사용 하는 전용 네트워크 연결을 통해 Microsoft 365 또는 Office 365에 연결 합니다. 비즈니스용 Skype 앱에 대 한 전용 연결에서는 신뢰할 수 있고 예측 가능한 성능이 제공 되 고 공개 인터넷에서의 프라이버시는 사라집니다. 이제 예측 가능 하 고 비즈니스 클래스 안정성 및 가동 시간 SLA를 추가 하는 비즈니스용 Skype Online 365 365에 대 한 더 나은 네트워크 연결을 구매할 수 있습니다.
  
> [!NOTE]
> 새 버전의 대역폭 계산기를 사용할 수 있습니다 ( [비즈니스용 Skype, 대역폭 계산기](https://go.microsoft.com/fwlink/?LinkId=715766)). 그러나이 문서의 지침에서는 Lync 2010 및 2013 대역폭 계산기를 사용 합니다. 
  
## <a name="skype-for-business-online-and-expressroute"></a>비즈니스용 Skype Online 및 Express 경로

Microsoft의 Express 파트너를 사용 하 여 작업 하는 경우 전용 연결을 통해 클라우드에서 비즈니스용 Skype Online을 비롯 한 다양 한 Microsoft 365 및 Office 365 응용 프로그램을 연결할 수 있습니다. 그러나 비즈니스용 Skype에 대 한 실시간 음성 및 영상 통신 기능을 위해서는 이러한 Microsoft 365 또는 Office 365 실시간 워크 로드를 지원 하도록 특별히 구성 된 네트워크 서비스가 필요 합니다. 여기에는 필요한 트래픽 볼륨을 운반 하기에 충분 한 대역폭을 보유 하 고 있으며 QoS (서비스 품질)를 지원 하 여 사용자에 게 비즈니스 수업 경험을 제공할 수 있는 네트워크가 포함 됩니다.
  
이 문서는 관리자와 네트워크 설계자가 실시간 통신을 지 원하는 데 필요한 특별 한 문제, Microsoft에서 제공 하는 도구를 사용 하 여 이러한 요구 사항을 지원할 수 있는 네트워크를 디자인 하는 데 도움이 되 고 사례 연구를 통해 디자인 프로세스를 안내 하도록 설계 되었습니다. 
  
이 문서의 첫 번째 부분에서는 [Lync 2010 및 2013 대역폭 계산기](https://go.microsoft.com/fwlink/?LinkId=690282) 를 사용 하 여 네트워크 디자인을 통해 대규모의 다중 사이트 비즈니스용 Skype express 배포에 대 한 네트워크 요구 사항을 예측 하는 데 도움이 되는 사례 연구를 안내 합니다. 이 문서의 두 번째 부분에서는 비즈니스 서비스 품질 (QoS)의 기본 개념과 비즈니스용 Skype 실시간 통신을 지 원하는 특정 기술 세부 정보에 대 한 자세한 내용, 그리고 필요한 특정 유형의 네트워크 서비스에 대해 자세히 설명 합니다.
  
이 문서의 모든 정보는 QoS 및 Express 경로에 대 한 기술적 세부 정보 및 이해를 제공 하 고, 사용자가 직면 하는 특정 문제를 파악 하 고, 비즈니스용 Skype 네트워크를 통해 Express 경로를 성공적으로 배포 하는 데 사용할 수 있는 도구와 기술에 대 한 실무 정보를 제공 합니다.

### <a name="setting-up-the-network"></a>네트워크 설정

Dewey 법률 L L C에 대해 일관성 있고 고품질 실시간 서비스를 제공 하기 위해 반드시 충족 해야 하는 몇 가지 기본 요구 사항이 있습니다.
  
- 정전 중에도 음성 서비스를 제공 하기 위해 네트워크 배포 스위치와 라우터가 PoE (power over Ethernet) IEEE 802.3 af 또는 802.3을 제공 해야 합니다.
    
- 또한 네트워크 스위치와 라우터에서 UPS (중단 되지 않은 전원)를 사용 하 여 정전 중에 계속 작동할 수 있어야 합니다.
    
    LAN 지사에 Wi-fi 연결이 있으므로 비즈니스용 [Skype 솔루션](https://go.microsoft.com/fwlink/?LinkId=690281)에서 공인 Skype For business wi-fi 인프라 파트너를 사용 하는 것이 좋습니다.
    
    > [!TIP]
    >  802.11 n 및 802.11 ac 무선 액세스 포인트를 사용 하는 것이 좋습니다.
  
- 가장 중요 한 것은 모든 사무실의 모든 LAN 네트워크가 QoS (서비스 품질)를 제공 하도록 설정 되어 있어야 한다는 것입니다. 여기에는 Pc, 노트북, 스위치 및 라우터와 같은 네트워킹 하드웨어 등이 포함 됩니다.
    
이제 Dewey 법률 L L C에 대해 비즈니스 점수 음성 서비스를 제공 하는 데 대 한 기본 사항이 적용 되었습니다. Azure Express 서비스에 연결 하는 네트워크 서비스 파트너에서 MPLS (멀티 프로토콜 레이블 전환) IP 서비스를 사용 하는 것이 좋습니다. MPLS는 지연, 지터 및 패킷 손실에 대 한 성능 보장으로 IP 서비스를 제공 합니다. 그러나 MPLS을 사용할 수 없는 경우에는 Express 경로 데이터 교환 파트너 중 하나에 연결 된 이더넷을 사용할 수도 있습니다.
  
MPLS 공급자는 여러 종류의 서비스를 제공 하지만 각각 다른 용어를 사용 하 여 해당 수준을 식별 합니다. [Lync 2010 및 2013 대역폭 계산기](https://go.microsoft.com/fwlink/?LinkID=690282) 에 입력 한 데이터와 사용할 수 있는 옵션을 이해 하 고 다양 한 Microsoft 365 및 Office 365 실시간 작업 부하 응용 프로그램에 권장 되는 것을 확인 하려면 공급자와 긴밀히 협력 해야 합니다.
  
비즈니스용 Skype 응용 프로그램의 데이터를 적절 한 MPLS 수업 서비스에 매핑할 수 있는 방법에는 두 가지 옵션이 있습니다.
  
- DSCP (DiffServ 제어 지점)를 사용 하는 트래픽의 끝점 표시
    
- 네트워크 ACL (액세스 제어 목록) 기반
    
끝점 표시를 구현 하려면 Dewey 법률 L L C에 대해 모든 도메인에 연결 된 Windows 컴퓨터를 구성 해야 합니다. 각 패킷을 적절 한 DiffServ 제어 지점 (DSCP) 표시로 표시 한 다음 모든 office 사이트의 모든 네트워크 스위치와 라우터에 QoS를 구현 하 여 QoS 표시가 유지 되 고 제거 되지 않았는지 확인 합니다. 네트워크 패킷의 DSCP 표시는 네트워크 패킷의 우선 순위를 지정 하는 방법을 서비스 공급자에 게 알립니다. **DSCP에 대 한 자세한 내용은 2 부의 QoS 섹션을.**
  
네트워크 ACL 기반 할당의 경우 DSCP 우선 순위 표시가 업스트림 라우터에서 구현 되 고 UDP 원본 포트를 기반으로 합니다. 각 응용 프로그램에 대해 권장 되는 포트 범위는 [네트워크 계획, 모니터링 및 Lync Server의 문제 해결](https://go.microsoft.com/fwlink/?LinkId=690286)섹션 2.6.1.1에 나열 되어 있습니다. 이는 Dewey 사법 L L C의 전반적인 QoS 구현 및 디자인과 다양 한 QoS 정책 및 패킷 표시 불일치에 대 한 주의를 기울여야 한다는 것이 중요 합니다.
  
각 Express 비트 네트워크 서비스 공급자에는 실시간 음성 및 비디오에 적합 한 서비스 수업 (QoS)이 제공 됩니다. 이 COS는 음성 용 ' 긴급 전달 ' (EF) 및 비디오에 대 한 ' 보장 전달 ' (AF) 이라고 합니다. 음성 EF 트래픽에 대해 구입 하는 대역폭의 크기는 매우 주의 해야 합니다. 이유는 서비스의 음성 클래스가 프로 비전 되는 클래스 보다 더 많은 음성 트래픽을 전송 하는 이벤트를 제공 하는 것입니다.
  
> [!TIP]
>  서비스 공급자의 약정을 초과 하 여 서비스의 voice 수업에 전송 된 트래픽은 즉시 음성 품질로 영향을 받게 됩니다.
  
Dewey 사법 L L C에 대 한 전반적인 디자인을 살펴 보세요. 네트워크에서 음성 트래픽을 지원 하는 데 필요한 네트워크 대역폭의 양을 정확 하 게 결정 하 고 음성에 대 한 DSCP 설정 (예: DSCP EF 46)으로 각 음성 패킷과 음성 패킷만 표시 하는 것이 매우 중요 합니다.
  
엔터프라이즈 네트워크에서 QoS를 구현 하려면 끝점 또는 라우터에서 적절 한 계층 3 우선 순위 표시기 (예: DSCP)를 사용 하 여 각 패킷을 표시 해야 합니다. 전체 네트워크 경로를 따라 각 스위치와 라우터에 QoS 옵션이 켜져 있어야 합니다. QoS를 설정 하지 않은 네트워크 스위치 또는 라우터를 하나만 사용 하는 경우 해당 스위치 또는 라우터를 통과 하는 음성 또는 비디오 패킷에서 QoS 표시가 제거 될 수 있습니다. 이렇게 하면 모든 다운스트림 스위치 및 라우터에서의 QoS를 사용 하지 않도록 설정 하 여 Express 경로 값을 낮춥니다.
  
또한 계층 3 및 계층 2 QoS 우선 순위에 대 한 연관은 각 점에 정의 되어 있어야 합니다. 레이어 2 우선 순위 메커니즘은 IEEE 802.1 p (유선 네트워크) 및 Wi-fi 네트워크용 802.11 e/WMM으로 정의 됩니다. 더 중요 한 것은 네트워크 서비스 공급자의 MPLS 네트워크를 향한 네트워크 라우터가 모든 아웃 바운드 패킷에 DSCP 설정을 유지 하 여 적절 한 MPLS 수업을 유지 관리 해야 한다는 것입니다. 
  
> [!TIP]
>  QoS 설정에 대 한 특정 세부 정보는 [Lync Server의 2.6 네트워크 계획, 모니터링 및 문제 해결]( https://go.microsoft.com/fwlink/?LinkId=760669)섹션을 참조 하세요. 또한 네트워크 계획 요구 사항이 더 필요한 경우 [비즈니스용 Skype 2015에 대 한 네트워크 요구 사항 계획](https://go.microsoft.com/fwlink/?LinkId=690287) 을 참조 하세요.
  
### <a name="ordering-network-access-services"></a>네트워크 액세스 서비스 주문

Express 경로를 지원 하기 위해 QoS 네트워크 선행 조건과 메커니즘이 있는 경우 다음 단계는 Express 경로 네트워크 액세스 서비스에 대 한 주문을 배치 하는 것입니다. Microsoft 네트워크 서비스 공급자 파트너 로부터 Dewey 법률 L L C에 대 한 Express 경로 액세스 서비스를 주문 하는 경우 다음 두 가지를 제공 해야 합니다.
  
- 각 사이트를 Express, Microsoft 365 또는 Office 365에 연결 하는 데 필요한 총 대역폭입니다.
    
- Dewey 법률 L L C에서 사용 중인 비즈니스용 Skype 앱을 지 원하는 데 필요한 각 서비스 종류에 필요한 총 대역폭입니다. 서비스 대역폭 요구 사항 클래스는 음성, 비디오, 메신저, 현재 상태, 화면 공유 등의 다양 한 비즈니스용 Skype 응용 프로그램에서 예상 하는 트래픽 량에 따라 제어 됩니다.
    
### <a name="determining-bandwidth-requirements-for-skype-for-business-applications"></a>비즈니스용 Skype 응용 프로그램의 대역폭 요구 사항 결정

Dewey 법 L L C의 경우, 이제 총 대역폭을 다양 한 서비스 수업 간에 나누는 방법에 대해 알고 있어야 합니다. 예를 들어, 각 비즈니스용 Skype 응용 프로그램의 대역폭 크기를 결정 합니다.
  
각 Dewey 법률 L L C에서 이러한 요구 사항을 확인 합니다. 사이트에는 [Lync 2010 및 2013 대역폭 계산기](https://go.microsoft.com/fwlink/?LinkID=690282)를 사용 합니다. 이 계산기는 음성, 비디오, 회의 및 화면 공유를 비롯 하 여 다양 한 비즈니스용 Skype 응용 프로그램의 예상 사용을 지정할 수 있는 Excel 기반 도구입니다. 계산기는 네트워크의 각 사이트에 대해 자동으로 예상 되는 대역폭 및 Co 요구 사항을 생성 합니다. Lync 2010 및 2013 대역폭 계산기를 다운로드할 때 사용 하는 방법에 대 한 세부 정보를 제공 하는 사용자 안내서도 다운로드 됩니다. 
  
스프레드시트에 대 한 도움말을 보려면 스프레드시트의 다양 한 셀이 색으로 구분 되어 있습니다.
  
- **초록** 다음은 일반적인 데이터 입력 영역입니다.
    
- **옐로우** 고급 데이터 입력 영역입니다. 이러한 기능을 변경할 수 있지만 주의 해야 합니다.
    
- **레드** 읽기 전용 영역이 며 잠긴 입력 값 이며 변경할 수 없습니다.
    
- **회색** 표시 전용 영역입니다. 일반 입력 영역의 결과 또는 데이터입니다.
    
Dewey 법률 L L C에 대 한 디자인 프로세스입니다. 먼저 사용자를 다른 ' 사용자 characterizing '로 전환 합니다. 정의한 각 가상 사용자에 대해 다양 한 비즈니스용 Skype 응용 프로그램의 예상 되는 용도 (' 없음 ', ' Low ', ' Medium ', ' High ' 또는 세 가지 정의 된 ' Custom ' 설정 중 하나)를 지정할 수 있습니다. 이 선택 항목은 ' 가상 사용자 ' 워크시트에 있습니다. 각각의 선택 (' Low ', ' Medium ' 또는 ' High ')에 대 한 특정 사용량은 제공 되지만 각 선택 항목의 기본값을 변경할 수 있습니다. 각 사이트에 있는 각 가상 사용자의 수를 식별 하 여 계산기가 각 위치에 필요한 총 대역폭을 계산할 수 있습니다.

Exchange 및 OneDrive 용 대역폭 계산기를 포함 하 여 다른 Microsoft 365 및 Office 365 서비스에 대 한 네트워크 계획 지침을 제공 합니다. 사이트 내 트래픽을 다시 추가 해야 하므로 네트워크 서비스 공급자에 대 한 대역폭 구독이 더 높습니다. Lync 2010 및 2013 대역폭 계산기는 예상 되는 트래픽만 제공 하므로 스트레스 테스트를 수행 하는 해당 트래픽 볼륨을 지 원하는 네트워크 기능을 확인 하는 것이 좋습니다. 
  
> [!TIP]
> 네트워크 사전 평가를 수행할 때 네트워크 스트레스 테스트를 강력히 권장 합니다. 
  
스트레스 테스트에는 인프라를 구성 하 고 구성한 다음 성능을 모니터링 하면서 예상 되는 트래픽 크기를 사용 하 여 작업을 실행 해야 합니다. 일부 지역에서는 트래픽 추정이 정확 하지 않을 수 있지만, 최소한 Lync 2010 및 2013 대역폭 계산기가 예측 한 트래픽 볼륨을 지원할 수 있다는 것을 알 수 있습니다. 적어도 몇 일 동안 스트레스 테스트를 실행 하는 것이 좋지만 더 오랜 시간 동안 실행 하는 것은 숫자를 구체화 하는 데 도움이 될 수 있습니다. 그러나 실제 사용자 네트워크 트래픽을 운반 하지 않는 네트워크 서비스 비용을 기준으로 스트레스 테스트 기간을 연장 하는 것을 고려 하 여 평가 해야 합니다. Microsoft는 네트워크 사전 평가 스트레스 도구를 포함 하 여 네트워크 관리 및 운영 도구를 제공 하기 위해 IT Pro 도구 프로그램의 일부로 여러 공급 업체를 인증 했습니다. 비즈니스용 Skype는 또한 인증 된 IT Pro 도구를 사용 하 고 네트워크 평가를 수행할 수 있는 SI (시스템 통합자)를 제공 합니다. [비즈니스용 Skype 솔루션에 대 한](https://go.microsoft.com/fwlink/?LinkID=690307)자세한 내용을 볼 수 있습니다 (IT Pro 도구).
  
스트레스 테스트는 네트워크에서 필요한 트래픽 볼륨을 지원할 수 있는 몇 가지 reassurance를 제공 하지만 실제로 Lync 2010 및 2013 대역폭 계산기 데이터는 여러 가지 이유로 해제 될 수 있습니다. 또한 대역폭을 충분히 유지 하 고 QoS 메커니즘이 제대로 작동 하도록 배포 된 후 지속적으로 네트워크 평가를 수행 하 여 계속 해 서 사이트 네트워크를 모니터링 해야 합니다. 더 많은 실제 사용자가 회선으로 전환 됨에 따라 네트워크 성능을 지속적으로 모니터링 하는 것이 중요 합니다.
  
## <a name="part-2-expressroute-skype-for-business-qos"></a>2 부: Express 비즈니스를 위한 비즈니스용 Skype QoS

Microsoft의 Express 서비스는 Azure 클라우드에 대 한 전용 연결을 제공 하지만, Office 365 실시간 작업 부하의 통신 서비스에는 트래픽 볼륨을 운반 하는 데 충분 한 대역폭이 있는 네트워크 서비스가 필요 하며, QoS (서비스 품질)를 지원 하 여 비즈니스 등급 사용자 환경을 제공할 수 있습니다. Qos를 지원 하지 않는 경로에는 전체 통화의 품질이 저하 될 수 있으므로 QoS를 사용할 수 있는 연결에는 종단 간 (PC, 네트워크 스위치 및 라우터를 클라우드로 구성 해야 합니다.)
  
이 섹션에서는 IP 네트워크에서 실시간 트래픽을 지원할 때 발생 하는 문제를 이해 하 고 Microsoft의 Express Exchange 공급자나 네트워크 서비스 공급자 파트너를 사용 하 여 Microsoft 365 또는 Office 365 실시간 작업 부하를 구성 하 고 지원할 수 있도록 합니다.
  
QoS는 Express 경로 네트워크 회로를 통해 네트워크에서 독점적으로 허용 되며 Microsoft 네트워크 내에서 비즈니스용 Skype 트래픽을 위해 사용 됩니다. 현재, Microsoft에서 일부 아웃 바운드 연결의 일부는 비즈니스용 Skype에 대 한 DSCP 값이 누락 되었습니다. 아웃 바운드 트래픽이 DSCP 값으로 완전히 표시 될 때까지이 문서의 **ACL (네트워크 액세스 제어 목록)을 사용 하 여 Qos 구현** 에서 설명한 대로 네트워크 경계의 트래픽에 qos 표시를 추가 하기 위한 지침을 따르는 것이 좋습니다.
  
### <a name="the-real-time-problem"></a>실시간 문제

비즈니스 음질을 제공 하는 음성 및 영상 서비스는 IP 네트워크에 특별 한 요구 사항을 배치 합니다. 실시간 트래픽은 UDP (사용자 데이터 그램 프로토콜)를 사용 하 여 수행 되는 RTP (실시간 전송 프로토콜)를 사용 합니다. 각 메시지에 오류를 번호를 지정 하 고 테스트 하는 TCP (전송 제어 프로토콜)와 달리 손실 되거나 errored 메시지를 감지 하 고 재전송 하는 다른 메커니즘을 포함 하는 UDP는 이러한 유형의 안정성을 제공 하지 않습니다. 메시지가 오류로 손상 되거나 버퍼 오버플로와 인해 손실 된 경우 손실 됩니다. UDP는 RTP와 함께 사용할 수 있습니다. 실시간 트래픽의 특성은 손실 된 메시지를 다시 보내는 경우에도 너무 늦게 도착 하 여 음성 메시지 흐름에 긍정적인 영향을 주지 않습니다.
  
개발자는 IP를 통해 음성 및 비디오의 성능을 개선 하기 위한 두 가지 접근 방법을 사용 하 여 손실 된 음성 패킷의 영향을 파악 합니다.
  
- 패킷이 손실 되 면 음성 코딩/디코딩 기능을 더욱 탄력적으로 활용 하세요. 이 작업은 FEC (정방향 오류 수정)를 사용 하 여 Microsoft 365 또는 Office 365 실시간 전송에서 찾을 수 있는 발생 한 오류에 대 한 백분율을 수정 하거나 Microsoft 코덱의 특성을 가진 손실 된 패킷의 영향을 마스킹하는 음성 디코딩 시스템을 디자인 하는 것입니다. 
    
- 서비스 품질 메커니즘을 사용 하는 전송 서비스를 사용 하 여 지연, 패킷 손실, 지터, 패킷 간의 지연 변동에 대해 네트워크 성능을 보장 합니다.
    
탄력적 음성 코딩은 패킷 손실의 문제만 해결 하므로 실시간 음성과 비디오를 전송 하는 데 사용 되는 네트워크에는 지연과 지터를 최소화 하는 메커니즘이 있습니다. 회복성 있는 (resilient) 코딩이 너무 많은 패킷이 손실 되는 경우에도 수신 스테이션에는 인식 가능한 음성 신호 버전을 다시 만드는 데 필요한 정보가 부족 합니다. 사용 되는 음성 인코딩 기술에 따라 음성 품질이 크게 저하 될 수 있는 손실 된 패킷의 백분율입니다. 그러나 모든 경우에 연속적인 패킷의 문자열을 손실 하면 매우 문제가 발생할 수 있습니다.
  
과도 한 지연은 대화 흐름에 영향을 줄 수 있고 스피커에 성가신 불편을 주는 이유 때문에 지연을 최소화 하는 것이 중요 합니다. 모범 사례는 음성에 대 한 종단간 지연 시간 (' 입-귀 ' 지연 "이라고 함)을 150 밀리초 (밀리초) 미만으로 유지 해야 함을 알려 드립니다. ' 라운드트립 ' 지연의 단방향입니다. 물론, 전파 지연 또는 신호가 케이블을 통해 물리적으로 이동 하는 데 소요 되는 시간을 감안 하 여 oceans에서 이동 하는 것과 같은 긴 전송 링크에서는 지연이 증가 합니다.
  
지연 시간이 150 밀리초 보다 높은 경우 단방향 이면 스피커에 이상한 효과가 있습니다. Psychologically는 발표자의 두뇌에서 시계가 꺼지고 받는 사람이 해당 사용자가 듣지 않았다는 것을 생각 하 고 마지막으로 말한 내용을 반복 합니다. 이것은 지금까지부터 지연 되는 응답과 충돌 합니다. 위성 채널을 사용 하 고 있다면이 효과를 인식 하 게 됩니다. 위성 채널을 통해 단방향 지연은 대략 250 밀리초 이며,이는 허용 되는 지연 보다 훨씬 더 많은 경우입니다.

### <a name="expressroute-as-part-of-a-business-grade-voice-network"></a>비즈니스 학년 음성 네트워크의 일부로 서 점

Express 경로는 3 개의 연결 옵션 중 하나에서 NSP (네트워크 서비스 공급자) 또는 Exchange 공급자 (EXP)를 통한 전용 연결을 제공 합니다. 
  
- 클라우드 Exchange 공동 위치
    
- 지점간 이더넷 연결
    
- 임의 (IPVPN) 연결
    
이는 높은 가용성 (99.9% 가동 시간 SLA) 및 보안이 유지 되는 신뢰할 수 있는 라우팅 (인터넷 전송 없음)과 트래픽 우선 순위에 대 한 영향을 받지 않는 (QoS는 아래에 설명 된) 서비스 표시의 품질을 제공 합니다. (으)로, 잘 계획 된 WAN과 함께 비즈니스 학년 음성 네트워크를 제공할 수 있습니다.
  
회로에 연결 된 사무실 또는 데이터 센터 (하이브리드 토폴로지가 있는 경우)에서 전송 되는 데이터에 대해 Express 경로를 사용할 수 있습니다. 사용자가 VPN에 연결 되어 있지 않고, Express 경로 지정 회로의 크기를 조정할 때 대역폭 추정치에 포함할 필요가 없는 경우, 예를 들어 가정용 사무실 또는 여행 등의 오프 사이트 사용자의 데이터는 Express 경로를 활용 하지 않습니다. 여러 국가의 고객 인 경우 각 지역에서 Express 경로를 구입 하 고 BGP 커뮤니티 태그를 사용 하 여 트래픽이 기본 비트 회로 (일반적으로 각 사이트에 가장 가까운 것)로 이동 하 고, 다른 회로는 장애가 단일 회로에 영향을 미치는 경우 중복성을 제공 하도록 할 수 있습니다. 
  
### <a name="if-expressroute-isnt-an-option"></a>To = 옵션이 선택 되어 있지 않은 경우

비용, 단위 단위 필수 조건 충족 되지 않음 또는 현재 NSP의 제한 사항으로 인해 모든 사이트를 Express 경로에 연결 하는 것은 불가능 합니다. Express를 사용할 수 없는 경우 아래 지침에 따라 네트워크 내에 QoS를 표시 하 고, QoS 기반 트래픽 우선 순위에 대해 충분 한 대역폭과 지원을 제공 하기 위해 NSP와 계약을 계획 하는 것이 좋습니다.
  
또한 여러 지역에 사무실이 있지만, 모든 지역에서 Express 경로를 사용 하지 않는 경우에는 불필요 한 장거리 수송을 피할 수 있도록 위성 사무실에 대 한 소통을 구성 하는 경우 지역 BGP 커뮤니티 태그가 있어야 합니다. 예를 들어 미국에서 호스팅되는 비즈니스용 Skype Online 조직이 있지만 유럽 지역에 지사를 사용 하는 회사는 실리콘 유역 에서만 단일 Express 회로를 보유 하 고 있는 것이 좋습니다. 대부분의 비즈니스용 Skype Online 트래픽은 조직이 호스팅되는 데이터 센터 (예: 회사 내 다른 사용자와의 컨퍼런스 통화)로 라우팅되며, 대부분의 트래픽에 대 한 Express 회로를 사용 하는 것이 좋습니다. 그러나 유럽의 사용자가 해당 조직이 유럽에 위치한 다른 회사에서 호스트 하는 전화 회의에 참가 하는 경우 해당 통화의 미디어 대상은 두 번째 회사가 위치한 유럽 데이터 센터가 됩니다. 실리콘 유역에서 Express 경로 회로를 통해 트래픽을 라우팅하는 것은 인터넷을 통해 사용할 수 있는 것 보다 직접적인 경로가 짧습니다. 이러한 경우 네트워크 내에서 라우터를 구성 하 고, 라우팅 규칙을 만들 때 커뮤니티 태그를 검사 하 고, 유럽 지역 태그가 있는 트래픽에 대해 실리콘 유역 Express 경로 회로가 아닌 인터넷을 통해 라우팅할 수 있습니다.
  
### <a name="basic-concepts-of-quality-of-service-qosclass-of-service-cos"></a>QoS (서비스 품질)의 기본 개념/서비스 종류 (CoS)

IP에서 QoS (서비스 품질)는 일부 패킷에 대 한 우선 순위 처리를 제공 하는 데 사용 되는 메커니즘을 설명 합니다. QoS는 국제 텔레커뮤니케이션 통합 (ITU) 정의에 따라 지연, 손실, 신호 대 잡음 비율, crosstalk, 에코, 인터럽트, 주파수 응답, 소음 수준 등을 포함 하 여 연결의 모든 품질 측면을 구성 합니다. 패킷 네트워크에서 QoS로 참조 하는 항목은 지연, 지터 및 패킷 손실에 대 한 성능 개선에 중점을 둔 CoS (서비스 수업)의 것으로, 더 일반적으로 사용 되는 것 처럼 계속 해 서 QoS 용어를 사용 하 게 됩니다.
  
두 가지 기본 구성 요소에 대 한 IP 네트워크 호출에 QoS를 제공 합니다.
  
- 실시간 트래픽에 대 한 각 링크에 정의 된 대역폭 크기를 예약 하는 것입니다. 해당 대역폭이 언제 든 지 실시간 트래픽에 필요 하지 않으면 다른 트래픽에 사용할 수 있습니다. 일반적인 지침은 모든 링크 용량의 30%를 음성 트래픽에 대해 할당 해야 한다는 것입니다.
    
- 경로에 스위치와 라우터에 대해 지정 하는 우선 순위 표시기를 사용 하 여 해당 패킷의 우선 순위를 표시 하는 것을 나타냅니다.
    
스위치나 라우터에서 패킷을 받으면 다음 레그 또는 홉에 대 한 출력 큐로 이동 됩니다. 다른 우선 순위 수준에 대해 서로 다른 출력 큐가 있습니다. 스위치 또는 라우터는 우선 순위가 낮은 큐 보다 더 자주 높은 우선 순위 대기열을 서비스 하는 알고리즘을 사용 합니다.
  
챌린지는 계층 2 (이더넷 또는 Wi-fi 레이어)와 계층 3 (즉 IP 계층)에서 구현 되는 서로 다른 QoS 기술이 있다는 것입니다. 이러한 다른 QoS 구현은 네트워크의 각 스위치와 라우터에, 그리고 네트워크와 네트워크 서비스 공급자의 네트워크 간 인터페이스에도 구성 해야 할 수 있습니다.
  
다양 한 비즈니스용 Skype 응용 프로그램의 데이터를 적절 한 서비스 수업에 매핑할 수 있는 방법에는 두 가지 옵션이 있습니다.
  
- DSCP (차별화 서비스 제어 점)를 사용 하 여 트래픽의 끝점 표시 
    
- 네트워크 ACL (액세스 제어 목록) 기반
    
### <a name="end-point-traffic-marking--differentiated-services-control-point-dscp"></a>끝점 트래픽 표시-DSCP (구분 서비스 제어 점)

### <a name="layer-2-qos-ieee-8021pwi-fi-multi-media-ieee-80211e"></a>계층 2 QoS: IEEE 802.1 p/Wi-fi 멀티 미디어 (IEEE 802.11 e)

DSCP는 계층 3에서 QoS를 구현 하기 위한 표준 메커니즘 이지만 유선 (즉, 이더넷) 및 무선 (즉 Wi-fi 네트워크)에는 서로 다른 레이어 2 QoS 메커니즘이 있습니다. 유선 네트워크용 QoS 메커니즘은 IEEE 802.1 p 표준으로 정의 됩니다. WLAN QoS 메커니즘은 IEEE 802.11 e로 정의 되며 Wi-fi 동맹은 "Wi-fi 멀티 미디어 인증 됨" (WMM 인증)으로 식별 됩니다.
  
IEEE 802.1 p는 3 비트 우선 순위 코드 점 (PCP)을 사용 하 여 메시지의 우선 순위를 식별 합니다. PCP는 VLAN 식별자도 함께 전달 하는 이더넷 헤더의 32 비트 필드의 일부입니다. PCP 값에 대 한 정의는 아래에 나와 있습니다.

여기에서 IEEE 802.1 p는 각 우선 순위 수준에 대해 서로 다른 우선 순위 대기열로 정렬 된 트래픽이 있는 DSCP와 거의 같은 방식으로 구현 되지만, Wlan의 공유 미디어 특성은 다른 접근 방법으로 호출 됩니다. 액세스 지점과 클라이언트는 서로 다른 우선 순위 수준에 대해 별도의 출력 큐를 유지 관리 하는 반면, 라디오 채널에서 프레임을 보내는 방법에도 차이점이 있습니다.
  
Wi-fi 네트워크에서 액세스 지점과 연결 된 모든 클라이언트는 단일 반이중 채널을 공유 합니다 (즉, 한 번에 하나의 클라이언트 스테이션 또는 액세스 지점만 보낼 수 있음). 라디오 채널에서 충돌의 가능성을 최소화 하기 위해 스테이션은 "프레임 간 간격" 이라고 하는 정의 된 기간 동안 채널이 유휴 상태가 될 때까지 대기 하 고, 스테이션이 송신으로 이동할 때 채널을 사용 하 고 있는 경우 임의의 기간을 백오프 합니다. 프레임을 전송한 후에는 보낸 사람이 받는 사람 으로부터 승인 메시지를 받지 못하는 경우 충돌이 발생 하거나 다른 장애가 있는 것으로 간주 하 고, 다시 전송 하기 위해 라디오 채널에 액세스 하기 전에 임의 간격을 뒤로 진행 합니다. 백오프 간격은 임의로, 두 개의 스테이션이 다시 충돌 하는 확률을 줄이는 데에도 해당 됩니다.
  
라디오 채널에 대 한 액세스 우선 순위를 지정 하기 위해 IEEE 802.11 e/WMM은 "중재 프레임 간 Spacings" (AFIS) 이라고 불리는 다른 사전 전송 대기 간격을 정의 하 고 다양 한 트래픽 클래스에 대해 서로 다른 백오프 범위를 지정 합니다. ' 액세스 범주 ' 라는 4 개의 우선 순위 수준이 정의 되어 있습니다.
  
우선 순위는 우선 순위가 높은 프레임의 값을 짧게 지정 하 여 지정 됩니다. 따라서 한 스테이션이 음성 프레임을 보내기 위해 대기 하 고 있고 다른 스테이션에서 데이터 프레임을 보내기 위해 대기 중인 경우에는 항상 먼저 음성 프레임을 보냅니다. 기술적으로 음성 및 비디오 프레임에는 같은 AFIS 값이 지정 되지만, 비디오 프레임의 백오프 간격 범위는 더 높습니다. 따라서 음성 및 비디오 프레임이 첫 번째 시도에서 충돌할 수 있지만 음성 프레임은 항상 더 빨리 재전송 됩니다. ### <a name="implementing-qos-using-network-access-control-list-acl"></a>네트워크 ACL (액세스 제어 목록)을 사용 하 여 QoS 구현

Express 경로 구성에서 QoS를 구현 하는 다른 방법은 네트워크 ACL (액세스 제어 목록)을 사용 하는 것입니다. 이 방법에서는 끝점을 사용 하는 대신 UDP 원본 포트를 기준으로 해당 DSCP 표시를 업스트림 라우터에서 수행할 수 있습니다. 모든 스위치와 라우터는 DSCP 설정이 유지 되도록 QoS를 지원 하도록 계속 구성 되어 있어야 합니다. 더 중요 한 것은 DSCP 설정이 기본적으로 네트워크 서비스 공급자에 게 해당 패킷을 처리 하는 방법에 대 한 지침 이므로, 서비스 공급자의 네트워크에 연결 된 라우터는 각 패킷의 헤더에 DSCP를 유지 해야 합니다.
  
각 비즈니스용 Skype 응용 프로그램의 권장 포트 범위는 [네트워크 계획, 모니터링 및 Lync Server](https://go.microsoft.com/fwlink/?LinkId=690286) 가이드의 2.6.1.1 섹션에 나열 되어 있습니다. 이는 조직의 전반적인 QoS 접근 방식에 맞게 조정 하는 것이 중요 하며, 다양 한 QoS 정책 및 잠재적 패킷 다시 표시 불일치에 대 한 감시를 받아야 합니다.
  
주요 이유 QoS 및 MPLS 네트워크 서비스는 실시간 음성 및 비디오에 대 한 적절 한 사용자 환경을 보장 하는 데 사용 되는 반면, 데이터 응용 프로그램에도 동일한 기능을 적용할 수 있습니다. 모든 응용 프로그램을 동등 하 게 처리 하는 대신 MPLS 네트워크를 통해 조직에서 일부 데이터 응용 프로그램에 대 한 우선 순위를 제공할 수 있습니다. MPLS를 사용 하면 신용 카드 거래 또는 화면 공유와 같은 실시간 응용 프로그램에서 시간을 덜 민감한 트래픽 (예: 전자 메일)에 우선 순위를 부여할 수 있습니다.
  
### <a name="understanding-the-types-of-ip-network-services--basic-ip-and-mpls"></a>IP 네트워크 서비스 유형 이해-기본 IP 및 MPLS

"최선의 노력" 이라는 원칙에 따라 작동 하는 원래의 IP 패킷 전달 이는 해당 IP 패킷을 전달 하는 라우터가 해당 사용자를 대상에 게 전달 하는 것이 가장 좋지만,이를 목적지에 도달 하는 시기와 관련 하 여 보장할 수는 없다는 것을 의미 합니다. 이것은 가정용 인터넷 연결을 비롯 한 기본 인터넷 서비스가 지금 작동 하는 방식입니다. 이 개념은 특정 응용 프로그램에 대 한 안정성이 필요한 경우 프로토콜 스택의 상위 수준에 제공 된다는 것입니다. 신뢰할 수 있는 배달 메커니즘은 전송 제어 프로토콜 (TCP)입니다. 실시간 음성 및 비디오에 사용 되는 UDP (사용자 데이터 그램 프로토콜)는 신뢰할 수 없습니다 (즉, "최상") 제공 메커니즘입니다. 
  
멀티 프로토콜 레이블 전환 (MPLS)은 네트워크 서비스 공급자가 지연, 지터, 패킷 손실 등의 성능 보장으로 IP 서비스를 제공 하는 수단으로 개발 되었습니다. 이러한 성능 보장을 제공 하기 위해 MPLS는 기존 IP의 unpredictability 중 일부를 차지 합니다. 첫 번째 방법은 각 패킷이 대상에 대 한 라우터 간 경로를 찾는 것이 아니라 각 패킷이 원본에서 대상에 이르기까지 다른 위치를 사용 하 여 "가상 회로" 연결의 모든 패킷을 MPLS (레이블 전환 경로) 이라는 고정 경로를 통해 라우팅합니다. 해당 경로의 링크 중 하나가 실패 하면 해당 링크를 사용 하는 모든 Lsp가 빠르게 다시 라우팅됩니다.
  
패킷이 MPLS 네트워크에 전송 되는 경우 네트워크 서비스 공급자의 edge 라우터가 해당 LSP를 통해 전달 하는 데 사용 되는 레이블을 포함 하는 다른 헤더를 패킷에 추가 합니다. MPLS 네트워크의 다른 쪽 끝에 있는 edge 라우터에 의해 레이블이 제거 됩니다.
  
전달 프로세스 단순화 옆의 다른 장점은 네트워크 관리 시스템에서 네트워크의 모든 링크에 대해 어떤 연결이 MPLS 알 수 있다는 것입니다. 트래픽이 네트워크를 통해 라우팅되는 방식을 제어 하 여 각 경로가 제공 하는 QoS를 운영자가 보장할 수 있습니다. 전통적인 또는 기본 IP의 최상의 성과와 달리 MPLS 연산자는 예측 가능한 성능을 제공 하는 IP 서비스를 제공할 수 있습니다. 또한이 LSP는 전통적인 인터넷 서비스 보다 MPLS을 더욱 안전 하 게 만듭니다. 따라서 기본 IP 서비스를 사용 하면 네트워크는 좋은 음질의 음성 및 사용 기법을 제공 하는 것이 FEC, 탄력성 있는 음성 코딩 등의 기능을 향상 시킬 수 있으므로, MPLS를 사용 하는 것이 가능 하다는 것을 알 수 있습니다.
  
MPLS 공급자는 다른 용어를 사용 하 여 해당 항목을 식별 하는 여러 가지 서비스를 제공 합니다. [Lync 2010 및 2013 대역폭 계산기](https://go.microsoft.com/fwlink/?LinkID=690282) 의 출력과 다른 Microsoft 365 또는 Office 365 실시간 작업 부하 응용 프로그램에 대 한 권장 옵션을 이해 하려면 공급자와 긴밀 하 게 작업 해야 합니다.
