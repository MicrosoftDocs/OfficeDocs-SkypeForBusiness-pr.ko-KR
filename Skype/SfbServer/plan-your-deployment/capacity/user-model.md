# <a name="capacity-planning-user-model-usage-for-skype-for-business-server"></a>비즈니스용 Skype 서버에 대 한 용량 계획 사용자 모델 사용

이 문서에서는 [비즈니스용 Skype 서버의 사용자 모델](user-models.md)에서 설명한 사용에 따라 해당 사이트의 사용자 수에 대해 사이트에서 필요로 하는 서버 수에 대 한 지침을 제공 합니다.

> [!NOTE]
> 이 문서의 모든 권장 사항은 서버에 비즈니스용 Skype 누적 업데이트, 11 월 2015 이상을 설치한 것으로 가정 합니다.

## <a name="front-end-server"></a>프런트 엔드 서버

> [!NOTE]
> 이 서버 역할에 대해 스트레치 풀이 지원 되지 않습니다.

프런트 엔드 풀에서 풀의 모든 서버에 대 한 하이퍼 스레드를 사용 하도록 설정 하 고 서버 하드웨어가 Skype에 대 한 [서버 요구 사항에 대 한 권장 사항을 충족 한다고 가정 하 여 풀에 있는 모든 6660 사용자에 대해 프런트 엔드 서버 한 대를 보유 해야 합니다. ](../requirements-for-your-environment/server-requirements.md)비즈니스용 [Skype server 2019에 대 한](../../../SfBServer2019/plan/system-requirements.md)비즈니스 서버 2015 또는 시스템 요구 사항 하나 이상의 프런트 엔드 풀에 있는 최대 사용자 수는 8만 이며,이 경우에는 풀의 모든 서버에서 하이퍼 스레드를 사용할 수 있습니다. 사이트에 8만 명 이상의 사용자가 있는 경우 두 개 이상의 프런트 엔드 풀을 배포할 수 있습니다.

프런트 엔드 풀의 사용자 수를 고려 하는 경우이 프론트 엔드 풀과 연결 된 지점에서 Survivable Branch 기기 및 Survivable Branch 서버에 속한 사용자를 포함 합니다.

활성 서버를 사용할 수 없는 경우 해당 연결이 자동으로 풀의 다른 서버로 전송 됩니다. 3만 사용자와 다섯 개의 프런트 엔드 서버가 있는 시나리오에서 서버를 사용할 수 없는 경우에는 사용자의 6000 연결을 나머지 4 개의 서버로 이전 해야 합니다. 이 4 개의 나머지 서버에는 각각 7500 사용자가 있으며 권장 보다 더 큽니다.

3만 사용자를 위해 6 개의 프런트 엔드 서버로 시작 하 고 하나를 사용할 수 없게 된 경우 총 5000 명의 사용자가 나머지 5 대 서버로 이동 해야 합니다. 이 다섯 개의 나머지 서버는 권장 되는 범위의 각 호스트 6000 사용자를 따릅니다.

프런트 엔드 풀의 최대 사용자 수는 8만입니다. 풀의 최대 프런트 엔드 서버 수는 12 개입니다.

8만 사용자가 있는 프런트 엔드 풀의 경우 12 개의 프런트 엔드 서버는 [비즈니스용 Skype 서버의 사용자 모델](user-models.md)을 따르는 일반적인 배포의 성능에 적합 합니다. 재해 복구 장애 조치 (failover)를 지원 하도록 설계 된 배포 4만는 두 개의 쌍으로 이루어진 각 프런트 엔드 풀에 호스트 될 수 있으며, 각 풀에는 두 풀의 사용자를 포함 하는 충분 한 프런트 엔드 서버가 있는데,이 경우 1 개 풀을 t e 2 개에 대해 실패 해야 합니다. o.

특정 프런트 엔드 풀에의 한 좋은 성능으로 지원 되는 사용자 수는 다음과 같은 이유로 이러한 번호와 다를 수 있습니다.

- 프런트 엔드 서버에 대 한 하드웨어가 권장 사항을 충족 하지 않습니다.

- 예를 들어 많은 회의 트래픽이 있는 경우 조직의 사용은 사용자 모델과 매우 다릅니다.

RSS를 사용 하도록 설정 하는 방법에 대 한 자세한 내용은 네트워크 어댑터 설명서를 참조 해야 합니다.

## <a name="conferencing-maximums"></a>회의 최대값

풀의 사용자 5%가 한 번에 회의에 있을 수 있는 사용자 모델에 대해 8만 사용자 풀은 동시에 4000 사용자에 게 제공 될 수 있습니다. 이러한 회의는 미디어 (일부 IM 전용, 오디오가 있는 일부 메신저, 일부 오디오/비디오 등)와 참가자 수를 혼합 하 여 사용 해야 합니다. 실제 회의 수에 대 한 하드 제한은 없으며 실제 사용량에 따라 실제 성능이 결정 됩니다. 예를 들어 조직에 사용자 모델에 포함 된 것 보다 더 많은 혼합 모드 회의가 있는 경우이 문서에 나와 있는 권장 사항 보다 더 많은 프런트 엔드 서버 또는 A/V 회의 서버를 배포 해야 할 수 있습니다. 사용자 모델의 가정에 대 한 자세한 내용은 [비즈니스용 Skype 서버의 사용자 모델](user-models.md)을 참조 하세요.

사용자를 호스트 하는 일반 비즈니스용 Skype 서버 프런트 엔드 풀에서 호스트 하는 지원 되는 최대 컨퍼런스 크기는 250 참가자입니다. 250 사용자 회의가 진행 되는 동안에도 풀 사용자의 총 5%가 동시에 진행 되는 것을 비롯 한 다른 컨퍼런스도 여전히 지원 됩니다. 예를 들어 12 개의 프런트 엔드 서버와 8만 사용자가 있는 경우 250 사용자 회의가 진행 되는 동안 비즈니스용 Skype 서버는 소규모 회의에 참여 하는 다른 사용자에 게 3750를 지원 합니다.

프런트 엔드 풀 또는 Standard Edition server에 속한 사용자 수와 관계 없이 비즈니스용 Skype 서버는 250 사용자 회의를 호스트 하는 동일한 풀이나 서버에서 더 작은 회의에 참여 하는 최소 125의 다른 사용자를 지원 합니다.

250와 1000 사용자 간에 회의가 가능 하도록 하려면 해당 회의를 호스트 하는 데 사용할 별도의 프런트 엔드 풀을 설정 하면 됩니다. 이 프론트 엔드 풀은 사용자를 호스팅하지 않습니다. 자세한 내용은 [비즈니스용 Skype 서버의 대규모 모임에 대 한 계획](../../plan-your-deployment/conferencing/large-meetings.md)을 참조 하세요.

조직에서 사용자 모델에 사용 되는 것 보다 훨씬 더 많은 혼합 모드 컨퍼런스를 사용 하는 경우이 문서에서 권장 하는 프런트 엔드 서버 (최대 12 대의 프런트 엔드 서버 제한)를 배포 해야 할 수 있습니다. 사용자 모델의 가정에 대 한 자세한 내용은 [비즈니스용 Skype 서버의 사용자 모델](user-models.md)을 참조 하세요.

## <a name="edge-server"></a>Edge 서버

> [!NOTE]
> 이 서버 역할에 대해 스트레치 풀이 지원 되지 않습니다.

사이트에 동시에 액세스 하는 모든 12000 원격 사용자에 대해 하나의 Edge 서버를 배포 해야 합니다. 최소한 고가용성을 위해 두 개의 Edge 서버를 사용 하는 것이 좋습니다. 이러한 권장 사항은 Edge 서버의 하드웨어가 [서버 하드웨어 플랫폼](https://technet.microsoft.com/library/c964c1c0-0153-472b-88ad-a38866e0df0c.aspx)의 권장 사항을 충족 한다고 가정 합니다.

Edge 서버의 사용자 수를 고려 하는 경우이 사이트의 프런트 엔드 풀과 연결 된 지점에서 Survivable Branch 기기 및 Survivable Branch 서버에 속한 사용자를 포함 합니다.

> [!NOTE]
> Edge 서버에서 A/V 회의에 대 한 서비스의 성능을 향상 시키려면 Edge 서버의 네트워크 어댑터에서 수신측 배율 (RSS)을 사용 하도록 설정 해야 합니다. RSS를 사용 하면 서버의 여러 프로세서가 들어오는 패킷을 병렬로 처리할 수 있습니다. 자세한 내용은[Windows Server 2012에서 "수신측 배율 (RSS)](https://go.microsoft.com/fwlink/p/?linkId=268731)"을 확인 하세요. RSS를 사용 하도록 설정 하는 방법에 대 한 자세한 내용은 네트워크 어댑터 설명서를 참조 해야 합니다.

## <a name="director"></a>Director

> [!NOTE]
> 이 서버 역할에 대해 스트레치 풀이 지원 되지 않습니다.

디렉터 서버 역할을 배포 하는 경우 사이트에 동시에 액세스 하는 모든 12000 원격 사용자에 대해 하나의 디렉터를 배포 하는 것이 좋습니다. 최소한 고가용성을 위해 2 개의 디렉터를 권장 합니다. 이러한 권장 사항은 Edge 서버의 하드웨어가 [서버 하드웨어 플랫폼](https://technet.microsoft.com/library/c964c1c0-0153-472b-88ad-a38866e0df0c.aspx)의 권장 사항을 충족 한다고 가정 합니다.

디렉터에 대 한 사용자 수를 고려 하는 경우이 사이트의 프런트 엔드 풀에 연결 된 지점에서 Survivable Branch 기기 및 Survivable Branch 서버에 속한 사용자를 포함 합니다.

## <a name="back-end-server"></a>백 엔드 서버

대부분의 데이터베이스 정보는 프런트 엔드 서버에 저장 되지만, 백 엔드 서버가이 섹션 및 [서버 하드웨어 플랫폼](https://technet.microsoft.com/library/c964c1c0-0153-472b-88ad-a38866e0df0c.aspx)의 앞부분에 나열 된 하드웨어 권장 사항을 충족 하는지 확인 해야 합니다.

백 엔드 서버의 가용성을 높이기 위해 AlwaysOn 가용성 그룹 또는 서버 미러링을 배포 하는 것이 좋습니다. 자세한 내용은 [비즈니스용 Skype 서버에서 백 엔드 서버 고가용성](../../plan-your-deployment/high-availability-and-disaster-recovery/back-end-server.md)을 참조 하세요.

## <a name="video-interop-server-capacity"></a>비디오 Interop 서버 용량

비디오 Interop 서버를 배포 하는 경우 용량을 확인 해야 하는 경우 동시 호출에 사용할 수 있는 최대 비디오 Teleconferencing 시스템 (VTCs)을 확인 합니다. 예를 들어 조직에 250 VTCs 있고 사용자 모델의 20%가 동시 통화 중일 수 있다고 예상 하는 경우, 50 동시 VTCs에 용량 계획을 수립 합니다.
