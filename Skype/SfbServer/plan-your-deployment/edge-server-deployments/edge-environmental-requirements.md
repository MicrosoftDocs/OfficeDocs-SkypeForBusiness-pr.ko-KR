# <a name="edge-server-environmental-requirements-in-skype-for-business-server"></a>비즈니스용 Skype 서버의에 지 서버 환경 요구 사항
 
**요약:** 비즈니스용 Skype 서버의에 지 서버에 대 한 환경 요구 사항에 대해 알아봅니다.
  
비즈니스용 Skype 서버에 지 서버 환경 자체의 외부에서 계획 및 준비를 많이 수행 해야 합니다. 이 문서에서는 아래 목록에 따라 조직 환경에서 수행 해야 하는 작업을 검토 합니다.
  
- [토폴로지 계획](edge-environmental-requirements.md#TopoPlan)
    
- [DNS 계획](edge-environmental-requirements.md#DNSPlan)
    
- [인증서 계획](edge-environmental-requirements.md#CertPlan)
    
- [포트 및 방화벽 계획](edge-environmental-requirements.md#PortFirewallPlan)
    
## <a name="topology-planning"></a>토폴로지 계획
<a name="TopoPlan"> </a>

비즈니스용 Skype 서버에 지 서버 토폴로지는 다음을 사용할 수 있습니다.
  
- 라우팅 가능한 공용 IP 주소
    
- **대칭적** NAT (network address translation)가 사용 되는 경우 라우팅할 수 없는 개인 IP 주소
    
> [!TIP]
> 각 서비스에 대해 개별 포트가 포함 된 단일 IP 주소를 사용 하 여에 지 서버를 구성 하거나 각 서비스에 대해 고유한 IP 주소를 사용할 수 있지만 기본적으로 TCP 443이 기본 포트를 사용 하는 것이 가능 합니다. 자세한 내용은 아래 IP 주소 요구 사항 섹션을 참조 하십시오. 
  
NAT를 사용 하 여 라우팅할 수 없는 개인 IP 주소를 선택 하는 경우 다음 사항을 염두에 두어야 합니다.
  
- **세 개의** 외부 인터페이스 모두에 대해 라우팅 가능한 개인 IP 주소를 사용 해야 합니다.
    
- 들어오는 트래픽과 나가는 트래픽에 대해 **대칭** NAT를 구성 해야 합니다. 대칭적 NAT는 비즈니스용 Skype 서버에 지 서버에서 사용할 수 있는 유일한 지원 되는 NAT입니다.
    
- 들어오는 원본 주소를 변경 하지 않도록 NAT를 구성 합니다. A/V에 지 서비스에서 들어오는 원본 주소를 수신 하 여 최적의 미디어 경로를 찾을 수 있어야 합니다.
    
- 에 지 서버는 공용 A/V에 지 IP 주소에서 서로 통신할 수 있어야 합니다. 방화벽에서이 트래픽을 허용 해야 합니다.
    
- DNS 부하 분산을 사용 하는 경우에는 확장 된 통합에 지 서버에 **만** NAT를 사용할 수 있습니다. HLB (하드웨어 부하 분산)를 사용 하는 경우에는 NAT **없이** 공용으로 라우팅할 수 있는 IP 주소를 사용 해야 합니다.
    
단일 및 확장 된 통합에 지 서버 토폴로지에 대 한 대칭 NAT를 수행 하는 라우터나 방화벽 뒤에 액세스, 웹 회의 및 A/V에 지 인터페이스 (하드웨어 부하 분산을 사용 하 고 있지 않은 경우)가 발생 하지 않습니다.

### <a name="ip-address-requirements"></a>IP 주소 요구 사항

기본 수준에서는 3 대의 서비스에 IP 주소가 필요 합니다. 액세스에 지 서비스, 웹 회의에 지 서비스 및 A/V에 지 서비스 각 서비스에 대해 하나씩 세 개의 IP 주소를 사용 하거나, 한 가지 및 옵트인을 사용 하 여 각 서비스를 서로 다른 포트에 추가할 수 있습니다 (일부에 대 한 자세한 내용은 [포트 및 방화벽 계획](edge-environmental-requirements.md#PortFirewallPlan) 섹션을 참조 하십시오). 통합 된 단일에 지 환경의 경우에는 이것을 상당히 많이 사용할 수 있습니다.
  
> [!NOTE]
> 위에서 설명한 것 처럼 세 가지 서비스 모두에 대해 하나의 IP 주소를 갖도록 선택 하 여 서로 다른 포트에서 실행할 수 있습니다. 하지만이 방법은 권장 되지 않습니다. 고객이이 시나리오에서 사용 하려는 대체 포트에 액세스할 수 없는 경우에는에 지 환경의 모든 기능에 액세스할 수 없습니다. 
  
통합 토폴로지가 확장 됨에 따라 다소 복잡할 수 있으므로, 토폴로지 선택에 대 한 기본 결정 지점이 고가용성 및 부하 분산 이라는 점을 고려 하 여 IP 주소 요구 사항을 레이아웃 하는 일부 테이블을 살펴보겠습니다. 고가용성 요구 사항에 따라 부하 분산 선택에 영향을 줄 수 있습니다 (테이블 뒤에는이에 대해 자세히 설명 함).

- **고가용성**: 배포에 고가용성이 필요한 경우 풀에에 지 서버를 두 개 이상 배포 해야 합니다. 단일에 지 풀이 최대 12 개의에 지 서버를 지원 하 고, 토폴로지 작성기를 사용 하 여 테스트 하거나 지원 하지 않는 최대 20 개까지 지원할 것 이므로이 작업을 수행 하지 않는 것이 좋습니다. 12 개의에 지 서버가 필요한 경우에는 추가에 지 풀을 만들어야 합니다.
    
- **하드웨어 부하 분산**: 대부분의 시나리오에서 DNS 부하를 분산 하는 것이 좋습니다. 물론 하드웨어 부하 분산도 지원 되지만 특히 DNS 부하 분산에 대 한 단일 시나리오에서 필요 합니다.
    
  - Exchange 2007 또는 Exchange 2010에 대 한 외부 액세스 (SP 없음) UM (통합 메시징)
    
- **Dns 부하 분산**: UM, EXCHANGE 2010 SP1 및 최신 버전은 dns 부하 분산을 통해 지원 될 수 있습니다. 이전 버전의 Exchange에 대 한 DNS 부하 분산을 사용 해야 하는 경우에는 작업을 수행 하지만이에 대 한 모든 트래픽은 풀의 첫 번째 서버로 이동 하 고, 사용할 수 없는 경우에는 해당 트래픽이 더 이상 실패 합니다.
    
    또한 다음을 사용 하 여 회사와 페더레이션 하는 경우 DNS 부하 분산을 권장 합니다.
- 비즈니스용 Skype 서버 2015:
    - Lync Server 2010
    - Lync Server 2013
    - Microsoft 365 또는 Office 365
- 비즈니스용 Skype 서버 2019:
    - Lync Server 2013
    - 비즈니스용 Skype 서버 2015
    - Microsoft 365 또는 Office 365
    
## <a name="dns-planning"></a>DNS 계획
<a name="DNSPlan"> </a>

비즈니스용 Skype 서버에 지 서버 배포와 관련해 서 DNS를 올바르게 준비 하는 것이 중요 합니다. 적절 한 레코드를 사용 하 여 배포 하는 것이 훨씬 더 쉽습니다. 위의 섹션에서 토폴로지를 선택한 후 개요를 수행 하 고, 이러한 시나리오에 대 한 DNS 레코드를 대략적으로 설명 하는 몇 가지 표를 나열 해 보겠습니다. 또한 필요한 경우 [비즈니스용 Skype 서버에 대 한 고급에 지 서버 DNS 계획](../../plan-your-deployment/network-requirements/advanced-edge-server-dns.md) 을 보다 심층적으로 읽을 수 있는 기능을 제공 합니다.

## <a name="certificate-planning"></a>인증서 계획
<a name="CertPlan"> </a>

비즈니스용 Skype 서버는 서버와 서버 간의 안전한 암호화 된 통신에 인증서를 사용 합니다. 예상 대로 인증서에는 서버에 대 한 DNS 레코드가 인증서의 모든 주체 이름 (SN) 및 주체 대체 이름 (SAN)과 일치 해야 합니다. 이 작업을 수행 하려면 이제 계획 단계에서 인증서에 대 한 SN 및 SAN 항목에 대해 DNS에 적절 한 Fqdn을 등록 했는지 확인 합니다.
  
외부 및 내부 인증서 요구 사항에 대해 별도로 논의 한 다음 두 가지 요구 사항을 모두 제공 하는 표를 확인 합니다.
  
### <a name="external-certificates"></a>외부 인증서

외부에 지 서버 인터페이스에 할당 되는 인증서는 최소한 공용 CA (인증 기관)에서 제공 해야 합니다. 사용자에 게 특정 CA를 추천할 수는 없지만, 기본 CA가 나열 되는지 확인 하는 데 도움이 되는 CAs, [통합 통신 인증서 파트너](/SkypeForBusiness/certification/services-ssl) 목록이 제공 됩니다.
  
이 공용 인증서에 대 한 요청을 CA로 제출 해야 할 시기는 언제 입니까? 이 작업을 수행 하는 방법에는 두 가지가 있습니다.
  
- 비즈니스용 Skype 서버를 설치 하 고에 지 서버 배포를 진행할 수 있습니다. 비즈니스용 Skype 서버 배포 마법사는 인증서 요청을 생성 한 후 선택한 CA로 보낼 수 있도록 하는 단계를 안내 합니다.
    
- 비즈니스 요구 사항 또는 배포 전략과 더 인라인 되는 경우에는 Windows PowerShell 명령을 사용 하 여이 요청을 생성할 수도 있습니다.
    
- 마지막으로 CA에는 Windows PowerShell 또는 다른 방법이 포함 된 자체 전송 프로세스가 있을 수 있습니다. 이 경우 참조를 위해 여기에 제공 된 정보 외에도 설명서를 사용 해야 합니다.
    
인증서를 구입한 후에는 비즈니스용 Skype 서버의 다음 서비스에이를 할당 해야 합니다.
  
- 액세스에 지 서비스 인터페이스
    
- 웹 회의에 지 서비스 인터페이스
    
- 오디오/비디오 인증 서비스 (오디오 및 비디오 스트림을 암호화 하는 데 인증서를 사용 하지 않는 것 처럼이를 A/V에 지 서비스와 혼동 하지 마십시오.)
    
> [!IMPORTANT]
> 모든에 지 서버 (동일한에 지 서버 풀에 속하는 경우)에는 미디어 릴레이 인증 서비스에 대해 동일한 개인 키를 사용 하 여 똑같은 인증서가 있어야 합니다. 
  
### <a name="internal-certificates"></a>내부 인증서

내부에 지 서버 인터페이스의 경우 공용 CA의 공용 인증서 나 조직의 내부 CA에서 발급 한 인증서를 사용할 수 있습니다. 내부 인증서에 대해 고려해 야 할 사항은 SN 항목을 사용 하 고 SAN 항목은 포함 하지 않으므로 내부 cert에서 SAN에 대해 걱정 하지 않아도 된다는 것입니다.
