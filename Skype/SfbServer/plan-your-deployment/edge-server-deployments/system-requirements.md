# <a name="edge-server-system-requirements-in-skype-for-business-server"></a>비즈니스용 Skype 서버의에 지 서버 시스템 요구 사항
 
**요약:** 비즈니스용 Skype 서버에서에 지 서버에 대 한 시스템 요구 사항에 대해 알아봅니다.
  
비즈니스용 Skype 서버에 지 서버 배포의 경우 환경 구조에 대 한 계획 뿐 아니라 환경 자체에 있는 서버에 대해 수행 해야 하는 작업은 다음과 같습니다. 토폴로지, DNS, 인증서 및 기타 인프라 관련 문제에 대 한 자세한 내용은 환경 요구 사항 설명서를 참조 하세요.
  
## <a name="components"></a>구성 요소

에 지 서버 환경에 대 한 자세한 내용은 경계 네트워크에 배포 된 구성 요소 (예를 들어, 작업 그룹이 나 비즈니스용 Skype 서버 도메인 구조 외부에 있는 도메인)를 포함 하는 대부분의 경우에 해당 됩니다.
  
이에 대 한 자세한 내용은 Edge를 배포할 때 염두에 두어야 하는 구성 요소입니다.
  
- [에 지 서버](system-requirements.md#EdgeServers)
    
- [역방향 프록시](system-requirements.md#ReverseProxies)
    
- [방화벽](system-requirements.md#Firewalls)
    
- [디렉터](system-requirements.md#Directors) (이 옵션은 선택 사항이 며 포함 된 경우 내부 네트워크에 배치 됨)
    
- [부하 분산 장치](system-requirements.md#LoadBalancers) (DNS 부하 분산 또는 HLB (하드웨어 부하 분산 장치)를 사용할 수 있지만 단일에 지 서버에 대해이는 필요 하지 않음)
    
아래에서 각 작업에 대해 자세히 설명 합니다.
  
### <a name="edge-servers"></a>에지 서버
<a name="EdgeServers"> </a>

주변 환경에 배포 된 비즈니스용 Skype 서버입니다. 여기서는 내부 비즈니스용 Skype 서버 배포에서 제공 하는 서비스에 대해 외부 사용자에 게 네트워크 트래픽을 보내고 수신 하는 역할을 합니다. 이 작업을 성공적으로 수행 하기 위해 각에 지 서버는 다음을 실행 합니다.
  
- **액세스에 지 서비스**: 아웃 바운드 및 인바운드 SIP (Session 착수 프로토콜) 트래픽에 대해 신뢰할 수 있는 단일 연결 지점을 제공 합니다.
    
- **웹 회의에 지 서비스**: 외부 사용자가 내부 비즈니스용 Skype 서버 환경에서 호스팅되는 모임에 참가할 수 있도록 합니다.
    
- **A/V에 지 서비스**: 외부 사용자가 오디오, 비디오, 응용 프로그램 공유 및 파일 전송을 사용할 수 있도록 합니다.
    
- **Xmpp 프록시 서비스**: 구성 된 Xmpp 페더레이션 파트너와의 xmpp (extensible messaging and 현재 상태 프로토콜) 메시지를 수락 하 고 보냅니다.
    
권한 있는 외부 사용자는에 지 서버를 사용 하 여 내부 비즈니스용 Skype 서버 배포에 연결할 수 있지만 그렇지 않은 경우에는 모든 사용자에 대해 내부 네트워크에 대 한 다른 액세스 권한을 제공 하지 않습니다.
  
> [!NOTE]
> 에 지 서버는 사용 하도록 설정 된 비즈니스용 Skype 클라이언트 및 기타에 지 서버 (페더레이션 시나리오)에 대 한 연결을 제공 하기 위해 배포 됩니다. 다른 끝점 클라이언트 또는 서버 유형에 서 연결할 수 없습니다. XMPP 게이트웨이 서버는 구성 된 XMPP 파트너와의 연결을 허용할 수 있습니다. 하지만 다시 사용할 수 있는 유일한 클라이언트 및 페더레이션 유형입니다. 

> [!NOTE]
> XMPP 게이트웨이 및 프록시는 비즈니스용 Skype 서버 2015에서 사용할 수 있지만 비즈니스용 Skype 서버 2019에서는 더 이상 지원 되지 않습니다. 자세한 내용은 [XMPP 페더레이션 마이그레이션을](../../../SfBServer2019/migration/migrating-xmpp-federation.md) 참조 하세요.
  
### <a name="reverse-proxies"></a>역방향 프록시
<a name="ReverseProxies"> </a>

역방향 프록시 (RP) 서버에는 비즈니스용 Skype 서버 역할이 없지만에는에 지 서버 배포의 필수 구성 요소입니다. 역방향 프록시를 통해 외부 사용자는 다음을 수행할 수 있습니다.
  
- 단순 Url을 사용 하 여 모임 또는 전화 접속 회의에 연결 합니다.
    
- 모임 콘텐츠를 다운로드 합니다.
    
- 메일 그룹을 확장 합니다.
    
- 클라이언트 인증서 기반 인증을 위해 사용자 기반 인증서 가져오기
    
- 주소록 서버에서 파일을 다운로드 하거나 주소록 웹 쿼리 서비스에 쿼리를 전송 합니다.
    
- 클라이언트 및 장치 소프트웨어에 대 한 업데이트를 받을 수 있습니다.
    
및 모바일 장치용:
  
- 모바일 서비스를 제공 하는 프런트 엔드 서버를 자동으로 검색할 수 있도록 합니다.
    
- Microsoft 365 또는 Office 365에서 모바일 장치로의 푸시 알림을 사용 하도록 설정 합니다.
    
현재 역방향 프록시 권장 사항은 [비즈니스용 Skype에 대 한 전화 통신 인프라](https://docs.microsoft.com/SkypeForBusiness/certification/infra-gateways) 페이지에서 찾을 수 있습니다. 따라서 역방향 프록시:
  
- 공용 인증서를 통해 환경에 도입 된 TLS (전송 계층 보안)를 사용 하 여 다음의 게시 된 외부 웹 서비스에 연결할 수 있어야 합니다.
    
  - 디렉터 또는 디렉터 풀
    
  - 프런트 엔드 서버 또는 프런트 엔드 풀
    
- 암호화를 위해 인증서를 사용 하 여 내부 웹 사이트를 게시 하거나, 필요한 경우 암호화 되지 않은 방법으로 게시를 게시할 수 있어야 합니다.
    
- FQDN (정규화 된 도메인 이름)을 사용 하 여 외부적으로 호스트 된 웹 사이트를 게시할 수 있어야 합니다.
    
- 호스트 된 웹 사이트의 모든 콘텐츠를 게시할 수 있어야 합니다. 기본적으로 **/\\** 대부분의 웹 서버에서 인식 되는 * 지시문을 사용 하 여 "웹 서버의 모든 콘텐츠 게시"를 의미 합니다. * */Uwca/ \\ * * *와 같이 "가상 디렉터리의 모든 콘텐츠 게시"를 의미 하는 지시어를 수정할 수도 있습니다.
    
- 게시 된 웹 사이트에서 콘텐츠를 요청 하는 클라이언트와의 TLS 연결이 필요 합니다.
    
- 주체 대체 이름 (SAN) 항목을 포함 하는 인증서를 허용 해야 합니다.
    
- 외부 웹 서비스 FQDN이 확인 하는 수신기 또는 인터페이스에 인증서 바인딩을 허용할 수 있어야 합니다. 수신기 구성은 인터페이스 보다 더 바람직합니다. 단일 인터페이스에 여러 수신기를 구성할 수 있습니다.
    
- 호스트 헤더 처리의 구성을 허용 해야 합니다. 요청 하는 클라이언트에서 보내는 원래 호스트 헤더를 역방향 프록시에서 수정 하는 대신 투명 하 게 전달 해야 하는 경우가 많습니다.
    
- 외부적으로 정의 된 특정 포트 (예: TCP 443)에서 정의 된 다른 포트 (예: TCP 4443)로 TLS 트래픽을 브리징 하도록 허용 해야 합니다. 역방향 프록시는 수신 시에 패킷을 암호 해독 한 다음 송신 시에 패킷을 다시 암호화 될 수 있습니다.
    
- tcp 80와 같은 한 포트에서 다른 포트로 암호화 되지 않은 TCP 트래픽을 브리징 하도록 허용 해야 합니다 (예: TCP 8080).
    
- 구성, 허용, NTLM 인증, 인증 안 함 및 통과 인증을 허용 해야 합니다.
    
역방향 프록시가이 목록의 모든 요구 사항을 처리할 수 있는 경우에는이 단계를 수행 하는 것이 좋지만, 앞에서 제공 하는 링크에 대 한 권장 사항을 염두에 두십시오.
  
### <a name="firewalls"></a>방화벽
<a name="Firewalls"> </a>

에 지 배포를 외부 방화벽 뒤에 배치 해야 하지만, 외부와에 지 및 내부 환경 간에 두 개의 방화벽을 설정 하는 것이 좋습니다. 이 시나리오의 모든 설명서에는 두 개의 방화벽이 있습니다. 두 개의 방화벽은 네트워크에 지 간에 엄격한 라우팅을 보장 하 고 내부 네트워크에 대 한 방화벽 보호를 두 배로 유지 하는 것이 좋습니다.
  
### <a name="directors"></a>디렉터
<a name="Directors"> </a>

이는 선택적 역할입니다. 디렉터 역할을 실행 하는 단일 서버 또는 서버 풀 일 수 있습니다. 이 역할은 내부 비즈니스용 Skype 서버 환경에서 찾을 수 있습니다.
  
디렉터는 비즈니스용 Skype 서버 내부 서버에 대해 향하는에 지 서버에서 인바운드 SIP 트래픽을 수신 하는 내부 다음 홉 서버입니다. 인바운드 요청을 사전에 인증 하 고 사용자의 홈 풀이나 서버로 리디렉션합니다. 이 사전 인증을 사용 하면 미확인 사용자 계정 요청을 삭제할 수 있습니다.
  
이와 같은 문제가 있는 이유는 무엇 인가요? 디렉터에 대 한 중요 한 기능은 DoS (서비스 거부) 공격과 같은 악의적인 트래픽에 대해 Standard Edition 서버 및 프런트 엔드 서버 또는 프런트 엔드 풀을 보호 하는 것입니다. 네트워크의 외부 트래픽이 잘못 된 경우에는 디렉터에서 트래픽이 중지 됩니다.
  
### <a name="load-balancers"></a>부하 분산 장치
<a name="LoadBalancers"> </a>

비즈니스용 Skype 서버의 확장 된 통합에 지 토폴로지는 새 배포에 대 한 DNS 부하 분산에 최적화 되어 있으며 권장 됩니다. 고가용성이 필요한 경우 한 가지 특정 상황에 대해 하드웨어 부하 분산 장치를 사용 하는 것이 좋습니다.
  
- Exchange 2013 **이전** 에 exchange um을 사용 하는 원격 사용자의 exchange um
    
> [!IMPORTANT]
> 부하 분산 장치를 혼합할 수 없다는 점에 유의 해야 합니다. 비즈니스용 Skype 서버 환경에서 모든 인터페이스는 DNS 또는 HLB를 사용 해야 합니다. 
  
> [!NOTE]
> DSR (Direct server return) NAT는 비즈니스용 Skype 서버에서 지원 되지 않습니다. 
  
#### <a name="hardware-load-balancer-requirements-for-edge-servers-edge-servers-running-the-av-edge-service"></a>A/V에 지 서비스를 실행 하는에 지 서버에 지 서버에 대 한 하드웨어 부하 분산 장치 요구 사항

A/V에 지 서비스를 실행 하는 모든에 지 서버에 대 한 요구 사항은 다음과 같습니다.
  
- 내부 및 외부 포트 모두에 대해 TCP nagling을 해제 합니다. 443 (nagling는 여러 개의 작은 패킷을 보다 효율적인 단일 패킷으로 결합 하 여 보다 효율적으로 전송 하는 프로세스입니다.
    
- 외부 포트 범위 5만-59999에 대해 TCP nagling을 해제 합니다.
    
- 내부 또는 외부 방화벽에 NAT를 사용 하지 마십시오.
    
- 에 지 내부 인터페이스는에 지 서버 외부 인터페이스와 다른 네트워크에 있어야 하며, 이러한 간의 라우팅은 사용 하지 않도록 설정 되어 있어야 합니다.
    
- A/V에 지 서비스를 실행 하는 모든에 지 서버의 외부 인터페이스는 공용으로 라우팅할 수 있는 IP 주소를 사용 해야 하며,에 지 외부 IP 주소에 NAT 또는 포트 변환이 없어야 합니다.
    
#### <a name="hlb-requirements"></a>HLB 요구 사항

비즈니스용 Skype 서버에는 쿠키 기반 선호도 요구 사항이 많지 않습니다. 따라서 비즈니스용 skype 서버 환경에서 Lync Server 2010 프런트 엔드 서버 또는 프런트 엔드 풀을 사용 해야 하는 **경우가 2015 아니면** 쿠키 기반 지 속성을 사용할 필요가 없습니다. Lync Server 2010에 권장 되는 구성 방법에는 쿠키 기반 선호도가 필요 합니다.
  
> [!NOTE]
> HLB에 대해 쿠키 기반 선호도를 설정 하도록 결정 한 경우 사용자 환경에 필요 하지 않은 경우에도 문제가 발생 하지 않습니다. 
  
사용자 환경에 쿠키 기반 선호도가 필요 **하지** 않은 경우:
  
- 포트 443에 대 한 역방향 프록시 게시 규칙에서 **정방향 호스트 헤더** 를 **True**로 설정 합니다. 이렇게 하면 원래 URL이 전달 됩니다.
    
쿠키 기반 선호도 **가 필요한** 배포의 경우:
  
- 포트 443에 대 한 역방향 프록시 게시 규칙에서 **정방향 호스트 헤더** 를 **True**로 설정 합니다. 이렇게 하면 원래 URL이 전달 됩니다.
    
- 하드웨어 부하 분산 장치 쿠키를 httpOnly로 표시 **해서는 안** 됩니다.
    
- 하드웨어 부하 분산 장치 쿠키에 만료 시간이 있어서는 **안** 됩니다.
    
- 하드웨어 부하 분산 장치 쿠키 이름은 **MS-WSMAN** (웹 서비스가 예상 하며 변경할 수 없는 값) **이어야 합니다** .
    
- 같은 TCP 연결에 대 한 이전 HTTP 응답에서 쿠키를 얻 었는 지 여부에 관계 없이 들어오는 HTTP 요청에 쿠키가 없는 모든 HTTP 응답에서 하드웨어 부하 분산 장치 쿠키를 설정 **해야** 합니다. 하드웨어 부하 분산 장치에서 쿠키를 최적화 하는 경우 TCP 연결 당 해당 최적화를 사용 **하지 않는** 것이 좋습니다.
    
> [!NOTE]
> HLB 구성에서는 클라이언트 사용량 및/또는 응용 프로그램 상호 작용을 통해 세션 상태를 유지 관리 하므로 비즈니스용 Skype 서버 및 클라이언트에 적절 한 원본 선호도 및 20 분간 TCP 세션 수명을 사용 하는 것이 일반적입니다. 
  
모바일 장치를 배포 하는 경우에는 HLB에서 TCP 세션 내의 개별 요청에 대 한 부하를 분산할 수 있어야 합니다 (실제로, 대상 IP 주소를 기준으로 개별 요청의 부하를 분산 해야 함).
  
> [!IMPORTANT]
> F5 HLBs에는 OneConnect 라는 기능이 있습니다. 이를 통해 TCP 연결 내의 각 요청이 개별적으로 부하 분산 됩니다. 모바일 장치를 배포 하는 경우 HLB 공급 업체가 같은 기능을 지원 하는지 확인 합니다. 최신 iOS 모바일 앱에는 TLS 버전 1.2이 필요 합니다. 더 자세히 알고 있어야 하는 경우에는 F5 키를 눌러이에 대 한 설정을 지정 합니다. 
  
(선택 사항) 디렉터 및 (필수) 프런트 엔드 풀 웹 서비스에 대 한 HLB 요구 사항은 다음과 같습니다.
  
- 내부 웹 서비스 Vip에 대해 HLB에서 Source_addr 지 속성 (내부 포트 80, 443)을 설정 합니다. 비즈니스용 Skype Source_addr 서버의 경우에는 단일 IP 주소에서 들어오는 여러 연결이 세션 상태를 유지 하기 위해 항상 하나의 서버로 전송 됩니다.
    
- TCP 유휴 시간 제한 (1800 초)을 사용 합니다.
    
- 역방향 프록시와 다음 홉 풀의 HLB 사이에 있는 방화벽에서 https: 포트의 트래픽 4443, 역방향 프록시에서 HLB로의 트래픽을 허용 하는 규칙을 만듭니다. HLB는 포트 80, 443 및 4443에서 수신 대기 하도록 구성 해야 합니다.
