# <a name="front-end-pool-disaster-recovery-in-skype-for-business-server"></a>비즈니스용 Skype 서버의 프론트 엔드 풀 재해 복구
 
재난 복구의 경우 비즈니스용 Skype 서버는 한 풀이 중단 되는 경우 장애 조치와 함께 풀 페어링을 제공 합니다.
  
비즈니스용 Skype Server의 강력한 재해 복구 옵션을 위해 지리적으로 분산 된 두 사이트에 프런트 엔드 풀 쌍을 배포 합니다. 각 사이트에는 다른 사이트의 해당 프런트 엔드 풀과 쌍을 둔 프런트 엔드 풀이 있습니다. 두 사이트가 모두 활성 상태 이며 백업 서비스는 실시간 데이터 복제를 제공 하 여 풀을 동기화 된 상태로 유지 합니다. 프런트 엔드 풀 페어링을 구현 하려면 [비즈니스용 Skype 서버에서 재해 복구를 위해 페어링된 프런트 엔드 풀 배포](../../deploy/deploy-high-availability-and-disaster-recovery/front-end-pools-for-disaster-recovery.md) 를 참조 하세요.
  
![서로 쌍을 이루는 두 사이트의 프런트 엔드 풀 표시](../../media/f74533c0-a10e-4f18-85a8-b9a008497573.jpg)
  
한 사이트의 풀에 장애가 발생 하는 경우 해당 풀의 사용자를 다른 사이트의 풀로 장애 조치할 수 있으며, 그런 다음 두 풀의 모든 사용자에 게 서비스를 제공 합니다. 용량 계획의 경우 재해가 발생 한 경우 두 풀의 모든 사용자에 대 한 작업 부하를 처리할 수 있도록 각 풀을 디자인 해야 합니다.
  
서로 연결 된 프런트 엔드 풀을 포함 하는 두 개의 데이터 센터가 떨어져 있을 수 있습니다. 두 데이터 센터를 동일한 월드 지역에 연결 하 고 그 사이의 고속 링크를 사용 하는 것이 좋습니다. 
  
전역 지역에 두 개의 데이터 센터를 사용할 수 있지만 데이터 복제의 대기 시간 때문에 재해가 발생 하는 경우 데이터가 더 손실 될 수 있습니다.
  
연결할 풀을 계획 하는 경우 다음 쌍만 지원 된다는 점에 유의 해야 합니다.
  
- Enterprise Edition 풀은 다른 Enterprise Edition 풀과만 쌍으로 연결 될 수 있습니다. 마찬가지로, 표준 버전 풀은 다른 스탠더드 버전 풀과만 쌍으로 연결 될 수 있습니다.
    
- 실제 풀은 다른 물리적 풀과만 쌍으로 연결 될 수 있습니다. 마찬가지로 가상 풀은 다른 가상 풀과만 쌍으로 연결 될 수 있습니다.
    
- 함께 쌍을 이루는 풀은 동일한 기본 운영 체제를 실행 해야 합니다.
    
토폴로지 작성기와 토폴로지 유효성 검사가 모두 이러한 권장 사항을 따르지 않는 방식으로 두 풀을 페어링 하지 못하게 됩니다. 예를 들어 토폴로지 작성기를 사용 하면 Enterprise Edition 풀을 스탠더드 버전 풀과 쌍으로 연결할 수 있습니다. 그러나 이러한 유형의 쌍은 지원 되지 않습니다.
  
## <a name="backup-registrar-relationships-and-survivable-branch-appliances"></a>등록 기관 관계 및 Survivable 분기 기기 백업

재해 복구 기능을 제공 하는 것 외에도 쌍을 두 번 연결 하는 풀은 서로에 대 한 백업 등록 기관 역할을 합니다. 각 풀은 한 개의 다른 프런트 엔드 풀에 대 한 백업만 될 수 있습니다.
  
두 프런트 엔드 풀 간의 백업 관계는 1:1이 고 대칭 이어야 하지만 각 프런트 엔드 풀은 Survivable 분기 기기의 백업 등록자 일 수도 있습니다.
  
비즈니스용 Skype는 Survivable Branch 기기에 속한 사용자에 대 한 재해 복구 지원을 확장 하지 않는다는 점에 유의 하세요. Survivable Branch 기기에 대 한 백업 역할을 하는 프런트 엔드 풀이 중단 되는 경우, Survivable Branch 기기에 로그인 한 사용자는 프런트 엔드 풀에 있는 사용자가 백업 프런트 엔드 풀로 장애 조치 되는 경우에도 복원 모드로 전환 됩니다.
  
## <a name="recovery-time-for-pool-failover-and-pool-failback"></a>풀 장애 조치(Failover) 및 풀 장애 복구(Failback)를 위한 복구 시간

풀 장애 조치 및 풀 장애 복구를 위해 RTO (복구 시간 목표)에 대 한 엔지니어링 대상은 15-20 분입니다. 관리자가 재난을 확인 하 고 장애 조치 절차를 시작한 후 장애 조치를 수행 하는 데 필요한 시간입니다. 관리자가 상황을 평가 하 고 의사 결정을 내리는 시간을 포함 하지 않으며 장애 조치 완료 후 사용자가 다시 로그인 하는 시간도 포함 하지 않습니다.
  
풀 장애 조치 및 풀 장애 복구를 위해 RPO (복구 시점 목표)에 대 한 엔지니어링 대상은 5 분입니다. 이는 백업 서비스의 복제 지연으로 인해 재해가 발생 하 여 손실 될 수 있는 데이터의 시간 측정값을 나타냅니다. 예를 들어 풀이 오전 10:00 시에 중단 되 고 RPO가 5 분 이면 오전 9:55 사이에 풀에 기록 된 데이터 10:00. M .이 (가) 백업 풀에 복제 되지 않았을 수 있으며, 손실 됩니다.
  
이 문서의 모든 RTO 및 RPO 번호는 두 개의 데이터 센터가 동일한 월드 지역 내에 있으며, 두 사이트 간에 고속, 짧은 지연 전송을 사용 한다고 가정 합니다. 이러한 번호는 데이터 복제에 백로그가 없는 미리 정의 된 사용자 모델과 관련 하 여 비즈니스용 Skype에 대해 사용 하도록 설정 된 4만 동시 활성 사용자와 20만 사용자가 있는 풀에 대해 측정 됩니다. 성능 테스트 및 유효성 검사에 따라 변경 될 수 있습니다.
  
## <a name="central-management-store-failover"></a>중앙 관리 저장소 장애 조치(failover)

중앙 관리 저장소는 배포의 서버 및 서비스에 대 한 구성 데이터를 포함 합니다. 각 비즈니스용 Skype 서버 배포에는 하나의 프런트 엔드 풀의 백 엔드 서버에서 호스팅되는 중앙 관리 저장소 하나가 포함 됩니다.
  
중앙 관리 저장소를 호스트 하는 풀을 쌍으로 연결 하는 경우 백업 중앙 관리 저장소 데이터베이스는 백업 풀에 설정 됩니다. 어느 시점에서 든 두 중앙 관리 저장소 데이터베이스 중 하나가 활성화 되 고 다른 하나는 대기 상태입니다. 콘텐츠는 백업 서비스에 의해 활성 데이터베이스에서 대기로 복제 됩니다.
  
![두 개의 프런트 엔드 풀 (활성 CMS 저장소와 수동 백업 CMS 저장소의 다른 하나)을 표시 합니다.](../../media/aa479398-eb56-4854-8d50-1eff39c58a56.jpg)
  
중앙 관리 저장소를 호스팅하는 풀과 관련 된 풀 장애 조치 (failover) 중에는 프런트 엔드 풀을 장애 조치 하기 전에 중앙 관리 저장소를 장애 조치 해야 합니다.
  
재해가 복구 된 후에 중앙 관리 저장소를 장애 조치할 필요는 없습니다. 중앙 관리 저장소는 장애가 발생 한 풀에 남아 있을 수 있습니다.
  
중앙 관리 저장소 장애 조치의 엔지니어링 대상은 복구 시간 목표 (RTO) 및 RPO (복구 시점 목표)에 대해 5 분입니다.
  
## <a name="front-end-pool-pairing-data-security"></a>프런트 엔드 풀 연결 데이터 보안

백업 서비스는 두 개의 쌍으로 된 프런트 엔드 풀 간에 사용자 데이터와 회의 콘텐츠를 계속 해 서 전송 합니다. 사용자 데이터에는 사용자 SIP Uri 및 컨퍼런스 일정, 연락처 목록 및 설정이 포함 되어 있습니다. 회의 콘텐츠에는 Microsoft PowerPoint 업로드 및 컨퍼런스에 사용 되는 화이트 보드 등이 포함 됩니다.
  
원본 풀에서이 데이터는 로컬 저장소, zip, 대상 풀로 전송 되며 압축을 푼 다음 로컬 저장소로 가져오게 됩니다. 백업 서비스는 두 데이터 센터 간의 통신 연결이 인터넷에서 보호 되는 회사 네트워크 내에 있다고 가정 합니다. 두 데이터 센터 간에 전송 된 데이터를 암호화 하지 않으며 HTTPS와 같은 보안 프로토콜 내에서 기본적으로 캡슐화 되지 않습니다. 따라서 회사 네트워크 내에서 내부 담당자의 중간자 공격이 가능 합니다.
  
여러 데이터 센터에 걸친 비즈니스용 Skype 서버를 배포 하 고 재해 복구 기능을 사용 하는 모든 기업은 회사 인트라넷으로 데이터 센터 간의 트래픽을 보호 해야 합니다. 내부 공격 보호를 담당 하는 기업은 데이터 센터 간의 통신 연결을 보호 해야 합니다. 이 요구 사항은 데이터 센터 간에 전송 되는 다양 한 유형의 회사 중요 데이터를 protech는 데 도움이 됩니다.
  
회사 네트워크 내에서 중간자 공격으로 인 한 공격의 위험은 인터넷에 소통량을 노출 하는 것과 비교 하 여 상대적으로 포함 되어 있습니다. 특히, 백업 서비스 (예: SIP Uri)가 제공 하는 사용자 데이터는 일반적으로 회사 내의 모든 직원이 전역 주소록이 나 다른 디렉터리 소프트웨어 등의 다른 수단을 통해 사용할 수 있습니다. 따라서 두 개의 연결 된 풀 간에 데이터를 복사 하는 데 백업 서비스를 사용 하는 경우 두 데이터 센터 간의 WAN 보안에 중점을 두어야 합니다.
  
### <a name="mitigating-security-risks"></a>보안 위험 경감

백업 서비스 트래픽에 대 한 보안 보호를 향상 시킬 수 있는 여러 가지 방법이 있습니다. 이 범위는 데이터 센터에 대 한 액세스를 제한 하 여 두 데이터 센터 간의 WAN 전송 보안을 유지 합니다. 대부분의 경우 비즈니스용 Skype 서버를 배포 하는 기업은 이미 필요한 보안 인프라를 보유 하 고 있을 수 있습니다. 지침을 찾는 엔터프라이즈의 경우 Microsoft는 보안 IT 인프라를 구축 하는 방법을 보여 주는 솔루션을 제공 합니다. 자세한 내용은을 참조 [https://go.microsoft.com/fwlink/p/?LinkId=268544](https://go.microsoft.com/fwlink/p/?LinkId=268544)하세요. 
  
이 솔루션이 유일한 솔루션 이라고 의미 하지는 않지만 비즈니스용 Skype 서버의 기본 솔루션 이라고 의미 하지는 않습니다. 기업 고객은 IT 보안 인프라 및 요구 사항에 따라 특정 요구 사항에 맞는 솔루션을 선택 하는 것이 좋습니다. 예제 Microsoft 솔루션은 서버 및 도메인 격리를 위해 IPSec 및 그룹 정책을 채택 합니다.
  
또 다른 문제 해결 방법은 백업 서비스 자체에서 보내는 데이터를 보호 하기 위해 IPSec을 사용 하는 것입니다. 이 방법을 선택 하는 경우 다음 서버에 대해 SMB 프로토콜에 대 한 IPSec 규칙을 구성 해야 하며, 여기에서 풀 A 및 풀 B는 2 쌍의 프런트 엔드 풀입니다.
  
- 풀 A의 각 프런트 엔드 서버에서 풀 B에 사용 되는 파일 저장소 까지의 SMB 서비스 (TCP/445)
    
- 풀 B의 각 프런트 엔드 서버에서 풀 A에 사용 되는 파일 저장소로 SMB 서비스 (TCP/445).
    
> [!CAUTION]
>  IPsec는 SSL/TLS와 같은 응용 프로그램 수준의 보안을 대체 하는 것이 아닙니다. IPsec을 사용할 때의 이점 중 하나는 기존 응용 프로그램에 대 한 네트워크 트래픽 보안을 변경할 필요 없이 제공할 수 있다는 것입니다. 두 데이터 센터 간의 전송을 안전 하 게 보호 하려는 기업은 해당 네트워킹 하드웨어 공급 업체에 문의 하 여 공급 업체 장비를 사용 하 여 보안 WAN 연결을 설정 하는 방법을 알아봅니다.
