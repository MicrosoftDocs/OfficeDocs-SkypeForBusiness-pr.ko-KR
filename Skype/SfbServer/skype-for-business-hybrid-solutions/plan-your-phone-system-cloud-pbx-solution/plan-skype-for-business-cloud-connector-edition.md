# <a name="plan-for-skype-for-business-cloud-connector-edition"></a>비즈니스용 Skype 클라우드 커넥터 에디션 계획

Office 365 (클라우드 PBX)에서 전화 시스템을 사용 하 여 온-프레미스 PSTN 연결을 구현 하는 패키지 된 Vm (가상 컴퓨터) 용 Skype 클라우드 커넥터 에디션에 대 한 정보를 찾습니다.

클라우드 커넥터 에디션은 기존 Lync Server 또는 비즈니스용 Skype Server 배포가 아직 없는 경우 조직의 올바른 솔루션 일 수 있습니다. 현재 비즈니스에 적합 한 Office 365 솔루션의 전화 시스템을 계속 조사 하 고 있다면 [Microsoft 전화 통신 솔루션](https://docs.microsoft.com/en-us/SkypeForBusiness/hybrid/msft-telephony-solutions)을 참조 하세요.

이 문서에서는 클라우드 커넥터 버전 요구 사항 및 지원 되는 토폴로지에 대해 설명 하 고 클라우드 커넥터 버전 배포를 계획 하는 데 도움을 줍니다. 클라우드 커넥터 환경을 구성 하기 전에이 항목을 참조 하세요. 클라우드 커넥터 에디션을 배포 하 고 구성할 준비가 되었으면 [비즈니스용 Skype 클라우드 커넥터 에디션 구성 및 관리](configure-skype-for-business-cloud-connector-edition.md)를 참조 하세요.

이제 클라우드 커넥터 버전 2.1를 사용할 수 있습니다. 2.1으로 아직 업그레이드 하지 않은 경우 [새 버전의 클라우드 커넥터로 업그레이드](upgrade-to-a-new-version-of-cloud-connector.md)를 참조 하세요. 에서 [https://aka.ms/CloudConnectorInstaller](https://aka.ms/CloudConnectorInstaller)설치 파일을 찾을 수 있습니다.

> [!NOTE]
> Microsoft는 새 버전이 출시 된 이후 60 일간의 이전 버전의 클라우드 커넥터 버전을 지원 합니다. Microsoft는 업그레이드 시간을 허용 하도록 2.1 릴리스 이후 60 일간 버전 2.0.1을 지원 합니다. 2.0.1 이전 버전은 더 이상 지원 되지 않습니다.

클라우드 커넥터 에디션은 Office 365에서 전화 시스템을 사용 하 여 온-프레미스 PSTN 연결을 구현 하는 패키지 된 Vm (가상 컴퓨터) 집합으로 구성 된 하이브리드 제공입니다. 가상화 된 환경에서 최소 비즈니스용 Skype 서버 토폴로지를 배포 하 여 클라우드에서 거주 하는 조직의 사용자가 Microsoft 클라우드에서 PBX 서비스를 받을 수 있지만, 기존 온-프레미스 음성을 통해 PSTN 연결이 제공 됩니다. 인프라.

![비즈니스용 Skype의 온-프레미스 배포에 클라우드 PBX를 연결 하는 클라우드 PBX 게이트웨이를 보여 주는 토폴로지 다이어그램입니다.](../../media/bd898e69-6458-4276-aebe-1854f28ed6fa.png)

클라우드 커넥터를 사용 하면 Office 365 서비스의 전화 시스템을 기존 전화 통신 환경 (예: PBX, 아날로그 장치, 통화 센터)과 통합할 수 있으므로 기존 전화 통신 솔루션에서 전화로 단계별 마이그레이션을 구현할 수 있습니다. Office 365의 시스템

예를 들어 회사에 Office 365의 전화 시스템에서 제공 하지 않는 특정 기능이 있는 정교한 통화 센터가 있다고 가정 합니다. 전화 센터 사용자를 기존 솔루션으로 유지 하도록 선택할 수 있지만, Office 365에서 다른 사용자를 전화 시스템으로 이동 합니다.

클라우드 커넥터는 구내 및 온라인에 있는 사용자 간 라우팅을 제공 하며, Office 365에서 전화 시스템을 사용 하 여 고유한 PSTN 공급자를 사용할 수 있습니다.

클라우드 커넥터 버전 배포 계획을 수립할 때 다음 사항을 고려 하세요.

- 클라우드 커넥터를 사용 하 여 클라우드 음성 솔루션을 활용 하려면 Office 365에서 전화 시스템을 포함 하는 Office 365 테 넌 트에 등록 해야 합니다. Office 365 테 넌 트가 아직 없는 경우 [office 365 For Business](https://products.office.com/en-us/business/office)를 사용 하 여 여기에 등록 하는 방법을 알아보세요. 비즈니스용 Skype Online을 포함 하는 요금제에 등록 해야 합니다.

- 비즈니스용 Skype Online 서비스를 사용 하 여 클라우드 커넥터 기기를 등록 하 고 다양 한 cmdlet 2.0을 실행 하려면 비즈니스용 Skype 테 넌 트 관리자 권한을 가진 전용 Office 365 계정이 필요 합니다. 2.0 이전의 클라우드 커넥터 버전에는 테 넌 트 전역 관리자 권한이 있는 전용 Office 365 계정이 필요 합니다.

- 클라우드 커넥터에는 전체 온-프레미스 비즈니스용 Skype Server 배포가 필요 하지 않습니다.

    현재 클라우드 커넥터는 Lync 또는 비즈니스용 Skype 온-프레미스 서버와 공존할 수 없습니다. 기존 Lync 또는 비즈니스용 Skype 사용자를 Office 365으로 이동 하 고 사용자에 게 온-프레미스 전화 통신을 계속 제공 하려면 기존 비즈니스용 Skype 서버 배포를 사용 하 여 온-프레미스 연결을 통해 Office 365의 전화 시스템을 고려해 보세요. 자세한 내용은 [비즈니스용 Skype Server의 온-프레미스 PSTN 연결을 사용 하 여](plan-phone-system-with-on-premises-pstn-connectivity.md)office [365 (클라우드 PBX) 솔루션에서 전화 시스템 계획](plan-your-phone-system-cloud-pbx-solution.md) 및 전화 365 시스템 계획을 참조 하세요.

- 이전에 비즈니스용 Skype 또는 Lync Server 배포를 수행 하 고 스키마를 확장 한 경우, 환경에서 비즈니스용 Skype 또는 Lync Server 구성 요소를 모두 제거한 경우에는 클라우드 커넥터 배포에 대 한 스키마를 정리할 필요가 없습니다.

- 사용자가 온라인 상태입니다.

- 조직에서 디렉터리 동기화 (DirSync)를 구성한 경우 하이브리드 음성에 대해 계획 된 모든 사용자 계정이 먼저 온-프레미스 배포에 만든 다음 클라우드와 동기화 해야 합니다.

- 필요한 경우 현재 PSTN 매체를 유지할 수 있습니다.

- 클라우드 커넥터에 호스팅되는 사용자에 게 전화 접속 회의를 제공 하려는 경우 PSTN 회의 라이선스를 구입 하거나 Microsoft에서 제공 하는 오디오 회의로 지불할 수 있습니다.

- 통화 에스컬레이션에는 오디오 회의 라이선스 (또는 사용자가 제공 하는 요금)도 필요 합니다. Skype for Business 사용자가 외부 PSTN 사용자 로부터 전화를 받고 해당 통화에 한 명 이상의 참가자를 추가 하려는 경우 (회의에 대 한 통화를 escalation) Microsoft 오디오 회의 서비스를 통해 escalation을 수행 합니다.

- 이제 클라우드 커넥터 2.0 이상에서 미디어 바이패스를 지원 합니다. 미디어 바이패스는 클라이언트가 PSTN (공용 전환 전화 네트워크) 다음 홉 인 게이트웨이 또는 세션 경계 컨트롤러 (SBC)로 직접 미디어를 보낼 수 있도록 하 고 미디어 경로에서 클라우드 커넥터 에디션 구성 요소를 제거 합니다. 자세한 내용은 [클라우드 커넥터 에디션의 미디어 바이패스 계획](plan-for-media-bypass-in-cloud-connector-edition.md)을 참조 하세요.

- 클라우드 커넥터 2.1 이상에는 OMS (Operations Management Suite)를 사용 하 여 클라우드 커넥터 모니터링이 지원 됩니다. 자세한 내용은 [OMS (Operations Management Suite)를 사용 하 여 클라우드 커넥터 모니터링](monitor-cloud-connector-using-operations-management-suite-oms.md) 을 참조 하세요.

- 클라우드 커넥터는 Office 365 Enterprise E5를 사용할 수 있는 모든 국가에서 사용할 수 있습니다.

## <a name="cloud-connector-edition-components"></a>Cloud Connector Edition 구성 요소
<a name="BKMK_Components"> </a>

클라우드 커넥터 에디션을 사용 하 여 Edge 구성 요소, 중재 구성 요소, CMS (중앙 관리 저장소) 역할 등의 비즈니스용 Skype Server 토폴로지를 포함 하는 패키지 된 Vm 집합을 배포 합니다. 또한 클라우드 커넥터의 내부 작동에 필요한 도메인 컨트롤러를 설치 합니다. 이러한 서비스는 비즈니스용 Skype Online services를 포함 하는 Office 365 테 넌 트와 하이브리드 용으로 구성 됩니다.

![Cloud Connector Edition 구성 요소](../../media/f2d4b8a7-c2f4-4cfc-8137-f187399c1298.png)

클라우드 커넥터 구성 요소는 다음과 같은 기능을 제공 합니다.

- **Edge 구성 요소** -온-프레미스 토폴로지와 온라인 서비스 간의 통신은 다음 구성 요소를 포함 하는 Edge 구성 요소를 통해 이루어집니다.

  - **액세스에 지** -온-프레미스 배포와 비즈니스용 Skype ONLINE 간의 SIP 라우팅 기능을 제공 합니다.

  - **미디어 릴레이** -중재 구성 요소와 다른 미디어 끝점 간에 미디어 라우팅을 제공 합니다.

  - **미디어 릴레이 인증/m ras** -미디어 릴레이에 대 한 액세스를 위한 토큰을 생성 합니다.

- **아웃 바운드 라우팅** -클라우드 커넥터 기기에 연결 된 게이트웨이 또는 SBCs 간 음성 트래픽에 대 한 부하 분산을 제공 합니다. 클라우드 커넥터 기기에 연결 된 모든 게이트웨이 또는 SBCs 간에 통화가 균등 하 게 분할 됩니다.

    정책에 따라 게이트웨이에 대 한 라우팅을 제공 합니다. 대상 (아웃 바운드) PSTN 번호를 기반으로 하는 글로벌 정책만 지원 됩니다.

- **Cms (중앙 관리 저장소) 역할** -Cms 파일 전송을 비롯 한 토폴로지 구성 요소에 대 한 구성 저장소를 포함 합니다.

- **Cms (중앙 관리 저장소) 복제본** -cms 역할 서버의 전역 CMS DB에서 구성 정보를 동기화 합니다.

- **도메인 컨트롤러** -클라우드 커넥터 구성 요소를 배포 하는 데 필요한 모든 전역 설정 및 그룹을 저장 하는 Active Directory 도메인 서비스 각 클라우드 커넥터 기기에 대해 하나의 포리스트가 생성 됩니다. 도메인 컨트롤러에는 프로덕션 Active Directory에 대 한 연결이 없어야 합니다. Active Directory 서비스에는 다음이 포함 됩니다.

  - Active Directory 도메인 서비스

  - Active Directory 인증서 서비스를 통해 내부 인증서 발급

- **중재 구성 요소** -비즈니스용 SKYPE와 PSTN 게이트웨이 간에 SIP 및 미디어 게이트웨이 매핑 프로토콜을 구현 합니다. 전역 CMS 데이터베이스에서 구성을 동기화 하는 CMS 복제본을 포함 합니다.

## <a name="cloud-connector-edition-topologies"></a>클라우드 커넥터 에디션 토폴로지
<a name="BKMK_Topologies"> </a>

이 논의를 위해 PSTN 사이트를 참조 하겠습니다. PSTN 사이트는 클라우드 커넥터 기기의 조합으로, 동일한 위치에 배포 되 고 공통 PSTN 게이트웨이와 연결 되어 있습니다. PSTN 사이트를 통해 다음을 수행할 수 있습니다.

- 사용자에 게 가장 가까운 게이트웨이에 대 한 연결을 제공 합니다.

- 하나 이상의 PSTN 사이트 내에 여러 클라우드 커넥터 기기를 배포 하 여 확장성을 허용 합니다.

- 단일 PSTN 사이트 내에 여러 클라우드 커넥터 기기를 배포 하 여 고가용성을 허용 합니다.

이 항목에서는 PSTN 사이트를 소개 합니다. PSTN 사이트 계획에 대 한 자세한 내용은 [클라우드 커넥터 에디션 PSTN 사이트 계획](plan-for-cloud-connector-edition-pstn-sites.md)을 참고 하세요.

다음과 같은 클라우드 커넥터 토폴로지를 배포할 수 있습니다.

- PSTN 사이트 당 단일 클라우드 커넥터 에디션 기기. 이 토폴로지는 높은 가용성을 제공 하지 않기 때문에 평가 목적 으로만 사용 하는 것이 좋습니다.

- 높은 가용성을 제공 하는 PSTN 사이트별로 여러 클라우드 커넥터 에디션 기기.

- 고가용성을 위한 확장성을 제공 하는 여러 클라우드 커넥터 에디션 기기가 있는 여러 PSTN 사이트 최대 200 사이트를 배포할 수 있습니다.

토폴로지를 계획할 때 다음 사항을 고려 하세요.

- 클라우드 커넥터 2.0 이상을 사용 하는 경우 하나의 PSTN 사이트는 최대 16 개의 클라우드 커넥터 기기를 가질 수 있습니다. 이전 버전은 사이트당 최대 4 개의 기기를 지원 합니다.

- 클라우드 커넥터로 테스트를 거친 두 가지 유형의 하드웨어 구성은 다음과 같습니다.

  - 더 큰 버전은 많은 동시 통화를 처리할 수 있으며 모든 유형의 프로덕션 환경에서 지원 됩니다.

  - 더 작은 버전은 낮은 수준의 하드웨어에서 실행 되는 것으로, 평가 목적 또는 저렴 한 통화 볼륨이 있는 사이트에 사용 될 수 있습니다. 더 작은 버전의 클라우드 커넥터를 배포 하는 경우에도 프로덕션 클래스 하드웨어 요구 사항 (예: 듀얼 전원 공급 장치)에 유의 해야 합니다.

- 클라우드 커넥터 버전 2.0 이상을 사용 하는 경우 16 개 기기의 최대 구성을 배포 하는 경우 (대규모 하드웨어), PSTN 사이트에서 최대 8000 동시 통화를 처리할 수 있습니다. 더 작은 버전을 배포 하는 경우 지원 되는 제한은 800입니다.

    또한 일부 기기는 고가용성을 위해 전용으로 사용 해야 합니다. 최소 권장 사항은 고가용성을 위해 하나의 기기를 예약 하는 것입니다.

  - 버전 2를 사용 하 여 15 + 1 구성을 배포 하는 경우 PSTN 사이트에서 최대 7500 개의 동시 통화를 처리할 수 있습니다.

  - 이전 버전을 사용 하는 경우 하드웨어를 더 크게 하 여 최대 3 + 1 개 구성을 배포 하면 PSTN 사이트에서 최대 1500 개의 동시 통화를 처리할 수 있습니다. 더 작은 버전을 배포 하는 경우 지원 되는 제한은 150입니다.

-  PSTN 사이트 별로 더 많은 통화를 해야 하는 경우 동일한 위치에 추가 PSTN 사이트를 배포 하 여 확장할 수 있습니다.

> [!NOTE]
> 별도로 언급 하지 않는 한 아래 다이어그램 및 예제에서는 더 많은 버전의 클라우드 커넥터를 사용 한다고 가정 합니다.

### <a name="single-cloud-connector-appliance-within-a-single-pstn-site"></a>단일 PSTN 사이트 내의 단일 클라우드 커넥터 기기

다음 다이어그램은 단일 PSTN 사이트 내의 단일 클라우드 커넥터 에디션 기기를 보여 줍니다. 클라우드 커넥터는 보안을 위해 경계 네트워크 내의 하나의 물리적 호스트 컴퓨터에 설치 된 4 개의 Vm으로 구성 되어 있습니다.

### <a name="multiple-cloud-connector-appliances-within-a-single-pstn-site"></a>단일 PSTN 사이트 내의 여러 클라우드 커넥터 기기

 확장성 및 고가용성을 위해 다음 다이어그램에 표시 된 것 처럼 단일 PSTN 사이트 내에 여러 클라우드 커넥터 버전을 제공 하도록 선택할 수 있습니다. 다음을 고려 하세요.

- 한 풀의 클라우드 커넥터 간에 임의의 순서로 통화가 배포 됩니다.

- 용량 계획을 위해 다음 계산에 따라 하나 이상의 클라우드 커넥터를 오프 라인으로 전환 하는 경우 로드를 처리 하는 기능을 고려해 야 합니다.

  - **N + 1 상자.** 더 많은 버전의 클라우드 커넥터의 경우 N + 1 개 상자는\*99.8%의 가용성으로 500 n 개의 동시 통화를 지원 합니다.

    더 작은 버전의 클라우드 커넥터의 경우 N + 1 개 상자는\*99.8%의 가용성으로 50 n 개의 동시 통화를 지원 합니다.

  - **N + 2 상자.** 클라우드 커넥터의 대규모 버전인 경우 N + 2 개 상자는 99.9%\*의 가용성으로 500 n 개의 동시 통화를 지원 합니다.

    더 작은 버전의 클라우드 커넥터의 경우 N + 2 개 상자는\*99.9%의 가용성으로 50 n 개의 동시 통화를 지원 합니다.

![1 PSTN 사이트 내의 두 클라우드 커넥터](../../media/fc0dc47f-5595-42cb-9432-9c8ff3e134e9.png)

### <a name="multiple-pstn-sites-with-one-or-more-cloud-connectors-per-site"></a>사이트 당 하나 이상의 클라우드 커넥터가 있는 여러 PSTN 사이트

각 사이트에서 하나 이상의 클라우드 커넥터 버전을 사용 하 여 여러 PSTN 사이트를 설치 하도록 선택할 수도 있습니다. PSTN 사이트가 동시 호출 제한에 도달 하는 경우 다른 PSTN 사이트를 추가 하 여 로드를 처리할 수 있습니다.

또한 여러 PSTN 사이트를 통해 사용자에 게 가장 가까운 게이트웨이에 대 한 연결을 제공할 수 있습니다. 예를 들어 시애틀 및 암스테르담에 PSTN 게이트웨이가 있다고 가정 합니다. 두 개의 PSTN 사이트 (시애틀에 하나씩, 암스테르담)를 배포 하 고 사용자에 게 가장 가까운 PSTN 사이트를 사용 하도록 지정할 수 있습니다. 시애틀 사용자는 시애틀 PSTN 사이트 및 게이트웨이로 라우팅되며, 암스테르담의 사용자는 암스테르담 PSTN 사이트 및 게이트웨이로 라우팅됩니다.

## <a name="requirements-for-deployment"></a>배포 요구 사항
<a name="BKMK_Requirements"> </a>

클라우드 커넥터 에디션을 배포 하기 전에 해당 환경에 대해 다음이 있는지 확인 합니다.

- **호스트 컴퓨터-** Hyper-v 역할을 사용 하도록 설정 하 여 Windows Server 2012 R2 Datacenter edition (영어)을 실행 하는 전용 하드웨어에 클라우드 커넥터 Vm을 배포 해야 합니다.

    버전 2.0 이상의 경우 비즈니스용 Skype Corpnet 스위치에 연결 된 호스트 컴퓨터 네트워크 카드는 클라우드 커넥터 회사 네트워크 컴퓨터와 동일한 서브넷에 구성 된 IP 주소를 사용 해야 합니다.

- 버전 2.1 이상에서는 호스트 기기에 .NET Framework 4.6.1 이상이 설치 되어 있어야 합니다.

- **가상 머신의 경우-** Windows Server 2012 R2 ISO (영어) 이미지 (.iso). 이 ISO는 비즈니스용 Skype 클라우드 커넥터 에디션을 실행할 가상 컴퓨터의 Vhd로 변환 됩니다.

- 배포의 각 클라우드 커넥터 에디션에 대 한 4 개의 Vm 설치를 지 원하는 데 필요한 하드웨어입니다. 다음과 같이 구성 하는 것이 좋습니다.

  - 64 비트 듀얼 프로세서, 6 개 코어 (12 실제 코어), 2.50 ghz (GHz) 이상

  - 64 기가바이트 (GB) ECC RAM

  - 4 600 GB (이상) 10K RPM 128M 캐시 SAS 6Gbps 디스크, RAID 5 구성에서 구성 됨

  - 인텔 i7 4790 쿼드 코어 (Intel 4600 그래픽 포함) (하이엔드 그래픽 필요 없음)

- 정규화 된 PBX/트렁크 또는 정식 SBC/Gateway (최소 두 개의 게이트웨이를 권장)

    클라우드 커넥터는 비즈니스용 Skype에 대해 인증 된 동일한 SBCs (세션 경계 컨트롤러)를 지원 합니다. 자세한 내용은 [비즈니스용 Skype의 전화 통신 인프라](https://docs.microsoft.com/SkypeForBusiness/certification/infra-gateways)를 참조 하세요.

- 호스트 서버에서 Hyper-v를 설치 하 고 구성 하는 권한이 있는 로컬 서버 관리자 계정 계정에는 Hyper-v가 설치 되 고 구성 된 로컬 서버에 대 한 관리자 권한이 있어야 합니다.

- 배포 중에 클라우드 커넥터 도메인에서 토폴로지를 만들고 게시 하는 권한이 있는 도메인 관리자 계정을 만들라는 메시지가 표시 됩니다.

- 설치 패키지에 포함 된 CloudConnector .ini 파일에 정의 된 외부 DNS 레코드:

  - Edge 구성 요소의 Access Edge 서비스에 대 한 외부 DNS 레코드 예를 들어 ap.\<도메인 이름\>입니다. PSTN 사이트 마다 레코드가 하나씩 필요 합니다. 이 레코드에는 해당 사이트에 대 한 모든 가장자리의 IP 주소가 포함 되어 있어야 합니다.

- 모든 필수 DNS 및 SRV 레코드가 만들어지고 있는 Office 365 테 넌 트입니다.

    > [!IMPORTANT]
    > 테 넌 트를 클라우드 Connector Edition과 통합 하는 경우 조직의 SIP 도메인으로 기본 도메인 접미사 onmicrosoft.com를 사용 하는 것은 지원 되지 않습니다. Sip를 사용할 수 >. \<이 DNS\> 레코드는 Office 365에서 사용 되기 때문에 클라우드 커넥터 경계에 대 한 액세스 프록시 인터페이스의 이름으로 도메인 이름입니다.

- 공개 CA (인증 기관)에서 가져온 외부 경계에 대 한 인증서입니다.

- 필요한 포트를 통해 트래픽을 허용 하는 방화벽 규칙이 완료 되었습니다.

- 호스트 컴퓨터와 Vm에 대 한 인터넷 연결. 클라우드 커넥터는 인터넷에서 일부 소프트웨어를 다운로드 합니다. 따라서 클라우드 커넥터 호스트 컴퓨터와 Vm이 인터넷에 연결 되어 필요한 소프트웨어를 다운로드할 수 있도록 게이트웨이 및 DNS 서버 정보를 제공 해야 합니다.

- 호스트 컴퓨터에 설치 된 테 넌 트 원격 PowerShell 모듈입니다.

- 원격 PowerShell을 실행 하려면 Office 365 비즈니스용 Skype 관리자 자격 증명입니다.

    > [!IMPORTANT]
    > 관리자 계정은 다단계 인증을 사용 하도록 설정 되어 있지 않아야 합니다.

> [!NOTE]
> 클라우드 커넥터 배포는 Microsoft Hyper-v 가상화 플랫폼 에서만 지원 됩니다. VMware 및 Amazon Web Services와 같은 다른 플랫폼은 지원 되지 않습니다.

> [!NOTE]
> 클라우드 커넥터를 실행 하는 최소 하드웨어 지침은 기본 하드웨어 용량 (코어, MHz, 기가바이트 등)에 기반 하 여 모든 컴퓨터의 아키텍처에 있는 무형의 성능 장애를 수용 하는 일부 버퍼를 사용 합니다. Microsoft는 최소 지침에 따라 상업적으로 사용할 수 있는 하드웨어를 통해 최악의 사례 부하 테스트를 실행 합니다. 미디어 품질과 시스템 성능이 확인 됩니다. Microsoft의 공식 클라우드 커넥터 기기 파트너는 독립적으로 성능 테스트를 거친 특정 클라우드 커넥터 하드웨어 구현 및 부하 및 품질 요구 조건을 충족 하기 위해 해당 하드웨어의 적합성으로 인해 성능이 향상 되었습니다.

> [!NOTE]
> 오디오 코드 및 Sonus에서 생성 되는 디바이스는 코드를 수정 하 고 Windows Server Standard 버전의 서버에서 실행 됩니다. 이러한 장치는 지원 됩니다.

## <a name="information-you-need-to-gather-before-deployment"></a>배포 하기 전에 수집 해야 하는 정보
<a name="BKMK_PlanDeployment"> </a>

배포를 시작 하기 전에 배포의 크기, 서비스를 받을 SIP 도메인, 배포 하려는 각 PSTN 사이트에 대 한 구성 정보를 결정 해야 합니다. 시작 하려면 다음을 수행 합니다.

- 회사에서 사용 중인 SIP Uri를 기반으로이 배포에 의해 제공 되는 모든 SIP 도메인을 식별 합니다.

- 배포 해야 하는 PSTN 사이트의 수를 결정 합니다.

- 각 클라우드 커넥터 에디션에 대해 설치할 4 개의 Vm을 지 원하는 데 필요한 하드웨어가 있는지 확인 합니다.

배포할 각 PSTN 사이트에 대해 다음을 수행 해야 합니다.

- 각 클라우드 커넥터 기기의 모든 구성 요소에 대 한 이름을 만듭니다 ( [배포 매개 변수 결정](plan-skype-for-business-cloud-connector-edition.md#BKMK_SiteParams)참조).

- 포트 범위 정의 ( [포트 및 프로토콜](plan-skype-for-business-cloud-connector-edition.md#BKMB_Ports)참조)

- Edge 구성 요소에 대 한 외부 DNS 레코드 만들기 ( [배포 요구 사항](plan-skype-for-business-cloud-connector-edition.md#BKMK_Requirements)참조)

- Edge 구성 요소에 대 한 인증서 요구 사항 결정 ( [인증서 요구 사항](plan-skype-for-business-cloud-connector-edition.md#BKMK_Certs)참조)

### <a name="ports-and-protocols"></a>포트 및 프로토콜
<a name="BKMB_Ports"> </a>

미디어 포트 범위를 정의할 때는 다음에 유의 해야 합니다.

- 클라이언트는 항상 미디어 트래픽에 대 한 포트 범위 5만-50019를 사용 하며,이 범위는 비즈니스용 Skype Online에 미리 정의 되어 있으며 변경할 수 없습니다.

- 중재 구성 요소는 기본적으로 포트 범위 49 152에서 미디어 트래픽에 대 한 57 500를 사용 합니다. 그러나 내부 방화벽을 통해 연결이 설정 되며, 보안상의 이유로 토폴로지에서이 포트 범위를 제한할 수 있습니다. 통화 당 최대 4 개의 포트가 필요 합니다. 중재 구성 요소와 PSTN 게이트웨이 간의 포트 수를 제한 하려면 게이트웨이에서 해당 포트 범위도 구성 해야 합니다.

- 주변 네트워크에 클라우드 커넥터를 배포 해야 합니다. 즉, 두 개의 방화벽을 사용할 수 있습니다.

  - 첫 번째 방화벽은 인터넷과 주변 네트워크 사이에 있는 외부입니다.

  - 두 번째 방화벽은 주변 네트워크와 내부 네트워크 간에 내부입니다.

  - 인터넷의 클라이언트는 Edge 구성 요소를 통해 외부 방화벽을 통해 PSTN에 연결 됩니다.

### <a name="dns-name-resolution-for-the-edge-component"></a>Edge 구성 요소의 DNS 이름 확인
<a name="BKMB_Ports"> </a>

Edge 구성 요소는 Office 365 서비스의 외부 이름과 다른 클라우드 커넥터 구성 요소의 내부 이름을 확인 해야 합니다.

각 Edge 구성 요소는 외부 및 내부 인터페이스를 포함 하는 멀티홈 컴퓨터입니다. 클라우드 커넥터는 경계 네트워크 내의 도메인 컨트롤러 구성 요소에 DNS 서버를 배포 합니다. Edge 서버를 모든 이름 확인을 위해 경계 내의 DNS 서버에 연결할 수 있지만, 이름을 참조 하는 외부 쿼리에 대해 하나 이상의 DNS A 레코드가 포함 된 DNS 영역을 설정 하 여 클라우드 커넥터 DNS 서버를 사용 하도록 설정 해야 합니다. 다른 공용 DNS 서버로의 조회.

.Ini 파일에서 SIP 도메인과 동일한 도메인 공간에서 게이트웨이의 FQDN 이름을 설정 하는 경우이 SIP 도메인에 대 한 신뢰할 수 있는 영역이 경계 내의 DNS 서버에 생성 됩니다. Edge 서버가 이름을 확인 하기 위해이 DNS 서버를 가리키고 있는 경우 Edge는 _sipfederationtls를 확인 하지 않습니다. \<바꿔야\> DNS 레코드-통화 흐름에 필요 합니다. 이 경우에는 Edge 외부 인터페이스에 DNS 서버를 제공 하 여 인터넷 이름 조회를 확인 하는 것이 좋으며, 각 Edge 구성 요소는 호스트 파일을 사용 하 여 다른 클라우드 커넥터 구성 요소 이름을 IP 주소로 확인 해야 합니다.

> [!NOTE]
> 보안상의 이유로 클라우드 커넥터 DNS 서버를 프로덕션 도메인에서 이름 확인을 위한 내부 서버를 가리키지 않는 것이 좋습니다.

### <a name="certificate-requirements"></a>인증서 요구 사항
<a name="BKMK_Certs"> </a>

각 Edge 구성 요소에는 공용 인증 기관의 인증서가 필요 합니다. Edge 구성 요소 간에 복사 하려면 인증서에 내보낼 수 있는 개인 키가 있어야 합니다. 인증서 요구 사항을 충족 하려면 다음 옵션을 결정 하 고 인증서에 대 한 주체 이름 (SN) 및 주체 대체 이름 (SAN)을 제공 해야 합니다.

 **단일 SIP 도메인이 있는 경우:**

- **옵션 1.** 주체 이름에는 Edge 구성 요소에 할당 한 풀 이름이 포함 되어야 합니다. 이 이름은 온라인 비즈니스용 Skype Edge 구성 요소에 예약 되어 있으므로 주체 이름을 sip.sipdomain.com 수 없습니다. SAN에는 sip.sipdomain.com 및 액세스에 지 풀 이름이 포함 되어야 합니다.

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com,
  acessedgepoolnameforsite1.sipdomain.com
  ```

- **옵션 2.** 배포 하는 모든 Edge 풀 서버에서 단일 와일드 카드 인증서를 사용 하려는 경우 인증서의 Edge 풀 이름 대신 와일드 카드 SAN 항목 \*을 사용 하 여 sipdomain.com를 사용할 수 있습니다. 주체 이름은 배포한 모든 Edge 풀의 액세스에 지 풀 이름이 될 수 있습니다.

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com, SAN = *.sipdomain.com
  ```

> [!NOTE]
> Sip에 대 한 외부 DNS 항목을 만들지 않아야 합니다. \<sipdomain\>는이 이름이 Office 365 배포에 속해 있기 때문에 발생 합니다.

> [!NOTE]
> 조직에 배포 된 모든 Edge 풀에 대 한 단일 인증서를 사용 하 고 옵션 2에 정의한 대로 와일드 카드 인증서를 사용할 수 없는 경우에는 인증서의 SAN 이름에 배포 된 모든 Edge 풀에 대 한 FQDN을 포함 해야 합니다.

 **여러 SIP 도메인이 있는 경우:**

모든 SIP 도메인 및 도메인당 액세스 경계 풀의 이름에 대 한 sip.sipdomain.com를 추가 해야 합니다 (하나의 물리적 풀이 있지만 이름은 서로 다를 수 있음). 다음은 여러 sip 도메인 시나리오에서 SN 및 SAN 항목의 예입니다.

- **옵션 1.** 주체 이름에는 Edge 구성 요소에 할당 한 풀 이름이 포함 되어야 합니다. 이 이름은 온라인 비즈니스용 Skype Edge 구성 요소에 예약 되어 있으므로 주체 이름을 sip.sipdomain.com 수 없습니다. SAN에는 sip.sipdomain.com 및 액세스에 지 풀 이름이 포함 되어야 합니다.

  ```
  SN = accessedgepoolnameforsite1.sipdomain1.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,
  acessedgepoolnameforsite1.sipdomain1.com
  ```

- <strong>옵션 2.</strong> 배포 하는 모든 Edge 풀 서버에서 단일 와일드 카드 인증서를 사용 하려는 경우 인증서의 Edge 풀 이름 대신 와일드 카드 SAN 항목 \*을 사용 하 여 sipdomain.com를 사용할 수 있습니다. 주체 이름은 배포한 모든 Edge 풀의 액세스에 지 풀 이름이 될 수 있습니다.

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,
  SAN = *.sipdomain1.com
  ```

> [!NOTE]
> Sip에 대 한 외부 DNS 항목을 만들지 않아야 합니다. \<sipdomain\>는이 이름이 Office 365 배포에 속해 있기 때문에 발생 합니다.

배포 목적으로 다음 표를 사용할 수 있습니다.

|**방법을**|**설명**|**상속자**|
|:-----|:-----|:-----|
|배포에 어떤 옵션을 사용 합니까?  <br/> |옵션 1 또는 2  <br/> ||
|만들려면  <br/> |인증서에 대 한 SN 제공  <br/> ||
|SAN  <br/> |인증서에 대 한 SAN을 제공 합니다.  <br/> ||

게이트웨이와 중재 서버 간에 TLS를 사용 하는 경우에는 게이트웨이에 할당 된 인증서에 대 한 루트 인증서 또는 전체 인증서 체인을 얻어야 합니다.

## <a name="high-availability-considerations"></a>고가용성 고려 사항
<a name="BKMK_HA"> </a>

고가용성을 위해 클라우드 커넥터 버전을 배포 하는 경우 서로에 대 한 백업 역할을 하는 최소 두 개의 기기를 배포 합니다. 각 기기는 경계, 중재, 중앙 관리 저장소 (CMS) 및 도메인 컨트롤러의 네 가지 구성 요소로 이루어집니다.

일반적으로 기기 내의 한 구성 요소가 다운 되는 경우 클라우드 커넥터 에디션은 계속 해 서 통화를 처리할 수 있지만 다음 사항을 고려해 야 합니다.

- **중재, CMS, 도메인 컨트롤러 구성 요소 고려 사항**

    한 기기의 CMS 또는 도메인 컨트롤러 구성 요소가 다운 되었다고 가정 합니다. 기기는 계속 해 서 인바운드와 아웃 바운드 통화를 처리할 수 있지만, 도메인 컨트롤러 또는 CMS 구성 요소에 연결 되지 않은 경우 중재 구성 요소를 다시 시작 하면 중재가 작동 하지 않습니다. 도메인 컨트롤러가 작동 하지 않는 경우에도 CMS 구성 요소를 다시 시작 하는 것과 같은 사항이 적용 됩니다.

    **권장 사항:** 구성 요소를 다시 시작 하기 전에 기기의 다른 구성 요소에 대 한 가용성을 확인 하세요.

- **Edge 구성 요소 고려 사항**

    한 기기의 Edge 구성 요소를 사용할 수 없는 경우 인바운드와 아웃 바운드 호출의 동작은 다음과 같이 달라 집니다.

  - **아웃 바운드 통화**-인터넷의 사용자가 PSTN 네트워크로 거는 통화입니다.

    클라우드의 통화 분배 메커니즘은 한 Edge 구성 요소가 중단 되었음을 확인 하 고, 모든 통화를 다른 기기에 라우팅하고, 이렇게 하면 아웃 바운드 통화가 성공적으로 수행 됩니다.

  - **인바운드 통화**-PSTN 네트워크에서 로컬 네트워크나 인터넷에 있는 사용자에 게 거는 통화입니다.

     통화를 받은 기기의 경계 구성 요소가 작동 하지 않는 경우 조정 구성 요소가 다른 기기의 Edge 구성 요소에 대 한 호출을 리디렉션할 수 없기 때문에이 기기에 대 한 인바운드 호출은 성공 하지 못합니다.

    **권장 사항:** 모니터링 시스템을 제자리에 설치 합니다. Edge 구성 요소의 오작동을 식별 한 후에는 Edge 구성 요소를 사용할 수 없는 기기의 모든 구성 요소를 종료 합니다.

## <a name="cloud-connector-media-flow"></a>클라우드 커넥터 미디어 흐름
<a name="BKMK_MediaFlow"> </a>

다음 다이어그램은 클라우드 커넥터 에디션을 통한 아웃 바운드 및 인바운드 호출의 흐름을 요약 합니다. 이는 연결이 설정 되는 방법을 이해 하는 데 유용한 정보입니다.

첫 번째 다이어그램에서 내부 사용자는 다음과 같이 아웃 바운드 호출을 배치 합니다.

1. Dave, 사용자가 온라인 상태 이지만 현재 내부 네트워크에 있는 경우 외부 PSTN 사용자에 게 전화를 겁니다.

2. SIP 트래픽은 비즈니스용 Skype Online으로 라우팅됩니다.

3. 비즈니스용 Skype Online은 번호의 역 번호 조회를 수행 합니다. 이 번호는 비즈니스용 Skype 조직의 모든 사용자에 게 속하지 않으므로 역 번호 조회가 실패 합니다.

4. 통화는 Edge 구성 요소 (온라인 Edge를 통해 SIP 및 미디어 흐름)로 라우트됩니다. 미디어는 내부 방화벽을 통해 중재 구성 요소로 이동 합니다.

5. 경로가 있는 경우에는 Edge 구성 요소가 경계 네트워크의 중재 구성 요소로 트래픽을 릴레이 합니다.

6. 중재 구성 요소는 PSTN 게이트웨이로 트래픽을 보냅니다.

![클라우드 커넥터에 대 한 아웃 바운드 미디어 흐름](../../media/c495a2bb-305c-46ef-b16d-b8f9f2b937a8.png)

다음 다이어그램에서 내부 사용자는 다음과 같은 인바운드 호출을 받습니다.

1. PSTN 게이트웨이는 온라인 상에 있지만 현재 내부 네트워크에 있는 사용자 Dave 대 한 통화를 받습니다.

2. SIP 트래픽은 중재 구성 요소로 라우팅됩니다.

3. 중재 구성 요소는 SIP 트래픽을 Edge 구성 요소에 전송 하 고 비즈니스용 Skype Online으로 이동 합니다.

4. 비즈니스용 Skype Online은 번호의 역 번호 조회를 수행 하 여 사용자 Dave를 찾습니다.

5. SIP 신호는 모든 Dave의 현재 지점으로 이동 합니다.

6. 게이트웨이 및 조정 구성 요소와 조정 구성 요소와 끝점 사이에 미디어 트래픽이 설정 됩니다.

![클라우드 커넥터에 대 한 인바운드 미디어 흐름](../../media/ba5da6f6-e357-43c6-9e8f-4bfdde97c176.png)

## <a name="monitoring-and-troubleshooting"></a>모니터링 및 문제 해결
<a name="BKMK_Monitor"> </a>

모니터링 및 문제 해결 메커니즘은 모든 클라우드 커넥터 기기와 함께 자동으로 설치 됩니다. 이 메커니즘은 다음 이벤트를 감지 합니다.

- 클라우드 커넥터 기기의 가상 머신 중 하나 이상이 내부 또는 인터넷 가상 스위치에 연결 되어 있지 않습니다.

- 클라우드 커넥터 기기의 가상 머신 중 하나 이상이 저장 또는 중지 상태입니다.

- 실행 되 고 있지 않은 서비스.

  다음 이벤트 중 하나가 감지 되는 경우, 고장난 기기에 대 한 통화를 설정 하려는 시도를 방지 하기 위해 전체 클라우드 커넥터 기기가 오프 라인으로 표시 됩니다. 클라우드 커넥터 자동 복구 기능은 계속 해 서 서비스를 복원 하 고 기기를 온라인으로 표시 합니다. 어떤 이유로 자동 복구에 실패 하는 경우 [클라우드 커넥터 배포 문제 해결](troubleshoot-your-cloud-connector-deployment.md)을 참조 하세요.

  - 중앙 관리 저장소 가상 컴퓨터에서 다음을 수행 합니다.

     - 비즈니스용 Skype 마스터 복제기 에이전트

     - 비즈니스용 Skype 복제본 복제기 에이전트

  - 중재 서버 가상 컴퓨터에서 다음을 수행 합니다.

     - 비즈니스용 Skype 복제본 복제기 에이전트

     - 비즈니스용 Skype 서버 중재

  - Edge 서버 가상 컴퓨터에서

     - 비즈니스용 Skype 복제본 복제기 에이전트

     -  비즈니스용 Skype 서버 액세스에 지

     - 비즈니스용 Skype 서버 오디오/비디오에 지

     - 비즈니스용 Skype 서버 오디오/비디오 인증

     - 비즈니스용 Skype 서버 웹 회의에 지

- Edge에서 "CS RTCSRV"에 대 한 Windows 방화벽의 인바운드 규칙, 중재 서버에서 "CS RTCMEDSRV"를 사용할 수 없습니다.

클라우드 커넥터 2.1 이상에는 OMS (Operations Management Suite)를 사용 하 여 클라우드 커넥터 모니터링이 지원 됩니다. 자세한 내용은 [OMS (Operations Management Suite)를 사용 하 여 클라우드 커넥터 모니터링](monitor-cloud-connector-using-operations-management-suite-oms.md) 을 참조 하세요.
